# Plan 6 — Clean Contrastive Run: Fixed Pooling + Multi-Positive InfoNCE

**Date:** 2026-05-03  
**Follows:** Progress_5_20260502.md

---

## 1. Motivation

Plan-5 established the full training infrastructure and produced the first contrastive results, but two bugs corrupted all trained checkpoints:

1. **EOS pooling bug** (critical): Qwen2-VL uses left-padding. The old `sum-1` formula indexed PAD tokens for shorter sequences rather than the last attended token. The correct `flip+argmax` formula was applied post-hoc to a Plan-5 checkpoint and yielded +0.098 R@10 on the dev set (0.212 → 0.310) from the same weights — confirming the bug was a major bottleneck.

2. **Duplicate target false negatives** (significant): The FACap training split has 53,686 triplets but only 28,252 unique target images (~47% collision rate). The diagonal-only InfoNCE treats same-target pairs as negatives — inflating the loss signal and penalizing the model for correctly learning similar embeddings for equivalent queries.

All Plan-5 checkpoints were trained under both bugs. Plan-6 trains from epoch 0 with both fixed.

## 3. Multi-Positive InfoNCE Design

### Problem
With all_gather across 8 GPUs (effective batch N=512), any batch almost certainly contains multiple queries sharing the same target image. Treating these as negatives pushes their embeddings apart — directly contradicting the goal of learning a consistent query embedding for the same product.

### Solution: masked log-sum-exp positives

For each query i, positive set = all j where `target_ids[i] == target_ids[j]` (including diagonal). The loss for one direction:

```
loss[i] = logsumexp(logits[i, all j]) − logsumexp(logits[i, j ∈ positives])
```

This is equivalent to: "move ALL co-positive embeddings toward query i, not just the diagonal one." Implemented in `src/training/loss.py:SymmetricInfoNCE.forward` via `_multi_positive_nce`.

### Cross-rank correctness
`target_ids` is gathered via plain `dist.all_gather` (no gradient needed) before building the positive mask, so the mask covers co-positives from all GPUs' batches — the local-sampler approach would not handle this.

### Expected effect
- Loss denominator is unchanged (all N samples still compete)  
- Loss numerator now sums over multiple positives → lower per-step gradient variance
- Training should be more stable and converge faster than diagonal-only

## 6. What to Watch

**Green flags (training on track):**
- dev R@10 ≥ 0.20 by epoch 0.5 (step 52)
- R@10 and R@50 both rising together (no divergence)
- Sensitivity gap (normal − stripped) ≥ +0.10 by epoch 1

**Yellow flags (watch but continue):**
- R@10 rising but R@50 flat → within-cluster discriminability issue, may resolve
- Loss oscillates but R@10 still trending up → batch noise, reduce LR at epoch 3

**Red flags (stop and debug):**
- R@10 < 0.20 at step 52
- Sensitivity gap ≤ 0 at any eval → model ignoring modification text
- NaN/Inf loss
