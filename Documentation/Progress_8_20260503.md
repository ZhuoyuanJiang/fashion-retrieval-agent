## 1. Context and Motivation

After Plan-6 finished training (best headline R@10 = 0.402 at ckpt_epoch16), the project still lived entirely on GitHub — no way for a mentor / advisor / non-engineer to actually try the system. Plan-8 is the bridge from "github-only code" to "click a link, see retrieval happen." The MVP requirements:

- Same UI, three backends, each clearly labelled (P1 caption-based; P2 contrastive embedding; P3 native-audio placeholder).
- Accept image + text now, image + audio later (audio is the input modality, not a separate pipeline; Whisper ASR transcribes once → both P1 and P2 receive the same text).
- Side-by-side top-K retrieved images with per-stage latency and (for curated examples) the rank of the true target.
- Reachable from a personal laptop with minimal deployment work.
- No new model training.

This progress log documents v0.1 — the **scripted** stage where retrieval results are read from a precomputed JSON cache (no GPU at request time) and ASR is mocked. v0.2 (live P2 + Whisper) and v0.3 (live P1 too) are deferred to follow-on work.

## 2. Plan / Design Decisions Locked During This Phase

The Plan_8 doc was iterated on heavily before any code was written; these are the design points that shaped v0.1 and are worth re-stating here so future-me doesn't have to re-derive them.

### 2.1 v0.1 / v0.2 / v0.3 staging

| Stage | What's live | What's cached / mocked | Where it runs |
|-------|-------------|------------------------|---------------|
| **v0.1 — Scripted demo** *(this progress)* | UI flow, mic widget, three-section layout, presets, P3 placeholder | All retrieval results read from `runs/demo/preset_cache.json`; **mock ASR** returns the preset's predefined text; latencies are **recorded**, not live | Personal laptop. No GPU at request time. |
| **v0.2 — Live Pipeline 2** | UI; Whisper ASR (real); P2 inference on preset & free-form input | P1 still cached on presets only; P3 still placeholder | Any 24 GB+ GPU host (3090 / 4090 / A6000). |
| **v0.3 — Full live** | Everything except P3 | Only P3 placeholder | A6000-class host (≥49 GB) for both VLMs concurrent. |

### 2.2 Cached vs. Live vs. Mocked (definitions)

These three terms come up everywhere in Plan_8 and the v0.x progression. Documented in detail in `notes_progress8_app.md` §Q1; brief recap:

- **Cached** — input is fixed (the 8 presets), output is the **real** model output (we ran the trained pipeline once and stored it verbatim), latency is the **recorded** wall-clock from the offline run. No GPU at demo time.
- **Live** — input is arbitrary, model is resident in GPU memory, latency is real wall-clock at request time. Costs GPU.
- **Mocked** — input is **ignored**, output is hard-coded. Used in v0.1 only for ASR (returns the preset's predefined transcript regardless of audio). Wizard-of-Oz.

Cached is preferred over mocked for retrieval results because cached answers are real (we ran the model) and mocked answers would be hand-picked good-looking matches — misleading.

### 2.3 Audio is the input modality, not a separate pipeline

Earlier drafts treated "audio" as Pipeline 3, where Pipeline 3 = Whisper → Pipeline 2. That was misleading: it implied the eventual native-audio model would be ASR-based. It's not. So:

- **Pipeline 3 = the *future* native-audio model that *skips* ASR.** Audio → VLM embedding directly. Preserves prosody and non-lexical cues for retrieval. Currently a placeholder column in the UI with a "coming soon" caveat.
- **Whisper ASR is a shared front-end for P1 & P2 audio mode.** When the user records audio, Whisper transcribes it to text once; the same text feeds both P1 and P2 (both inherently text-based). When the native-audio model lands, P3's column flips to live with no other UI change.

### 2.4 P1 cached by default, "live" behind a flag

VRAM trade-off, made explicit in Plan_8 §6:

| Mode | Pros | Cons |
|------|------|------|
| **Cached** *(v0.1 default)* | Instant. No VRAM contention. Demo rock-solid even over slow links. | P1 only runs on the 8 presets; free-form uploads show a "cached-preset mode" note in P1's section. |
| **Live** *(v0.3 only)* | P1 runs on any user upload — true "fair fight". | Two VLMs on one A6000 (~40 GB) leaves ~9 GB headroom; long mod-text inputs may OOM. |

Decision for v0.1: ship cached-only. The flag (`LIVE_PIPELINE_1` in `src/demo/config.py`) is a one-line switch reserved for v0.3.

## 4. Implementation Details

### 4.1 `src/demo/config.py`

Stage flag plus all paths. Read at import time by `app.py`:

| Constant | Value | Purpose |
|---|---|---|
| `STAGE` | `"v0.1"` (env-overridable via `DEMO_STAGE`) | Drives later live/cached branching. |
| `PRESET_CACHE_JSON` | `runs/demo/preset_cache.json` | The cached retrieval results. |
| `PRESET_THUMBS_DIR` | `runs/demo/preset_thumbs/` | Local thumbnails (laptop portability). |
| `GALLERY_DIR` | `/ssd1/zhuoyuan/facap-images` | Fallback when not on local thumbs. |
| `CKPT_DIR` | `runs/plan5/run_bs64_8xA6000_plan6_20260503_011214/ckpt_epoch16` | Plan-6 best checkpoint. |
| `TARGET_CACHE_NPY` | `runs/plan5/target_emb_cache_marqo-fashionclip.npy` | FashionCLIP gallery embeddings. |
| `CAPTION_DB_DIR` | `runs/baseline_v1_speechqwen2vl/caption_db` | (legacy default; finalize uses Marqo DB explicitly) |
| `K_DEFAULT / K_MIN / K_MAX` | 10 / 5 / 50 | UI slider range. User asked to allow up to 50. |
| `LIVE_PIPELINE_1` | `False` | Reserved for v0.3. |
| `GRADIO_SERVER_PORT / GRADIO_SHARE` | env-driven | Launch options. |

### 4.2 `src/demo/gallery.py`

Single function `image_path(target_id) -> Path` that prefers `PRESET_THUMBS_DIR/<id>.jpeg` (local, ships with the cache) and falls back to `GALLERY_DIR/<id>.jpeg` (the `/ssd1` mount). This is what makes the demo runnable on a laptop without `/ssd1` — every gallery item shown by the v0.1 cache has its thumbnail copied locally during finalize.

### 4.3 `src/demo/pipelines/base.py`

```python
@dataclass
class PipelineResult:
    target_ids: list[str]
    scores: list[float]
    image_paths: list[Path]
    latency: dict[str, float]      = field(default_factory=dict)
    intermediate: dict[str, str]   = field(default_factory=dict)
    true_target_id: str | None     = None
    true_target_rank: int | None   = None
    note: str                      = ""
```

The shared contract returned by every pipeline. v0.1 reads dicts from JSON and converts them into this shape inside `app.py`; v0.2+ will swap the source from "JSON" to "live model call" without touching the consumer code.

### 4.4 `src/demo/precompute_presets.py` (the offline cache builder)

This is the most substantive new file. Two CLI subcommands:

#### `survey` — pre-selection helper
- Loads the existing P1 dump from `runs/baseline_v1_speechqwen2vl_marqo-fashionclip/qualitative/results.jsonl` to get P1 ranks for the 1000-query headline split.
- Stratifies by P1 rank tier (top-1, 2–5, 6–10, 11–50, >50) and samples N candidates per tier (default 30 total).
- Loads Plan-6 ckpt_epoch16 + the FashionCLIP gallery cache, runs P2 inference on each sampled query.
- Prints a `P1 rank | P2 rank` comparison table to help eyeball which queries make the most interesting story for the demo.

#### `finalize` — writes the cache for chosen `query_id`s
For each chosen query:

1. **P1 top-50**: encode the cached caption (already in the P1 dump) with Marqo-FashionCLIP, cosine-search the caption DB, take top-50. *No VLM needed* — P1's caption is deterministic and already cached, so we only re-run the encoder + cosine search to expand from top-10 (dumped) to top-50.
2. **P2 top-50**: load Plan-6 ckpt_epoch16 (`set_peft_model_state_dict(model.vlm, load_peft_weights(ckpt_dir / "vlm_lora"))` + `model.proj.load_state_dict(...)`), run forward pass on `(image, mod_text)`, cosine-search the FashionCLIP gallery target cache, take top-50.
3. **Thumbnails**: copy every referenced gallery image plus the candidate image into `runs/demo/preset_thumbs/`. After all 8 presets, the directory holds **751 jpegs (11 MB)** — the full set of images the v0.1 demo can possibly display.
4. **Write JSON** with `metadata` (schema version, timestamp, ckpt_dir, encoder, K) and `presets[]` (one entry per preset; see §5.2 for fields).

Two bugs hit and fixed while running this:

- **`ContrastiveQwen2VL.proj` device mismatch.** The class constructs `self.proj = _ProjectionHead(d_target).float()` on CPU; training relies on Accelerator to place. In a standalone load script there's no Accelerator, so `model.proj.to(device)` is required immediately after `load_state_dict`. Without it: CUDA tensor × CPU weight → device-mismatch error on the first forward pass.
- **First finalize attempt clipped P1 rank to None when > 10.** The dumped P1 results only had top-10, so when the true target's rank was >10 the "rank of truth" became None. Fixed by re-encoding the cached caption against the full caption DB during finalize → real top-50 ranks recovered (e.g. preset_03 has true rank 109, which now shows correctly as ">50" in the UI).

### 4.5 `src/demo/app.py` (Gradio Blocks UI)

The single largest file (456 lines). Key components:

- **`PRESET_NARRATIVES`** — dict of 8 entries, each with `emoji` (🏆 / 🤝 / ❌), `title` (short noun phrase), `story` (1–3 sentence narrative explaining what each preset shows). Rendered in (a) the active-preset header above the input row, and (b) the "About these 8 examples" expander at the bottom.
- **Layout** — three input columns at the top (candidate image | mod-text + mic | ground-truth image with label), then a single K-slider row with master "Run ALL" button, then **three full-width vertical sections** (one per pipeline), each with its own "Run Pipeline N" button and a 10-cols × 5-rows gallery (top-50). User's explicit ask: vertical, full-width per pipeline; per-pipeline Run buttons in addition to Run ALL; ground truth visible.
- **Event handlers** — `on_preset_click` (returns 6 outputs: state, candidate img, mod-text, transcript, ground-truth img, narrative label), `on_run_p1`, `on_run_p2`, `on_run_all`. v0.1 handlers all read from `preset_cache.json`; v0.2 will swap `on_run_p2` to call the live model.
- **`launch()`** — `allowed_paths=[GALLERY_DIR, PRESET_THUMBS_DIR, runs/demo/]` so Gradio is allowed to serve files from outside `cwd`; `theme=gr.themes.Soft()`; reads `GRADIO_TEMP_DIR` env var to avoid the shared `/tmp/gradio` permission issue (see §6.1).

### 4.6 `scripts/run_demo.sh`

`bash -lc` launcher. Reads `DEMO_STAGE`, `GRADIO_SHARE`, `GRADIO_TEMP_DIR` env vars; sets `HF_HOME` to a local SSD path for v0.2+ stages (per the no-large-files-in-home rule); calls `python -m src.demo.app`. Documented in its header that the gallery path is server-specific (`/ssd1/zhuoyuan/...`), so the script is meant to be run on a server that has that mount or after `preset_thumbs/` has been populated.

## 5. The 8 Curated Presets

### 5.1 Selection criterion

From the 1000-query headline split, presets were picked to tell a balanced story:

- **5 P2 wins** (where the contrastive model beats caption-based by a meaningful margin) — the headline result.
- **2 P1 wins** (where caption-based still beats the contrastive model) — honest balance, shows the older pipeline isn't dead.
- **1 P2 fail** (true target outside top-50) — honesty about the R@10 = 0.40 ceiling.

Selection method: ran `precompute_presets.py survey` on a stratified sample of 30 candidate queries across P1-rank tiers, then picked manually based on the rank-comparison table and the modification-text content (to get visual variety: red dress, gray top, pastel pink, royal blue, dark denim, white lace, color-block, etc.).

### 5.2 Final 8 presets

`runs/demo/preset_cache.json` schema: top-level `{metadata, presets[]}`. Per-preset fields: `preset_id`, `candidate_image_id`, `modification_text`, `mock_transcript`, `true_target_id`, `category`, `notes`, `p1{target_ids, scores, latency, intermediate{caption}, true_target_rank}`, `p2{target_ids, scores, latency, intermediate{}, true_target_rank}`.

## 6. Bugs Hit and Fixes

### 6.1 `/tmp/gradio` permission denied
The shared `/tmp/gradio` directory was owned by another user on the lab server. **Fix:** set `GRADIO_TEMP_DIR=$HOME/.cache/gradio_tmp_$USER` in `scripts/run_demo.sh`. (Note: `~/.cache` is on the home NAS; per the no-large-files rule this is a small-files exception — Gradio's temp is just resized image upload caches, kilobytes per upload.)

### 6.2 Gradio 6.x API changes (vs older docs / old Plan_8 draft)
- `show_api=False` keyword no longer accepted by `demo.launch()` → removed.
- `theme=` argument moved from `gr.Blocks(theme=...)` to `demo.launch(theme=...)` → moved.
- `gr.Audio(source="microphone")` is now `gr.Audio(sources=["microphone"])` → updated.

### 6.3 `InvalidPathError` serving `/ssd1/zhuoyuan/facap-images`
Gradio 6.x refuses to serve files outside `cwd` and the system temp dir by default. **Fix:** added `allowed_paths=[str(GALLERY_DIR), str(PRESET_THUMBS_DIR), str(PRESET_CACHE_JSON.parent)]` to `demo.launch()`.

### 6.4 ContrastiveQwen2VL device mismatch on standalone load
See §4.4. Fixed by `model.proj.to(device)` immediately after `load_state_dict`.

### 6.5 P1 rank clipped at 10
First finalize iteration set `true_target_rank = None` whenever the P1 dump's top-10 didn't contain the true target. Fixed by re-running encoder + cosine search against the full caption DB during finalize.

### 6.6 (Self-inflicted) auto-committed without user accept
While preparing the commit, `git commit` was run before the user had accepted the message. User wants to commit themselves. Recovered with `git reset --soft HEAD~1`, message was re-shown for accept, user-driven commit then ran. **Workflow rule saved to memory** (`feedback_user_commits.md`) so this won't repeat.

## 7. UI Design Iterations

The layout went through several user-driven iterations. Worth recording because future v0.x stages will inherit these choices:

1. **Initial draft** — three side-by-side columns, one per pipeline, each ~33% wide. *Rejected: thumbnails too small, can't show top-50.*
2. **Vertical sections** — each pipeline gets its own full-width row with a 10-cols × 5-rows gallery. ✅ User explicitly asked for this.
3. **K slider up to 50** — user wanted to see top-50, not just top-10. K_MAX = 50 in config.
4. **Per-pipeline Run buttons + master Run ALL** — user wanted the option to inspect one pipeline at a time without re-running the others. Both options coexist now.
5. **Ground truth visible in preset mode** — user asked "ground truth是哪一个呀?" → added a third input column showing the true target image with a "Ground truth" label. Free-form mode hides this column (no ground truth available).
6. **Preset narratives with emojis** — user asked for an explanation per preset. Each preset now has `emoji + title + story` rendered above the inputs when active, plus a comparison table in the "About these 8 examples" expander.
7. **P1 displays top-50** — initially P1 only had 10 results (from the dump). User asked why P1 looked sparse → finalize now re-runs encoder+cosine to get real top-50.

## 9. Known Limitations / Out of Scope for v0.1

- **No live inference** — every retrieval result is from the cache. Free-form image upload doesn't produce P1 or P2 results in v0.1; that's v0.2/v0.3.
- **Mocked ASR** — recording audio yields the preset's predefined transcript regardless of what was said. Whisper integration is v0.2.
- **P3 placeholder only** — the native-audio model isn't trained yet. The column shows architecture text; no retrieval.
- **No authentication / multi-tenant** — single-user demo, `concurrency=1`.
- **Server-coupled gallery path** — `/ssd1/zhuoyuan/facap-images` exists only on this lab server. Mitigated by `preset_thumbs/` for v0.1.
- **Latencies are recorded, not live** — labelled as such in the UI to avoid misleading reviewers.
