## 1. Context

Plan-13 (shared-backbone two-tower, bs=24) reached headline **R@10 = 0.6540** on
the FACap dress slice — the project's best result. The roadmap's Phase C is the
audio extension.

This plan does **one thing**: swap the query tower's modification input from
typed text to **spoken audio**, with the audio entering the model directly
through speechQwen2VL's Whisper encoder + audio projector — no ASR step.

Decided constraints (from discussion 2026-05-15):

- **Reuse Plan-13's architecture and recipe verbatim.** Shared backbone + two
  PEFT adapters, multi-positive InfoNCE with cross-GPU `gather`, the Plan-12
  PEFT-aware gradient-checkpointing `context_fn`, per-epoch gallery refresh,
  18 epochs, standard AdamW. The *only* change is the query modification input.
- **Train fresh from scratch.** No warm-start from ckpt_epoch14, no frozen
  target tower. The audio two-tower co-trains both towers like Plan-13. The
  0.6540 text result is a **comparison reference only** — not an
  initialization, not an alignment target.
- No ASR baseline run: a sufficiently accurate ASR→text pipeline would just
  reproduce the text tower, so the existing 0.6540 already proxies the
  perfect-ASR case.

### speechQwen2VL audio path (verified 2026-05-15)

`DanJZY/Qwen2-VL-7B-Speech` = Qwen2-VL-7B + frozen Whisper encoder + ~17M-param
audio projector (Stage-1) + Stage-2 LoRA. The audio path is exposed by forked
`transformers` + `qwen-vl-utils` (`ZhuoyuanJiang/transformers` @
`speech-qwen2vl`) — **both already installed editable in the `fashion_retrieval`
conda env**. Plan-10/12/13 ran on this exact transformers; they just never fed
`audios=`. So there is no model-class / version blocker.

Audio input: raw `.wav`. Invocation: messages with `{"type":"audio","audio":...}`
→ `process_vision_info` → `processor(text=, images=, audios=)`.

## 2. Research questions

**Primary — how good is audio-native composed retrieval?** Train a two-tower
model fresh from scratch where the query's modification is spoken audio
(entering via speechQwen2VL's Whisper encoder + audio projector, no ASR), and
measure the headline R@{1,5,10,50} it reaches on the FACap dress slice. This
absolute number is what we want to know.

**Supplementary — how does it compare to text?** Once the audio-native number
is in, compare it against the Plan-13 text two-tower (R@10 = 0.6540) to
quantify what the modality change costs or gains. A secondary read, not the
goal.

## 3. The audio data

FACap modifications are **text-only** (~55k strings, full sentences, median
~90 chars). There is no audio, so we synthesize it. This stays audio-native:
TTS manufactures the *training input* offline, once; the model itself ingests
audio directly through the Whisper encoder; no ASR at train or inference time.

Order matters: **first decide what audio data we want (§3.1), then pick the TTS
that produces it (§3.2).**

### 3.1 What the ideal audio data looks like

The audio tower must learn the modification's **semantic content**,
speaker-invariantly, so the product works for whoever speaks to it — the focus
stays on retrieval, not on training an audio model. The synthesized dataset
should have:

| Property | Target | Why |
|---|---|---|
| Speaker diversity | ~110 distinct speakers, gender-balanced, multiple English accents | model learns content, not one synthetic timbre |
| Speaker pools | ~100 for training + the primary eval; ~10 held-out only for a separate OOD-voice eval | keeps the headline R@K from being confounded by speaker-OOD |
| Prosody | natural, non-robotic | smaller synthetic→real gap |
| Speaking rate | light per-utterance jitter | real users vary |
| Format | 16 kHz mono `.wav` | Whisper-standard |
| Channel noise / reverb | clean for v1; augmentation is an explicit later lever | keep the first number interpretable |

In more detail:

- **Many distinct speakers (~110).** If every clip used one voice, the model
  could overfit that timbre instead of learning the words. Many speakers force
  it to key on content — and directly serve the product goal: it should work
  for whoever speaks to it.
- **Two eval voice-conditions, separate numbers.** The *primary* headline
  number uses dev/headline modifications spoken by **in-training speakers**
  (in-distribution voice): the eval triplets are already held-out *content*
  (the real retrieval task), so the voice should stay in-distribution —
  otherwise speaker-OOD stacks on content-OOD and depresses the number for the
  wrong reason. A *separate* OOD-voice eval re-synthesizes the same dev/headline
  modifications with the ~10 held-out speakers; reported as its own number.
- **Gender balance** — roughly 50/50 male/female voices.
- **Accent variety** — multiple English accents (American, British, Scottish,
  Irish, …).
- **Natural prosody** — intonation and pacing close to real speech, not robotic
  — this shrinks the gap to real human input.
- **Light speaking-rate variation** — small per-utterance tempo jitter.
- **Correct pronunciation of fashion vocabulary** — necklines, fabrics,
  silhouettes ("peplum", "boatneck", "A-line") must come out right; verified in
  the pilot QC.
- **Clean audio for v1** — no background noise or reverb in the first version,
  so the first R@K is interpretable. Channel-noise / mic-variation augmentation
  is a deliberate *later* lever, out of scope here.

### 3.2 How we produce it — voice-cloning TTS + a speaker reference bank

To get ~110 distinct natural voices we do **not** need 110 TTS systems:

- **Voice-cloning TTS** — a TTS model that takes *two* inputs: the text to
  speak, and a short reference audio clip of someone's voice. It outputs the
  text spoken *in that reference voice*. Swap the reference clip → a different
  voice, same one model.
- **Speaker reference bank** — a set of reference clips, one per voice we want.
  We draw it from **VCTK**, a standard public speech corpus: ~110 real English
  speakers (different people), gender-balanced, several English accents, clean
  studio recordings, research-friendly license. We do **not** train on VCTK —
  we just borrow one short clip per speaker as a reference voice.

Pipeline: for each FACap modification string → assign a speaker → feed
`(modification text, that speaker's VCTK reference clip)` to the voice-cloning
TTS → get the modification spoken in that voice. The ~100 training-pool
speakers cover all train modifications **and** the primary dev/headline eval;
the ~10 held-out speakers are used only to re-synthesize dev/headline for the
separate OOD-voice eval.

**Choosing the TTS engine (pilot, M2).** From the §3.3 candidate shortlist,
synthesize the **same** handful (~5) of representative modifications — chosen
to include fashion-vocabulary-heavy and long ones — with each model (a few VCTK
reference voices), then choose on:
*(a)* **intelligibility** — transcribe the synthesized audio with Whisper and
measure WER vs the source text; low WER means the words survived, which is what
retrieval needs (and Whisper is the same encoder family speechQwen2VL uses
internally, so this is an on-target proxy); *(b)* **naturalness + voice
distinctness** — a side-by-side listening demo: the same sample captions
rendered by each candidate, collected into one folder so they can be played
back-to-back and picked by ear. Speed is not a constraint (8×3090, offline).

### 3.3 TTS model candidates

Selection priority: **intelligibility (low WER)** first — retrieval depends on
the words surviving — then naturalness, then voice-cloning fidelity. Every
candidate must run locally (open weights) and do zero-shot voice cloning from a
reference clip. Landscape as of 2026-05 (web research; public leaderboards are
noisy — the M2 pilot is what actually decides):

| Model | Why a candidate | Notes |
|---|---|---|
| **IndexTTS-2** | reported to lead SOTA zero-shot TTS on WER + speaker similarity | best fit for the intelligibility-first priority |
| **Fish Audio S2 Pro** | best-in-class English WER (~1%); tops EmergentTTS-Eval | open weights; commercial use needs a paid license (research use fine) |
| **Higgs Audio V2** | top-trending open TTS on HuggingFace; Llama-3.2-3B base, 10M+ hr audio | large, strong |
| **Fish Speech V1.5** | Apache-2.0, MOS ~4.1, fully open | clean-license fallback in the Fish family |
| **Chatterbox-Turbo** | MIT license, 350M, benchmarks favorably vs ElevenLabs | small + fast backup |
| **CosyVoice 2** | 2026 "top-3" open model; same Alibaba ecosystem as Qwen | — |
| **LongCat** | refreshes SOTA on speaker similarity; waveform-latent | newer |

Excluded: **Kokoro** (highest naturalness MOS but no reference-clip cloning —
incompatible with the VCTK ~110-speaker plan), **VibeVoice** (no zero-shot
cloning). Commercial engines (Eleven v3, Gemini 3.1 Flash TTS, …) are the
quality ceiling but not local — reference only.

**Pilot shortlist (M2):** IndexTTS-2, Fish (S2 Pro or Speech V1.5), Higgs
Audio V2; Chatterbox-Turbo as backup.

### 3.4 Scale & storage

~55k utterances for training + the primary eval (train ≈ 53.7k + dev 500 +
headline 1000), plus ~1.5k dev/headline clips re-synthesized with held-out
voices for the OOD eval. 16 kHz mono `.wav` on local SSD (~12–16 GB). A manifest
maps each clip to `{triplet index, wav path, speaker id, voice-condition}` so
the dataset and both evals look up the right audio.

## 4. The model change

- **Query message:** `[image, audio, fixed text instruction]` in the user turn;
  system prompt left at its default. The instruction is a constant task framing
  (it does **not** contain the modification — that is entirely in the audio).
  User-turn placement keeps the input parallel to Plan-13's text tower and to
  speechQwen2VL's own ASR training; system-prompt placement is a valid lever,
  unused in v1.
- **Target tower:** unchanged — `target image + "Describe this image in
  detail."`.
- Everything else = Plan-13.

## 6. Smoke test — image + audio multimodal forward

The one genuine unknown: speechQwen2VL was only ever fed `audio + text` (ASR);
Qwen2-VL natively does `image + text`. **`image + audio` in one message is
untested.** One-off check (not a new harness): load `DanJZY/Qwen2-VL-7B-Speech`,
confirm Whisper-encoder / projector weights load with no missing keys, feed
`[image, audio, text]`, assert finite hidden states + finite pooled embedding.
Also confirm `bs=2` padded. Gate before any tower work.

## 7. Training & eval

- Launch: `run_plan10.sh --arch shared --query-modality audio ...`.
- **Batch size TBD** — audio adds the Whisper-encoder forward + audio tokens vs
  the text run; do not assume bs=24 holds. Run the same kind of memory check
  Plan-13 §11.2 did and pick the largest stable bs.
- W&B `plan14/audio_query_<descriptor>`; `--first-eval-step 5`; dev InfoNCE loss
  logged every eval; 3-way sensitivity probe.
- Eval — three numbers, reported separately:
  1. **Primary headline R@K** — audio R@{1,5,10,50} on dev (500) + headline
     (1000), modifications spoken by **in-training speakers** (in-distribution
     voice). Isolates retrieval quality; this is *the* result.
  2. **OOD-voice eval** — the same dev/headline modifications re-synthesized
     with the ~10 held-out speakers. A separate secondary number for
     unheard-voice robustness.
  3. **Supplementary** — compare (1) against the Plan-13 text two-tower
     (R@10 = 0.6540).

### Decision points (headline R@10, audio)

| R@10 | Read |
|---|---|
| ≥ 0.60 | Audio-native two-tower is competitive. Proceed (demo refresh, etc.). |
| 0.45–0.60 | Works with a real gap — diagnose TTS quality / projector capacity / epochs; one re-run. |
| < 0.45 | Structural problem — audit image+audio wiring, TTS intelligibility, training. |

## 8. Risks

1. **Synthetic → real speech gap** — mitigated by ~100-speaker training
   diversity; the separate OOD-voice eval measures the unheard-voice gap;
   real-recorded-speech eval is a future plan.
2. **`image + audio` processor wiring** — §6 smoke gate.
3. **TTS quality on fashion vocabulary** — caught by the M2 pilot QC.
4. **Batch-size / memory unknown** — audio overhead vs Plan-13; memory check (§7).
5. Whisper 30 s input limit — non-issue (~90-char modification ≈ 6–10 s).
