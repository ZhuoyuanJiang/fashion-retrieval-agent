## Summary

Plan 15 swaps the two-tower retrieval model's **query-side modification input
from typed text to spoken audio** — audio entering the model directly through
speechQwen2VL's Whisper encoder, no ASR step. M1 (image+audio forward smoke
test) and M2 (TTS synthesis of ~55k modifications) were completed in
Progress_14. This document covers **M3** (the code) and **M4** (the run).

- **M3** wired the audio path through four source files. The architecture and
  recipe are Plan-13's two-tower setup reused verbatim — the only change is the
  query modification input.
- **M4** ran a batch-size memory scan, then launched the 18-epoch run. It was
  first launched on a 24 GB GPU host where memory forced bs=6 (36 in-batch
  negatives); after 2 healthy epochs it was migrated to a 48 GB RTX 6000 Ada
  host, where the scan supported **bs=32/GPU × 8 = 256 global batch (255
  in-batch negatives)** — larger than the text reference run (bs=24 / 192
  negatives).
- Result: headline R@10 = **0.624** at the dev-selected peak (epoch 17), or
  **0.643** at the best-headline epoch (12) — both inside the "competitive"
  band vs the 0.6540 text two-tower. The 3-way sensitivity probe confirms the
  result is genuinely audio-driven.

## 1. Context

The text two-tower (Plan-13, shared backbone + two PEFT adapters, bs=24) reached
headline **R@10 = 0.6540** on the FACap dress slice — the project's best result
and the comparison reference for the audio extension. Plan 15's research
question: does swapping the modification from text to TTS-synthesized speech,
audio entering via the Whisper encoder, trained fresh from scratch, reach
headline R@K competitive with that 0.6540?

Decided constraints (Plan 15): reuse Plan-13's architecture and recipe verbatim;
train fresh from scratch (no warm-start, no frozen target tower); 0.6540 is a
comparison reference only. M1/M2 (Progress_14) confirmed the `image + audio`
processor path works and synthesized the audio dataset.

### 2.2 What trains, what is frozen

The query message is `[candidate image, audio, fixed instruction]`; the
instruction is a constant task framing — the modification content lives
entirely in the audio. Trainable vs frozen:

- **Trainable (~48.8M params):** two PEFT LoRA adapters (query + target) + two
  projection heads + the learnable `logit_scale`.
- **Frozen:** the LLM base weights, the **Whisper audio encoder**, and the
  **audio projector**. The audio waveform is a non-grad leaf input, so the
  audio → Whisper → projector → embedding subgraph stores no backward
  activations — the audio encoder is a transient compute cost, not a memory
  cost (this is why the audio run is no heavier than the text run; see
  `notes_progress_10_two_tower_training.md` Q8).
- `exclude_modules` is required because the Whisper encoder's attention uses the
  same `q_proj/k_proj/v_proj` names matched by `LORA_TARGET_MODULES`; without it
  LoRA would wrongly attach to the frozen encoder.

### 2.3 Verification

CPU-only structural checks (dataset returns audio paths, collate batches them,
manifest assertion passes) plus an 8-GPU forward + memory probe (model
constructs, `exclude_modules` assertion passes, `image + audio` forward produces
finite embeddings). No parallel test harness was built — the run's own
`--first-eval-step 5` path is the integration check, and it passed (see §5).

## 3. M4a — batch-size memory scan

The audio path adds the Whisper-encoder forward and audio tokens versus the text
run, so the Plan-13 bs=24 could not be assumed. A worst-case probe ran real
forward+backward on batches of the **longest** audio clips (so the chosen batch
size is safe against audio-length variance over an unattended 18-epoch run).

### 3.1 24 GB GPU host (initial launch)

`torch.cuda.max_memory_allocated`, real fwd+bwd:

| bs | peak | verdict |
|---|---|---|
| 1 / 2 / 4 | 17.5 / 18.0 / 18.8 GB | ok |
| 8 | 21.2 GB | ok |
| 12 | 23.2 GB | ok |
| 16 | — | **OOM** |

With eval overhead and audio-length variance on a 24 GB card, the run launched
conservatively at **bs=6** (36 in-batch negatives across 6 GPUs).

### 3.2 48 GB RTX 6000 Ada host → bs=32

Worst-case probe (longest clips), real fwd+bwd, `torch.cuda.max_memory_allocated`:

| bs | peak | headroom (of 51 GB) |
|---|---|---|
| 12 | 24.7 GB | 26.3 GB |
| 16 | 27.6 GB | 23.4 GB |
| 20 | 30.6 GB | 20.4 GB |
| 24 | 33.3 GB | 17.8 GB |
| 28 | 35.9 GB | 15.1 GB |
| **32** | **38.6 GB** | **12.4 GB** |
| 36 | 41.2 GB | 9.8 GB |
| 40 | 43.9 GB | 7.1 GB |

No OOM through bs=40. **bs=32 was chosen** — bs=40 fits too, but 32 keeps real
margin for an unattended 18-epoch run (the probe is already worst-case on clip
length). bs=32 × 8 GPUs = 256 global batch / 255 in-batch negatives, exceeding
the text reference's 192.

## 4. M4 — the 18-epoch run

### 4.1 Migration: 24 GB host → 48 GB RTX 6000 Ada host

The run was first launched on a 24 GB GPU host at bs=6. It reached epoch 2
healthily (dev R@10 = 0.50, headline R@10 = 0.47), but bs=6 gives only 36
in-batch negatives — far below the text reference's 192, and contrastive
retrieval benefits strongly from more negatives. The run was therefore migrated
to a free 8× 48 GB RTX 6000 Ada host, where bs=32 (255 negatives) is feasible.

Migration steps: stop the 24 GB run; copy the audio dataset (14 GB / 55,186
in-dist + 1,500 OOD wavs) to the new host's local SSD; rewrite the manifest's
56,686 absolute wav paths to the new location; fresh memory scan (§3.2);
relaunch. The base model + LoRA and FACap images were already present on the
new host. The 24 GB run's 2 epochs were abandoned.

### 4.2 Run configuration

| Parameter | Value |
|---|---|
| Host | 8× RTX 6000 Ada (49 GB) |
| Architecture | shared backbone + 2 PEFT LoRA adapters (Plan-13 setup) |
| Query modality | audio |
| Per-GPU batch / global / in-batch negatives | 32 / 256 / 255 (`gather` ON) |
| Epochs / steps-per-epoch / total steps | 18 / 209 / 3762 |
| LR schedule | **constant** (`--lr-schedule none`) — LoRA 2e-5, heads 1e-4 |
| Optimizer | standard AdamW |
| Gradient checkpointing | PEFT-aware `context_fn` (Plan-12) |
| Eval | `--first-eval-step 5`, every epoch, `--eval-batch-size 16` |
| Allocator | `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` |
| NCCL watchdog timeout | 120 min (`InitProcessGroupKwargs`) |
| Audio manifest | `/ssd2/zhuoyuan/plan14_audio/manifest.json` |
| Run dir | `/ssd2/zhuoyuan/plan15_runs/audio_query_bs32` |
| W&B run | `mg1q6j8a` — `plan14/audio_query_shared_bs32_8xA6000_20260517` |
| Launched | 2026-05-17 14:53 |

### 4.3 Launch issues fixed

- **Eval OOM** on the 24 GB host — `run_dev_loss_two_tower` at the default
  batch_size was too heavy for audio. Fix: added `--eval-batch-size`.
- **NCCL collective timeout** — the rank-0 audio eval is slow; other ranks
  tripped NCCL's 10-min watchdog. Fix: `InitProcessGroupKwargs(timeout=120min)`.
- **Two failed relaunches** on the new host — a missing `cd` into the repo
  (relative script path). Fix: invoke the launcher by absolute path.

### 4.4 Memory behaviour

Steady-state per-GPU memory holds at **~38–39 GB / 49 GB** (hot rank 39.3 GB),
checked through epoch 4. It crept ~1 GB early then flattened — the text run's
one-shot epoch-4 allocator jump (+~12 GB, → 44.8 GB at bs=24) did **not** recur,
which is the intended effect of `expandable_segments` (see Q8 in
`notes_progress_10_two_tower_training.md`). ~10 GB headroom, no OOM risk.

### 4.5 Throughput

~19 s/step; all 8 GPUs sustain ~100% utilization → **compute-bound**, not
input-starved. The extra cost versus the text run is the Whisper-encoder forward
on every audio clip every step — genuine audio-modality compute. Estimated total
wall-clock ~24–26 h.

### 5.2 Sensitivity probe — is the model actually using the audio?

The 3-way probe ran every epoch on the dev set: **normal** (real clip),
**stripped** (image-only, audio removed), **shuffled** (mismatched clip).
dev R@10 under each:

| | normal | stripped (image-only) | shuffled (mismatched) | gap (normal − stripped) |
|---|---|---|---|---|
| epoch 12 (peak) | 0.670 | 0.076 | 0.016 | +0.594 |
| epoch 18 (final) | 0.684 | 0.064 | 0.022 | +0.620 |

This is the key qualitative result. Removing the audio collapses dev R@10 from
~0.68 to ~0.06–0.08; feeding a *mismatched* clip collapses it further to ~0.02
(a wrong spoken instruction actively misleads — worse than no instruction).
The model is genuinely grounded in the spoken modification; it is not exploiting
an image-only shortcut. The gap is large and stable (+0.58–0.62) across the run.

### 5.3 Checkpoint selection — which epoch is "the peak"?

`dev` (500q) is the validation set; `headline` (1000q) is the reported set. The
peak checkpoint should be selected on `dev` and the result *reported* on
`headline`. Selecting the epoch with the best `headline` means selecting on the
very number being reported — optimistic, since it picks the top of a noisy
~18-eval band.

| selection criterion | epoch | dev R@10 | headline R@10 |
|---|---|---|---|
| best headline R@10 | 12 | 0.670 | **0.643** |
| best dev R@10 | 17 | **0.694** | 0.624 |
| best dev loss (0.6637) | 14 | 0.672 | 0.624 |

Both dev-based criteria — best dev R@10 (epoch 17) and best dev loss
(epoch 14) — land on **headline R@10 = 0.624**. Only selecting directly on
`headline` gives 0.643, and epoch 12 is not a `dev` standout (dev 0.670, tied
with several epochs), so its 0.643 is partly headline-side noise.

**The dev-selected peak — epoch 17, headline R@10 = 0.624 — is the defensible
reported number; 0.643 (epoch 12) is the best-headline / optimistic bound.**

### 5.4 vs the text two-tower reference

| | headline R@10 | checkpoint picked by |
|---|---|---|
| Text two-tower (Plan-13, bs=24) | 0.6540 | best headline |
| Audio two-tower — best-headline epoch (12) | 0.643 | best headline |
| Audio two-tower — dev-selected peak (epoch 17) | 0.624 | best dev |

The text reference 0.6540 was itself selected as the best-headline checkpoint
(Progress_12 §11.4). The apples-to-apples comparison — best-headline on both
sides — is **0.643 vs 0.654, gap −0.011**. The methodologically clean number
(dev-selected) is **0.624**; a fully clean comparison would also need the text
run's best-*dev* headline, which Progress_12 did not record.

Either way, swapping the modification input from typed text to TTS-synthesized
speech costs only ~0.01–0.03 headline R@10.

## 6. Verdict

Per the Plan-15 decision bands (headline R@10):

| band | reading | result |
|---|---|---|
| **≥ 0.60** | audio-native two-tower is competitive — proceed | **← 0.624–0.643** |
| 0.45–0.60 | works with a real gap — diagnose and re-run | |
| < 0.45 | structural problem — audit wiring / TTS / training | |

**The audio-native query tower is competitive.** However the peak checkpoint is
selected — dev-selected headline R@10 = 0.624, or best-headline 0.643 — the
result is inside the "competitive" band and within ~0.01–0.03 of the 0.6540
text two-tower. The sensitivity probe confirms the result is genuinely
audio-driven. The Plan-15 research question — *does the modification entering
as TTS-synthesized speech via the Whisper encoder, trained fresh from scratch,
reach R@K competitive with the text two-tower?* — is answered **yes**.

Notes / caveats:

- Reported peak is the **dev-selected** checkpoint `ckpt_epoch17` (headline
  R@10 = 0.624). `ckpt_epoch12` (0.643) is the best-headline / optimistic
  bound; final `ckpt_epoch18` = 0.618.
- Constant LR (`--lr-schedule none`); no late-epoch decay lift. A cosine
  schedule is an untried lever that may close the gap to 0.6540.
- The result uses TTS-synthesized speech. Real-recorded-speech eval and
  channel-noise robustness remain future work (Plan-15 §8 risk 1).
- W&B run `mg1q6j8a`; run dir `/ssd2/zhuoyuan/plan15_runs/audio_query_bs32`
  (`metrics.json` + per-epoch checkpoints `ckpt_epoch1..18` on the run host's
