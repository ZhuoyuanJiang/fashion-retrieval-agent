## 1. Context

Plan-6 produced the project's first contrastive model (best headline R@10 = 0.402,
`runs/plan5/run_bs64_8xA6000_plan6_20260503_011214/ckpt_epoch16/`). It still trailed
the Phase-A caption + Marqo FashionCLIP baseline (R@10 = 0.533).

Mentor's 2026-05-03 feedback (`Documentation/meeting_memo_20260503.md`): Plan-6 is
not a real two-tower architecture — its target tower is a **frozen FashionCLIP image
encoder**, so the Qwen2VL query tower is effectively imitating FashionCLIP's
embedding space. Ceiling is FashionCLIP's discrimination ability. Mentor's proposed
direction:

> Replace the FashionCLIP target tower with **a second trainable Qwen2VL tower**.
> Query takes (reference_image, modification_text); target takes (target_image,
> "Describe this image in detail."). Train contrastively with multi-positive
> InfoNCE on triplet labels. Ground truth is the actual target image, NOT a
> FashionCLIP embedding.

The embedding space is **co-constructed by both towers** during training, not
inherited from a frozen teacher. This is the CLIP training dynamic — both encoders
trainable from scratch, joint embedding space emerges from contrastive loss on
labelled pairs.

Plan-10 V1 is the minimum-viable instantiation: one comparable headline R@10 number
to decide whether to invest in V2 (trainable-surface ablations), V3 (target-prompt
sweep), or V4 (audio extension on the query side).

## 3. Implementation Summary

End-to-end Option B implementation: dataset extension, two-tower model wrapper,
distributed gallery refresh, training script with grad-receipt assertion, launcher.

### New files

- **`src/training/two_tower_model.py`** — `TwoTowerSeparateBackbones` wrapping two
  independent `ContrastiveQwen2VL` instances. The target tower's `forward` is
  rebound via `__get__` to use the locked target prompt
  (`"Describe this image in detail."`) instead of inheriting Plan-6's query
  template ("Given this product image, find the item..."). Parent
  `forward(cand_images, mod_texts, tgt_images) → (q_emb, t_emb)` is the DDP-safe
  entry point used after `accelerator.prepare(model)`. `encode_query` and
  `encode_target` are `@torch.inference_mode` eval helpers for the
  **unwrapped** model.

- **`src/training/train_plan10.py`** — training loop derived from
  `src/training/train_plan5.py`. Differences:
  - 5 optimizer param groups: q_lora, q_proj, t_lora, t_proj, logit_scale
    (Plan-6 had 3).
  - Step-1 **grad-receipt assertion**: enumerates `lora_B` params per adapter and
    asserts at least one received gradient. Catches DDP / requires_grad
    regressions early. Designed to handle both Option A (`.query.`/`.target.`
    naming) and Option B (`query_tower.`/`target_tower.` naming).
  - `--first-eval-step` CLI flag (default 5): triggers the existing eval pipeline
    once after step N to catch OOM / sharding / wiring bugs ~2 min into the run,
    instead of waiting for the regular 0.5-epoch eval (~25 min). Avoids
    re-implementing a parallel smoke harness.
  - **Initial gallery encoding** before the training loop (all 7 ranks
    participate), writes `gallery_emb_epoch0.npy`.
  - **End-of-epoch gallery refresh**: re-encodes the 59k gallery via the (now
    updated) target tower, overwrites in-memory `gallery_lookup` + `gallery_db`.
    ~3-5 min × 18 epochs ≈ ~1 hr added to wall time.
  - Eval block factored into `_run_eval(step, epoch_frac)` and called once more
    after the outer epoch loop — captures the post-last-epoch-refresh R@K which
    the in-loop schedule would miss (see §5 finding 2).
  - W&B run-name default auto-detects GPU class via `torch.cuda.get_device_name`
    (`A6000`, `3090`, `A100`, ...) and appends launch date → e.g.
    `plan10/v1_separate_bs8_7xA6000_20260512`.

- **`scripts/run_plan10.sh`** — multi-GPU launcher. Defaults: 8 GPUs, bs=8,
  `--gather` ON. `HF_HOME` and `WANDB_DIR` on local SSD (`/ssd1/zhuoyuan/`) per
  the no-large-files-in-home rule. No target-cache build step — Plan-10's gallery
  is dynamic (refreshed each epoch).

### Modified files

- **`src/data/contrastive_dataset.py`** — `FacapContrastiveDataset` gains a
  `load_target: bool = False` constructor arg. When True, `__getitem__` also loads
  the target PIL image. `contrastive_collate` adds `tgt_images` to the batch dict
  only when items contain `tgt_image`. Plan-5/6/7 path (load_target=False) is
  preserved.

- **`src/training/online_eval.py`** — two new helpers for Plan-10's dynamic target
  tower:
  - `run_dev_loss_two_tower(model, loss_fn, dev_items, tid_to_idx, base_ds,
    device, batch_size=32)` encodes the 500 dev items' target embeddings
    **on-the-fly** via `model.encode_target(images)`, instead of looking them up
    from a precomputed FashionCLIP cache.
  - `encode_gallery_with_tower(model, gallery_ids, gallery_paths, batch_size,
    accelerator, out_dir, epoch_tag)` shards the 59,048-image gallery across
    ranks. Each rank pads its shard to the **global ceiling** (so `all_gather`
    doesn't deadlock when N is not divisible by world_size), gathers paired
    `(embedding, original_index)` pairs, then `argsort` reorders the result back
    to canonical gallery order. Writes `.npy` + `.meta.json` (matches
    `target_cache.load_target_cache` contract).
  - `run_retrieval_eval` switched to `model.encode_query(images, texts)` which
    works for both `ContrastiveQwen2VL` (Plan-5/6/7 path) and
    `TwoTowerSeparateBackbones` (Plan-10).

### Cross-review history

Five rounds of plan-doc review (artifacts in
`.cross_review/Plan_10_20260510_2026-05-11_14-40/`). Notable findings that were
agreed and fixed in the plan before any code was written:

1. **PEFT `set_adapter` footgun** (round 1) — PEFT 0.17.1's `BaseTunerLayer.set_adapter`
   flips `requires_grad_(False)` on inactive adapters. Means Option A's
   `set_adapter("query")` → forward → `set_adapter("target")` → forward sequence
   would silently zero out gradients for the query LoRA before backward fires.
   Plan-10 §4.3 documents this and Appendix B's Option A walkthrough has an
   explicit `for n,p in self.backbone.named_parameters(): if "lora_" in n:
   p.requires_grad = True` reset before the parent forward returns.
2. **Distributed gallery refresh** — naive `accelerator.gather` deadlocks when
   gallery size N=59,048 isn't divisible by world_size=8. Pad-to-ceiling +
   paired-id gather + argsort permutation (now implemented in
   `encode_gallery_with_tower`).
3. **DDP-unsafe submodule calls** — original plan had
   `model.forward_query(...)` calls which bypass DDP after `accelerator.prepare`.
   Fixed: parent `forward(cand, mod, tgt) → (q_emb, t_emb)` is the single
   DDP-safe entry.
4. **Buggy step-1 grad assertion** — original asserted nonzero grad on `lora_A`
   at step 1, but PEFT inits `lora_B = 0` so step-1 `∂loss/∂lora_A = 0` exactly.
   Fixed: check `lora_B` only at step 1.
5. **Option B parameter naming for the grad-receipt assertion** — assertion was
   written for Option A's `.query.`/`.target.` naming; Option B's PEFT adapter
   name defaults to `"default"` so the binning logic missed. Generalized to
   handle both.

All five rounds ended with "No findings" on the locked plan.

## 5. Findings

### Finding 1 — What `target_modules=["q_proj","k_proj","v_proj","o_proj"]` actually attaches to

The plan-6-inherited LoRA target list was a substring/suffix match against module
names. To verify what it actually hits in speechQwen2VL we loaded the model on the
meta device and walked `named_modules()`. Results:

| Sub-model | Module path prefix | Layers | Attention modules per layer | Match? | LoRA modules added |
|---|---|---|---|---|---|
| Language model | `model.language_model.layers.<N>.self_attn.{q,k,v,o}_proj` | 28 | `q_proj`, `k_proj`, `v_proj`, `o_proj` | ✅ all 4 | 28 × 4 = **112** |
| Audio encoder | `...audio_tower.layers.<N>.self_attn.{q,k,v}_proj` + `.out_proj` | 32 | `q_proj`, `k_proj`, `v_proj`, **`out_proj`** (Whisper convention) | ✅ q/k/v match; **`out_proj` does NOT end with `o_proj`** → output projection skipped | 32 × 3 = **96** |
| Vision encoder (ViT) | `model.visual.blocks.<N>.attn.qkv` | 32 | fused `qkv` (single Linear → split internally) | ❌ no `q_proj`/`k_proj`/`v_proj`/`o_proj` match | **0** |

**Total LoRA modules per tower: 208** — matches the smoke-test grad-receipt count
`query: 112/208 lora_B got grad, target: 112/208`.

**Why only 112 receive gradient.** In V1 the towers process `(image, text)` only.
The audio encoder is never executed → its LoRA forward never runs → its `lora_B`
(init = 0) is never updated. Those 96 audio-side LoRA modules sit at init forever
and contribute zero to outputs. ViT has no LoRA at all (frozen image encoder, same
as Plan-6).

**Verification of Stage-2 audio LoRA safety.** A concern that came up during
implementation: would our `requires_grad = True` reset for Option A's PEFT footgun
mitigation accidentally re-enable Stage-2's audio adapter weights? Answer: no.
Stage-2's actual adapter config (read directly from
`/ssd1/zhuoyuan/hf_cache/.../adapter_config.json`) shows
`target_modules = "model.language_model.layers.\d+.(self_attn|mlp).<projection>"` —
i.e., Stage-2 trained **LM-only LoRA** (28 layers × 7 modules = 196 modules). It
never touched the audio encoder. Then Stage-2 is `merge_and_unload`-ed into the
base before our contrastive LoRA is attached, so its weights live in the base
matrices as plain `nn.Linear` weights and are not exposed as `lora_*` params
anymore. Any `if "lora_" in n: p.requires_grad = True` reset we do touches only
our fresh contrastive LoRA.

**Implications:**
1. **The 96 audio LoRA modules are dead code in V1** — ~3 MB of VRAM per tower
   for params that never train. Cosmetic waste. V1 keeps the setting verbatim for
   strict Plan-6 comparability. V2/V3 cleanup option:
   `target_modules=r"model\.language_model\.layers\.\d+\.self_attn\.(q_proj|k_proj|v_proj|o_proj)$"`
   (regex) or PEFT's `exclude_modules=["audio_tower"]`.
2. **ViT-LoRA is intentionally out of scope for V1.** Adapting the image encoder
   is a real lever (Marqo FashionCLIP's win came from fashion-tuning the image
   encoder), but it adds trainable params on a small dataset (overfitting risk)
   and departs from Plan-6. Recorded as a V2/V3 ablation candidate — to enable,
   add `qkv` to `target_modules` so PEFT matches `model.visual.blocks.<N>.attn.qkv`.
3. **Grad-receipt assertion was originally too strict.** Initial version required
   ALL 208 `lora_B` per adapter to receive gradient at step 1. First smoke crashed
   on this. Relaxed to "at least 1 per adapter" — proves the tower participates
   in the forward graph, which is the only check the assertion actually needs.

### Finding 2 — End-of-epoch eval never fires

The in-loop eval schedule (`eval_every=0.5` epoch) fires at steps `0, 5, 479, 958,
1437, 1916, …`. The epoch-boundary eval (e.g., step 958 for epoch 0) only triggers
when the inner batch loop iterates to that step — which happens at the **start of
the next epoch**. After the **final** epoch's batch loop completes, the outer loop
exits and the step-`max_epochs * steps_per_epoch` eval is never reached.

For the smoke (`max_epochs=1`): we got evals at step 0, 5, 479. The end-of-epoch
gallery refresh (`gallery_emb_epoch1.npy`) ran but no eval against the freshly
refreshed gallery followed.

**Fix applied:** factored the eval block in `train_plan10.py:467+` into a
`_run_eval(step, epoch_frac)` helper, then added one final call to it after the
outer `for epoch in range(...)` loop exits. The helper reads the `gallery_db`
binding from enclosing scope, so it picks up whichever gallery was last refreshed.
For the full 18-epoch run this gives us the post-epoch-17-refresh R@K, which is
the only R@K that reflects the **final** model state against **final** target
embeddings.

### Finding 3 — Conda env not picked up in non-interactive SSH

First smoke launch failed at `scripts/run_plan10.sh: line 83: accelerate: command
not found`. Root cause: the launcher's `conda run -n fashion_retrieval which
accelerate` fallback to `accelerate` (bare command) doesn't resolve in
non-interactive SSH because conda's init isn't sourced. Fix at the SSH-command
layer (not the launcher): prepend
`export PATH=~/miniconda3/envs/fashion_retrieval/bin:$PATH` before invoking the
launcher. The launcher's `conda run` lookups now resolve.

### Finding 4 — Mentor's V4 angle strengthens Option A

The mentor's 2026-05-11 follow-up: Option A's shared backbone is the principled
choice because in V4 (audio extension), the target tower stays **literally
unchanged** (same shared backbone, same target LoRA, same projection head, same
input prompt). Only the query-side input pipeline changes (text → audio). Option
B is functionally equivalent in V4 outcomes, but the target tower lives in a
separate `nn.Module` instance whose weights are managed independently;
preserving target-side invariance is a discipline, not a default.

This corrected understanding is now in Plan_10 §3, Appendix A Q1 follow-up, and
Appendix B side-by-side table. Q1 is not yet formally closed — implementing
Option B first to validate the architecture, then implementing Option A is the
agreed plan.

### Why server 11 (not server 10)
Option A's full run was launched first on **server 10** in tmux session
`plan10_a`. To run Option B's full run **in parallel** for a head-to-head
comparison without waiting ~30 hours, the user authorized launching Option B
on **server 11**. (Server 10 has 8× A6000; server 11 has 7× A6000 — close
enough for an apples-to-apples comparison.) This required first rsyncing the
data + cached HF weights from server 10 → server 11, since server 11 had no
prior fashion-retrieval-agent state.

### Data rsync (server 10 → server 11)
All three rsyncs ran from server 10 over the LAN. Targets land on server 11's
`/ssd2` so we don't touch the (full) `/ssd1` on either machine and avoid the
NAS home quota.

```

ssh vllab10 'tar -cf - -C /ssd1/zhuoyuan facap-images' \
    | ssh vllab11 'tar -xf - -C /ssd2/zhuoyuan'

# 2. Qwen2-VL-7B-Speech base + Stage-2 LoRA (~17 GB safetensors)

          /ssd2/zhuoyuan/hf_cache/models--DanJZY--Qwen2-VL-7B-Speech/
```
