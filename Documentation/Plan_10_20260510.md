## 1. Context

Plan-6 produced the project's first contrastive model (best headline R@10 = 0.402 at `runs/plan5/run_bs64_8xA6000_plan6_20260503_011214/ckpt_epoch16/`). It still trails the Phase-A caption + Marqo FashionCLIP baseline (headline R@10 = 0.533).

mentor's 2026-05-03 feedback (recorded in full at `Documentation/meeting_memo_20260503.md`): Plan-6 is **not really a two-tower model**. Its "target tower" is a **frozen FashionCLIP image encoder**, so the trainable Qwen2VL query tower is effectively being forced to imitate FashionCLIP's pre-existing embedding space. This caps the model's ceiling at FashionCLIP's discrimination ability and weakens the resume story ("you re-projected into someone else's space" vs. "you trained a new retrieval space end-to-end").

His proposed alternative — **the goal of this plan**:

> Replace the FashionCLIP target tower with **a second trainable Qwen2VL tower**. Query tower takes `(reference_image, modification_text)`. Target tower takes `(target_image, "Describe this image in detail.")`. Both produce L2-normalized embeddings via projection heads. Train contrastively with global multi-positive InfoNCE on triplet supervision. **Ground truth is the actual target image, NOT a FashionCLIP embedding.**

The critical conceptual point: the embedding space is **co-constructed during training by both towers**, not inherited from any frozen teacher.

| | Plan-6 (current) | Plan-10 V1 (this plan) |
|---|---|---|
| Target tower trainable? | ❌ frozen FashionCLIP | ✅ trainable Qwen2VL LoRA |
| Embedding space | inherited from FashionCLIP | constructed by both towers from scratch |
| Loss signal | "match FashionCLIP's coords for the right target" | "agree with the target tower on which (query, target) pairs are matched" |
| Ceiling | FashionCLIP's fashion-discrimination ability | Qwen2VL's discrimination + dataset signal |

### Why this is not "Qwen2VL chasing Qwen2VL" — the CLIP analogy

A natural worry: "if both towers are Qwen2VL, isn't this circular? Aren't we just training `Qwen2VL_query` to match `Qwen2VL_target`'s embeddings?"

**It is not circular**, because the target tower's embedding is **not fixed**. It is not "Qwen2VL learns to match the original pretrained Qwen2VL". It is "Qwen2VL_query and Qwen2VL_target **both learn together** to maximize the mutual information between `(reference + modification)` and `(target_image + description_prompt)` pairs from the dataset". The supervision comes from the **triplet labels** (which target image is the correct one for which query), not from any fixed teacher embedding.

**This is exactly how CLIP itself was originally trained** (Radford et al., February 2021, *"Learning Transferable Visual Models From Natural Language Supervision"*): an image encoder and a text encoder, **both trainable**, learning a shared embedding space from scratch via symmetric InfoNCE on 400M image-text pairs. It works extremely well. Plan-10 V1 applies the same training dynamic to our composed-retrieval task — the difference is only that each side processes image+text rather than image-only and text-only.

**Caveat.** CLIP used two completely separate encoders because the two sides had different modalities (you cannot share a ResNet with a Transformer). Our two towers both process image+text, so weight sharing between them is a legitimate architectural choice CLIP did not have. See §4.3 (Q1) for the trade-off.

**However**, this symmetry will break in the future V4 audio extension: the query tower will process **image + audio** while the target tower still processes **image + text**. Both V1 towers already share the same speechQwen2VL backbone (which supports both text and audio inputs natively via its added audio embedder), so V4 is **not** a backbone swap — it's an input-modality change on the query side only. This simplifies the V4 transition under either Option A or Option B: only the query-side input pipeline changes; no new model weights load.

## 4. Architecture — Two Towers, V1

### 4.1 Query Tower

```
(reference_image, modification_text)
      ↓
speechQwen2VL backbone (DanJZY/Qwen2-VL-7B-Speech + Stage-2 LoRA merged, frozen;
trainable retrieval LoRA on q_proj/k_proj/v_proj/o_proj, rank 32, alpha 64)
      ↓
last hidden state, pooled at last real token (flip+argmax over attention_mask)
      ↓  (B, 3584) bf16
Projection head: Linear(3584, 1024) → GELU → LayerNorm → Linear(1024, 512)   (fp32)
      ↓  (B, 512) fp32
L2-normalize  →  q_emb
```

Same hidden-state pooling as Plan-6 (the flip+argmax fix from `src/training/contrastive_model.py:169-181`).

### 4.2 Target Tower

```
(target_image, "Describe this image in detail.")
      ↓
speechQwen2VL backbone (same as query tower; same LoRA pattern)
      ↓
last hidden state, pooled at last real token (same pooling)
      ↓  (B, 3584) bf16
Projection head: Linear(3584, 1024) → GELU → LayerNorm → Linear(1024, 512)   (fp32)
      ↓  (B, 512) fp32
L2-normalize  →  t_emb
```

**Target prompt for V1 (locked):** `"Describe this image in detail."` (mentor's literal suggestion from memo §6). Alternate prompts deferred to V3.

### 4.3 Weight sharing — **Q1, TBD**

> **Both options below are spelled out so the rejected one can be deleted once decided. Full implementation walkthrough for both is in Appendix B.**

#### Option A — Shared backbone + separate LoRA adapters + separate heads

One `speechQwen2VL (DanJZY/Qwen2-VL-7B-Speech)` backbone in GPU memory (frozen base weights). Two LoRA adapters attached via PEFT: `lora_query` and `lora_target`. Two projection heads. The same backbone forward is invoked twice per training step, with `model.set_adapter("query")` before the query forward and `model.set_adapter("target")` before the target forward.

> **Gradient propagation footgun (verified in PEFT 0.17.1 — `BaseTunerLayer.set_adapter`)**: `set_adapter(name)` does TWO things — (1) marks `name` as the active adapter for forward-time application, AND (2) flips `requires_grad_(True)` on the active adapter and `requires_grad_(False)` on all inactive adapters. So calling `set_adapter("target")` before the target forward will silently set the query LoRA's `requires_grad=False`, and the query LoRA receives no gradient at `backward()` time even though it participated in the query forward.
>
> **Required workaround**: right before `loss.backward()`, re-assert `requires_grad=True` on all LoRA parameters across both adapters. Concretely:
>
> ```python
> q_emb = model.forward_query(...)   # set_adapter("query"); query.requires_grad = True; target.requires_grad = False
> t_emb = model.forward_target(...)  # set_adapter("target"); target.requires_grad = True; query.requires_grad = False
> # Re-enable grads on the deactivated query LoRA before backward:
> for n, p in model.backbone.named_parameters():
>     if "lora_" in n:
>         p.requires_grad = True
> loss = loss_fn(q_emb, t_emb, gather=True, target_ids=tid_tensor)
> accelerator.backward(loss)
> ```
>
> This is the canonical pattern documented in PEFT's docstring itself: *"this function will set the specified adapters to trainable. If this is not desired, [override `requires_grad` manually]."* Add a smoke-test assertion (`sum(p.requires_grad for n,p in model.named_parameters() if "lora_" in n) == expected_count`) before the first backward, since silent grad loss is the most insidious failure mode here. Option B has no equivalent footgun because each tower's parameters live in a separate model object and are never deactivated.

- **Resident VRAM:** ~15.3 GB (one backbone) + ~300 MB (two LoRA) + ~20 MB (two heads)
- **Per-GPU batch size:** 24–32 on A6000 (49 GB)
- **Effective negatives at bs=24 × 8 GPUs gather:** 192
- **Hardware:** A6000 + RTX 3090 (3090 needs bs reduction to 6–8/GPU)

#### Option B — Two completely separate Qwen2VL instances

Two independent `ContrastiveQwen2VL` objects: `query_model`, `target_model`. Each has its own backbone weights, own LoRA, own head. No PyTorch graph sharing; two model objects.

- **Resident VRAM:** ~30.3 GB (two backbones) + ~300 MB (two LoRA) + ~20 MB (two heads)
- **Per-GPU batch size:** 8–16 on A6000
- **Effective negatives at bs=8 × 8 GPUs gather:** 64
- **Hardware:** A6000 only (3090 will not fit)

The full deliberation (including future-V4 audio-swap implications) is preserved in **Appendix A**. The full implementation walkthrough is in **Appendix B**.

> Note: the precise breakdown of which sub-modules `target_modules=["q_proj","k_proj","v_proj","o_proj"]` actually attaches LoRA to (LM 112 / audio 96 / ViT 0 = 208 total per tower) is documented in `Documentation/Progress_10_20260512.md` §5. V1 keeps Plan-6's LoRA-target verbatim for strict comparability; narrowing to LM-only or extending to ViT is a V2/V3 question.

### 5.3 Training loop changes (`train_plan10.py` vs `train_plan5.py`)

```python
# Plan-6 (current train_plan5.py line ~404):
t_emb_np = np.stack([gallery_lookup[tid] for tid in batch["target_ids"]])
t_emb    = torch.from_numpy(t_emb_np).to(device)            # frozen FashionCLIP lookup

# Plan-10 V1 (train_plan10.py) — DDP-safe pattern:
# After accelerator.prepare wraps model, ALWAYS call model(...) not model.forward_*(...).
# Submodule method calls bypass DDP.forward dispatch and may fail under multi-GPU.
q_emb, t_emb = model(
    batch["cand_images"],
    batch["mod_texts"],
    batch["tgt_images"],
)

tid_tensor = torch.tensor([tid_to_idx[t] for t in batch["target_ids"]],
                          dtype=torch.int64, device=device)
loss = loss_fn(q_emb, t_emb, gather=args.gather, target_ids=tid_tensor)
accelerator.backward(loss)
optimizer.step(); optimizer.zero_grad()
```

The parent `forward(cand_images, mod_texts, tgt_images) → (q_emb, t_emb)` is defined on both `TwoTowerSharedBackbone` (Option A) and `TwoTowerSeparateBackbones` (Option B) in Appendix B. It handles the per-tower instruction templates internally and (for Option A) takes care of the PEFT `requires_grad` reset before returning. The eval helpers (`encode_query` / `encode_target`) are reserved for the unwrapped model after `accelerator.unwrap_model(...)` and use `torch.inference_mode()`.

### 5.4 Eval gallery refresh cadence

The target tower changes every step, so the precomputed `.npy` cache pattern from Plan-6 is no longer valid for retrieval eval. V1 uses **end-of-epoch refresh** with an explicit **all-rank distributed protocol** (the existing eval block in `train_plan5.py:463` is gated under `accelerator.is_main_process`, but gallery encoding must NOT be — only rank-0-encoding would either crash or produce a partial gallery).

**Refresh cadence:**

- **Startup:** encode the full 59k gallery once via `encode_gallery_with_tower(...)`. ~3–4 min on 8× A6000-class with all-rank sharding. Cache to `<run_dir>/gallery_emb_epoch0.npy` + `<run_dir>/gallery_emb_epoch0.meta.json` for safety / resume (`<run_dir>` per §5.5: `runs_local_plan10/<name>/` on server10, `runs/plan10/<name>/` on server6).
- **End of each full epoch (NOT every 0.5-epoch eval):** re-encode the full 59k gallery, overwrite the in-memory `gallery_lookup` dict and `CaptionDB`. ~3–4 min × 18 epochs ≈ ~1 hour added to total wall time.
- **Mid-epoch 0.5-epoch evals:** use the most recent end-of-epoch cache. R@K at those intermediate evals reflects "last epoch's target tower" but is still a useful trend signal.
- **Dev loss (`run_dev_loss`):** always uses the on-the-fly target tower forward on the 500 dev positives (~10s). Always fresh; the primary overfitting signal.

**Distributed protocol for `encode_gallery_with_tower`** (called outside the `accelerator.is_main_process` gate):

The gallery cardinality (59,082) is **not divisible by 8**, which rules out the naive equal-stride sharding pattern — different ranks would produce different-shaped tensors and `accelerator.gather` would fail. The simplest correct pattern is **pad to ceiling, gather, then truncate**, with paired-id tracking so the row-to-id mapping is recoverable. Pseudo-code (illustrative — the implementer must verify against a single-rank reference):

```python
import math
from src.training.target_cache import _gallery_ids_and_paths  # canonical source

def encode_gallery_with_tower(model, gallery_ids, gallery_paths,
                              batch_size, accelerator):
    """
    model: the UNWRAPPED two-tower module (accelerator.unwrap_model(prepared_model)).
           Must expose `encode_target(images) -> (B, D)` (defined on both
           TwoTowerSharedBackbone and TwoTowerSeparateBackbones in Appendix B).
    gallery_ids, gallery_paths: list[str] of length N in canonical row order
    (use src/training/target_cache.py:_gallery_ids_and_paths to build them).
    Must be called from ALL ranks (do NOT gate on is_main_process).
    """
    model.eval()
    rank, world = accelerator.process_index, accelerator.num_processes
    N = len(gallery_ids)
    pad_to = math.ceil(N / world) * world
    # Pad with copies of the last index so all ranks process the same count
    padded_ids   = gallery_ids   + [gallery_ids[-1]]   * (pad_to - N)
    padded_paths = gallery_paths + [gallery_paths[-1]] * (pad_to - N)

    my_idx       = list(range(rank, pad_to, world))      # equal length on every rank
    my_paths     = [padded_paths[i] for i in my_idx]
    my_orig_idx  = torch.tensor(my_idx, dtype=torch.long, device=accelerator.device)

    my_embs = []
    with torch.inference_mode():
        for c in range(0, len(my_paths), batch_size):
            imgs = [Image.open(p).convert("RGB") for p in my_paths[c:c+batch_size]]
            embs = model.encode_target(imgs)  # eval helper; uses locked target prompt internally
            my_embs.append(embs.detach().to(accelerator.device).float())
    my_embs = torch.cat(my_embs, dim=0)                  # (pad_to/world, 512)

    # All-gather paired (embedding, original-index) so we can re-sort canonically.
    all_embs = accelerator.gather(my_embs)               # (pad_to, 512)
    all_idx  = accelerator.gather(my_orig_idx)           # (pad_to,)

    # Reorder to original canonical order; drop padded rows
    perm = torch.argsort(all_idx)
    full = all_embs[perm][:N]                            # (N, 512) in canonical row order

    accelerator.wait_for_everyone()
    if accelerator.is_main_process:
        np.save(out_dir / f"gallery_emb_epoch{epoch}.npy", full.cpu().numpy())
        json.dump({"gallery_ids": gallery_ids,
                   "embedding_dim": full.shape[1],
                   "n_images": N,
                   "image_hash": _ids_hash(gallery_ids)},  # same hash as target_cache.py
                  open(out_dir / f"gallery_emb_epoch{epoch}.meta.json", "w"))
    return full   # all ranks now have the full gallery embeddings
```

Notes for the implementer:
- The pad-and-gather pattern is the cleanest way to handle `N % world != 0`. The alternative — `accelerator.gather_for_metrics(...)` with an unequal-shape input — only works in newer Accelerate; check the version pinned in `requirements.txt` before relying on it.
- `model.encode_target(imgs)` (eval-only helper, `@torch.inference_mode`) lives on the parent two-tower module (both `TwoTowerSharedBackbone` and `TwoTowerSeparateBackbones`) and internally uses the locked target prompt. Don't try to thread the prompt through the helper signature. Call this on the unwrapped model (`accelerator.unwrap_model`) — DDP semantics don't apply during eval.
- Add a smoke-test assertion against single-rank encoding on a 32-image subset before trusting the production refresh:
  `assert torch.allclose(full[i], single_rank_encode([gallery_ids[i]])[0], atol=1e-4)` for a few sampled `i`. The §10.1 first-eval trigger naturally hits this codepath ~2 min into the run, so a sharding bug surfaces before the 25-min mark of the actual first epoch.

This cadence trades small intra-epoch staleness for a 12× wall-time reduction vs. "re-encode every eval".

## 10. Verification

### 10.1 First-eval-at-step-N + explicit grad-receipt assertion

Two cheap, real-code-path checks at the start of every run, instead of a parallel smoke harness:

**(a) First-eval-at-step-N — catches validation OOM / sharding bugs in ~2 min**

- New CLI flag: `--first-eval-step` (default `5`; set to `0` to disable).
- After step N (default 5), trigger the **existing** eval pipeline once — gallery encoding (the distributed protocol in §5.4) + `run_dev_loss` + `run_retrieval_eval` — then resume the normal `--eval-epochs 0.5` cadence.
- Reuses the actual eval code paths. No parallel smoke harness to keep in sync. The first 5 training steps also implicitly serve as a training-side sanity (shape errors, adapter switching, backprop wiring all crash before step 5 if they're going to crash).

```python
def _should_eval(step):
    if args.first_eval_step > 0 and step == args.first_eval_step:
        return True
    return step > 0 and step % steps_per_eval == 0
```

**(b) Grad-receipt assertion after step 1 — catches the PEFT `set_adapter` silent-grad-loss footgun (§4.3)**

A successful eval (a) does NOT prove gradients flowed correctly during training, because eval is inference-only. The silent grad-loss bug in Option A is invisible to eval — training appears to run, loss decreases on the active adapter, but the inactive adapter never updates. Add this assertion **immediately after the first `optimizer.step()`** in `train_plan10.py`:

> **Why this needs to be carefully written.** Two subtleties (both raised in the 2026-05-11 cross-review):
> 1. **Do NOT filter by `p.requires_grad`** to enumerate which params to check. The PEFT footgun is exactly that `requires_grad` was wrongly flipped to False — filtering by it would silently skip the broken adapter and the check would pass with only the working adapter visible.
> 2. **Do NOT assert nonzero gradient on `lora_A` at step 1.** PEFT 0.17.1's `reset_lora_parameters` initializes `lora_A` with Kaiming (nonzero) and `lora_B` with zeros (verified in `peft/tuners/lora/layer.py`). So on the first backward, `δ = B·A·x` with `B = 0` ⇒ `dloss/dA = (...) · B^T = 0` exactly. The `lora_A` zero-grad at step 1 is the correct behavior, not a bug. Check `lora_B` params for nonzero grad instead — they always receive gradient when the adapter is correctly trained.

```python
# Right after the first optimizer.step() call (global_step == 1):
if global_step == 1 and accelerator.is_main_process:
    unwrapped = accelerator.unwrap_model(model)
    expected_adapters = {"query", "target"}

    def _adapter_for(name):
        # Option A: parent has ONE backbone with two PEFT adapters named "query"/"target".
        # Param names contain ".query." or ".target." (the adapter name within PEFT).
        if ".query." in name:  return "query"
        if ".target." in name: return "target"
        # Option B: parent has two TOWERS (query_tower / target_tower), each its own PEFT
        # model with adapter_name="default". Param names start with "query_tower." or
        # "target_tower." module-prefix.
        if name.startswith("query_tower.")  or ".query_tower."  in name: return "query"
        if name.startswith("target_tower.") or ".target_tower." in name: return "target"
        return None

    by_adapter = {a: {"lora_A": [], "lora_B": []} for a in expected_adapters}
    for n, p in unwrapped.named_parameters():
        if "lora_" not in n:
            continue
        adapter = _adapter_for(n)
        if adapter is None:
            continue
        kind = "lora_A" if "lora_A" in n else "lora_B" if "lora_B" in n else None
        if kind is None:
            continue
        by_adapter[adapter][kind].append((n, p))

    for adapter in expected_adapters:
        a_count = len(by_adapter[adapter]["lora_A"])
        b_count = len(by_adapter[adapter]["lora_B"])
        assert a_count > 0 and b_count > 0, (
            f"[Plan-10 grad-receipt] adapter '{adapter}' has no lora_A/lora_B "
            f"params (A={a_count}, B={b_count}). Was add_adapter called?"
        )
        # Every adapter param must be trainable (catches the requires_grad footgun)
        for n, p in by_adapter[adapter]["lora_A"] + by_adapter[adapter]["lora_B"]:
            assert p.requires_grad, (
                f"[Plan-10 grad-receipt] {n} has requires_grad=False at step 1. "
                f"This is the PEFT set_adapter footgun (see §4.3). Re-enable "
                f"requires_grad on ALL LoRA params before optimizer.step()."
            )
        # lora_B should have nonzero gradient on step 1 (lora_A may be zero because
        # PEFT inits B=0 → dloss/dA = 0 on the first backward, by design).
        b_with_grad = sum(
            1 for _, p in by_adapter[adapter]["lora_B"]
            if p.grad is not None and p.grad.abs().sum().item() > 0
        )
        assert b_with_grad == b_count, (
            f"[Plan-10 grad-receipt] adapter '{adapter}': only "
            f"{b_with_grad}/{b_count} lora_B params received gradient. "
            f"Did the adapter participate in the forward graph at all?"
        )
    print(f"[Plan-10] step 1 grad-receipt OK: "
          f"{ {a: f'{len(by_adapter[a][\"lora_B\"])} lora_B nonzero' for a in expected_adapters} }")
```

This is ~25 lines added once and runs once per training session. Without it, an Option-A run with the `requires_grad` reset missing would train only the target adapter for the entire 18 epochs and the bug would surface as "headline R@10 mysteriously low" 8 hours later. The check is robust to PEFT's LoRA init convention (zero gradient on `lora_A` at step 1 is correct) and does NOT use `requires_grad` as a filter (the very thing the footgun corrupts).

The full Q1 / Q2 / Q3 deliberation including options considered, trade-offs, and the user's reasoning. Preserved here so the thinking process is not lost if Q1 is revisited.

### Q1 — Weight sharing strategy

#### Option A — Shared backbone + separate LoRA + heads (open)

One Qwen2VL-7B backbone in memory (frozen base). Two LoRA adapters: `lora_query` and `lora_target`. Two projection heads. PEFT's `set_adapter()` toggles which adapter is applied per forward pass.

- VRAM: ~15 GB resident + ~300 MB LoRA + heads. A6000 supports bs=24–32/GPU.
- Pros: Cheapest VRAM that still allows specialization. Adapters can specialize. Faster startup.
- Cons: Both towers constrained to same backbone's representation space. Slightly more code for adapter switching.

#### Option B — Two completely separate Qwen2VL instances (open)

Two independent `ContrastiveQwen2VL` objects. No PyTorch graph sharing.

- VRAM: ~30 GB resident. A6000 supports bs=8–16/GPU. **Will not fit on RTX 3090 (24 GB).**
- Pros: Most literal reading of "another Qwen2VL". Most expressive — target representation space genuinely independent.
- Cons: 2× VRAM. Shrinks effective negatives (64–128 vs Plan-6's 512). Locks Plan-10 to A6000-class hardware.

#### Option C — Fully shared (Siamese) [RULED OUT by user, 2026-05-10]

Single model, single LoRA, single projection head. Called twice per batch.

- VRAM: ~15 GB; bs=48–64/GPU.
- Pros: Simplest. Largest batch.
- Cons: No way for the model to specialize between query and target inputs. Composed retrieval is asymmetric. Furthest from mentor's stated recommendation — weakest resume story.

**User's stance (2026-05-10):** Option C ruled out. Option A is recommended by assistant; choice between A and B still open.

### Q1 follow-up — How weight sharing interacts with V4 audio extension

User's question (2026-05-10): *"In the future we might change the text part to audio. Would option A be limited? Can we simply swap text with audio if we use option A's architecture?"*

**Updated answer (post 2026-05-11 cross-review)**: standardizing V1 on **speechQwen2VL** for both towers simplifies V4 dramatically — V4 is no longer a backbone swap at all, it's an input-modality change on the query side. The backbone already speaks both text and audio. Under either Option A or Option B:

- V4 query input changes from `(reference_image, modification_text)` → `(reference_image, modification_audio)`.
- V4 target input stays `(target_image, "Describe this image in detail.")` — text on the target side.
- No backbone load / swap is needed in either option.

**Option A specifically benefits from this in V4.** Because the backbone is shared and the target side processes a literally unchanged input (same image, same generic text prompt) through a literally unchanged `target` adapter, the target tower's behavior is **bit-exact identical** before and after V4 — no possibility of subtle drift in target embeddings from a separate retraining run. Only the `query` adapter (and possibly the audio embedder of the shared backbone, if we choose to update it) sees gradient in V4. Option B is functionally equivalent in V4 outcomes, but the target tower lives in a separate `nn.Module` instance whose weights are managed independently; preserving target-side invariance requires deliberate care (load the same V1 target tower checkpoint and freeze it during V4 finetuning). Option A makes that invariance the default rather than a discipline.

(The Q1 decision still rests primarily on the V1 trade-offs — VRAM headroom for negatives, PEFT footgun risk, code complexity — but A's V4 target-invariance property is a strict architectural pro that the earlier "B has cleaner audio swap" framing got backwards.)

(The original V4-swap analysis below is retained for historical traceability of the deliberation.)

> Original answer (2026-05-10, before V1 standardized on speechQwen2VL): Option A would require swapping the shared backbone from vanilla Qwen2VL to speechQwen2VL and retraining both LoRAs; Option B would swap only the query backbone and reuse the target LoRA verbatim. Comparison table is now stale and superseded by the V1 backbone standardization.

### 1. Model setup

#### Option A — Shared backbone + separate LoRA adapters + separate heads

```python
# src/training/two_tower_model.py (new file for Plan 10)
# Mirrors ContrastiveQwen2VL's __init__ (src/training/contrastive_model.py:52-109):
# speechQwen2VL base + Stage-2 LoRA merged into backbone, fresh retrieval LoRAs
# added on top with task_type="CAUSAL_LM", input-grad enablement, gradient
# checkpointing, use_cache=False, and lm_head stub replacement.

from transformers import Qwen2VLForConditionalGeneration, AutoProcessor
from peft import LoraConfig, PeftModel, get_peft_model
import torch
import torch.nn as nn

BASE_REPO = "DanJZY/Qwen2-VL-7B-Speech"
STAGE2_LORA_REPO = "DanJZY/Qwen2-VL-7B-Speech-LoRA"
QWEN2VL_HIDDEN_DIM = 3584

LORA_CFG_KWARGS = dict(
    r=32, lora_alpha=64,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    lora_dropout=0.0, bias="none",
    task_type="CAUSAL_LM",   # NOT FEATURE_EXTRACTION — matches ContrastiveQwen2VL
)

class TwoTowerSharedBackbone(nn.Module):
    def __init__(self, d_target=512, device_map=None):
        super().__init__()

        # Step 1: load base + Stage-2 LoRA, merge (mirrors _load_qwen2vl_base)
        from src.training.contrastive_model import _load_qwen2vl_base
        backbone, processor = _load_qwen2vl_base(
            base_repo=BASE_REPO, lora_repo=STAGE2_LORA_REPO,
            merge_stage2=True, device_map=device_map,
        )

        # Step 2: attach the FIRST retrieval LoRA adapter ("query")
        self.backbone = get_peft_model(backbone, LoraConfig(**LORA_CFG_KWARGS),
                                        adapter_name="query")

        # Step 3: attach the SECOND retrieval LoRA adapter ("target")
        self.backbone.add_adapter("target", LoraConfig(**LORA_CFG_KWARGS))

        # CRITICAL — required for gradient checkpointing under PEFT
        self.backbone.enable_input_require_grads()
        self.backbone.gradient_checkpointing_enable(
            gradient_checkpointing_kwargs={"use_reentrant": False})
        self.backbone.config.use_cache = False

        # Replace lm_head with 1-output stub (saves ~2 GB at bs=16, seq=512;
        # we only consume hidden_states[-1], never logits)
        _lm_head = self.backbone.base_model.model.lm_head
        self.backbone.base_model.model.lm_head = nn.Linear(
            QWEN2VL_HIDDEN_DIM, 1, bias=False, dtype=torch.bfloat16
        ).to(next(_lm_head.parameters()).device)
        del _lm_head

        # FOOTGUN: add_adapter("target") deactivates "query" via requires_grad.
        # Re-enable BOTH adapters' LoRA params now so the optimizer sees them.
        for n, p in self.backbone.named_parameters():
            if "lora_" in n:
                p.requires_grad = True

        # Two independent projection heads (fp32)
        self.head_query  = _ProjectionHead(d_target).float()
        self.head_target = _ProjectionHead(d_target).float()
        self.processor = processor

    def _forward_one_side(self, images, texts, head, instruction_template):
        # Build messages with chat template (required for Qwen2VL image-token wiring;
        # mirrors ContrastiveQwen2VL.forward at src/training/contrastive_model.py:129-181)
        messages_list = []
        for img, txt in zip(images, texts):
            instr = instruction_template.format(txt=txt) if "{txt}" in instruction_template \
                    else instruction_template
            messages_list.append([{
                "role": "user",
                "content": [
                    {"type": "image", "image": img},
                    {"type": "text",  "text": instr},
                ],
            }])
        text_inputs = [self.processor.apply_chat_template(
                          m, tokenize=False, add_generation_prompt=True)
                       for m in messages_list]
        device = next(self.backbone.parameters()).device
        inputs = self.processor(text=text_inputs, images=images,
                                return_tensors="pt", padding=True)
        inputs = {k: v.to(device) if hasattr(v, "to") else v for k, v in inputs.items()}
        out = self.backbone(**inputs, output_hidden_states=True, use_cache=False)
        # flip+argmax pooling (Plan-6's EOS-fix; left-padded sequences)
        last_hs = out.hidden_states[-1]
        seq_ends = (inputs["attention_mask"].shape[1] - 1
                    - inputs["attention_mask"].flip(dims=[1]).long().argmax(dim=1))
        pooled = last_hs[torch.arange(len(images), device=device), seq_ends, :]
        return F.normalize(head(pooled.float()), dim=-1)

    QUERY_TMPL  = ("Given this product image, find the item that looks like "
                   "the image but with the following modification: {txt}")
    TARGET_TMPL = "Describe this image in detail."

    def forward(self, cand_images, mod_texts, tgt_images):
        """DDP-safe entry point. The training loop calls model(cand, mod, tgt)
        AFTER accelerator.prepare wraps this module — that routes correctly
        through DDP.forward, unlike submodule method calls.

        Also handles the PEFT set_adapter requires_grad footgun by restoring
        all-LoRA requires_grad=True before returning.
        """
        q_emb = self._forward_query(cand_images, mod_texts)
        t_emb = self._forward_target(tgt_images)
        # PEFT set_adapter footgun cleanup: re-enable both adapters' grads
        for n, p in self.backbone.named_parameters():
            if "lora_" in n:
                p.requires_grad = True
        return q_emb, t_emb

    def _forward_query(self, images, texts):
        self.backbone.set_adapter("query")
        return self._forward_one_side(images, texts, self.head_query, self.QUERY_TMPL)

    def _forward_target(self, images):
        self.backbone.set_adapter("target")
        return self._forward_one_side(images, [""] * len(images), self.head_target,
                                      self.TARGET_TMPL)

    # Eval-only helpers; call ONLY on the unwrapped model (after
    # accelerator.unwrap_model). DDP semantics do not apply here.
    @torch.inference_mode()
    def encode_query(self, images, texts):
        return self._forward_query(images, texts)

    @torch.inference_mode()
    def encode_target(self, images):
        return self._forward_target(images)
```

**What's in memory now:**
- 1× Qwen2VL backbone (~15 GB bf16, frozen)
- 2× LoRA adapters (~150 MB each, trainable)
- 2× projection heads (~10 MB each, trainable, fp32)
- **Total resident:** ~15.3 GB

#### Option B — Two separate Qwen2VL instances

```python
# src/training/two_tower_model.py (new file for Plan 10)

class TwoTowerSeparateBackbones(nn.Module):
    """
    Two independent speechQwen2VL+LoRA instances. The query tower reuses
    ContrastiveQwen2VL verbatim (its forward already wraps mod_text in the
    Plan-6 instruction template). The target tower needs a DIFFERENT
    instruction template ("Describe this image in detail.") instead of
    the Plan-6 wrap, so we cannot reuse ContrastiveQwen2VL.forward as-is —
    instead we instantiate ContrastiveQwen2VL (to inherit __init__'s
    backbone/LoRA/checkpointing setup) and override forward.
    """
    def __init__(self, d_target=512, device_map=None):
        super().__init__()
        self.query_tower = ContrastiveQwen2VL(d_target=d_target,
                                              device_map=device_map)
        # Target tower: same init, then bind a different forward
        self.target_tower = ContrastiveQwen2VL(d_target=d_target,
                                               device_map=device_map)
        # Override the target tower's forward to use the locked target prompt
        # instead of the Plan-6 modification-instruction wrap.
        self.target_tower.forward = self._target_forward.__get__(
            self.target_tower, type(self.target_tower))

    @staticmethod
    def _target_forward(self, images, _ignored_texts, max_mod_len=512):
        """Replaces ContrastiveQwen2VL.forward for the target tower.

        Same pooling, projection, normalization as the query forward;
        only the instruction text changes: every sample sees
        "Describe this image in detail." regardless of input text.
        """
        messages_list = [[{
            "role": "user",
            "content": [
                {"type": "image", "image": img},
                {"type": "text",  "text": "Describe this image in detail."},
            ],
        }] for img in images]

        text_inputs = [self.processor.apply_chat_template(
                         m, tokenize=False, add_generation_prompt=True)
                       for m in messages_list]

        device = next(self.vlm.parameters()).device
        inputs = self.processor(text=text_inputs, images=images,
                                return_tensors="pt", padding=True)
        inputs = {k: v.to(device) if hasattr(v, "to") else v
                  for k, v in inputs.items()}
        out = self.vlm(**inputs, output_hidden_states=True, use_cache=False)
        last_hs = out.hidden_states[-1]
        seq_ends = (inputs["attention_mask"].shape[1] - 1
                    - inputs["attention_mask"].flip(dims=[1]).long().argmax(dim=1))
        pooled = last_hs[torch.arange(len(images), device=device), seq_ends, :]
        return F.normalize(self.proj(pooled.float()), dim=-1)

    def forward(self, cand_images, mod_texts, tgt_images):
        """DDP-safe entry point. Same contract as Option A's forward."""
        q_emb = self.query_tower(cand_images, mod_texts)         # Plan-6 instruction wrap
        t_emb = self.target_tower(tgt_images, [""] * len(tgt_images))  # bound _target_forward
        return q_emb, t_emb

    # Eval-only helpers; call ONLY on the unwrapped model.
    @torch.inference_mode()
    def encode_query(self, images, texts):
        return self.query_tower(images, texts)

    @torch.inference_mode()
    def encode_target(self, images):
        return self.target_tower(images, [""] * len(images))  # bound _target_forward uses locked prompt
```

**Why we can't just call `target_tower(images, ["Describe this image in detail."] * B)`:** the existing `ContrastiveQwen2VL.forward` (`src/training/contrastive_model.py:129-141`) wraps any non-empty text as `"Given this product image, find the item that looks like the image but with the following modification: {txt}"`. So the literal call would send the target tower *"Given this product image, find the item that looks like the image but with the following modification: Describe this image in detail."* — wrong. Passing empty string would trigger the fallback *"Describe the product shown in this image."* which is also not the locked target prompt. Hence the explicit `_target_forward` override above.

**What's in memory now:**
- 2× Qwen2VL backbones (~15 GB each, frozen)
- 2× LoRA adapters (~150 MB each, trainable, one per tower)
- 2× projection heads (~10 MB each, trainable, fp32, one per tower)
- **Total resident:** ~30.3 GB

That ~15 GB difference is the entire VRAM trade-off in one number.

### 2. One training step (forward → loss → backward)

Identical for both Option A and Option B — both expose a DDP-safe `forward(cand_images, mod_texts, tgt_images) → (q_emb, t_emb)` (defined in the class blocks above) so the training loop never reaches into submodules after `accelerator.prepare`:

```python
def training_step(model, batch, loss_fn, tid_to_idx, device, accelerator):
    # Single entry point — goes through DDP.forward correctly:
    q_emb, t_emb = model(
        batch["cand_images"],
        batch["mod_texts"],
        batch["tgt_images"],
    )
    # For Option A, the model.forward already restored requires_grad=True on
    # both adapters' LoRA params before returning. For Option B, there's no
    # adapter switching, so requires_grad never flipped.

    tid_tensor = torch.tensor([tid_to_idx[t] for t in batch["target_ids"]],
                              dtype=torch.int64, device=device)
    loss = loss_fn(q_emb, t_emb, gather=True, target_ids=tid_tensor)
    accelerator.backward(loss)
```

The step-1 grad-receipt assertion (§10.1.b) runs immediately after the first `optimizer.step()` to verify that both adapters' `lora_B` parameters received gradient (Option A — defends against the PEFT footgun) or that both towers' params received gradient (Option B — sanity check).

### 3. Optimizer construction

#### Option A

```python
# PEFT's multi-adapter param naming: lora_A.<adapter_name>.weight / lora_B.<adapter_name>.weight.
# We filter by adapter_name + "lora_" substring.
def _lora_params(model, adapter_name):
    return [p for n, p in model.named_parameters()
            if "lora_" in n and f".{adapter_name}." in n]

optimizer = torch.optim.AdamW([
    {"params": _lora_params(model.backbone, "query"),  "lr": 2e-5},
    {"params": _lora_params(model.backbone, "target"), "lr": 2e-5},
    {"params": model.head_query.parameters(),  "lr": 1e-4},
    {"params": model.head_target.parameters(), "lr": 1e-4},
    {"params": [loss_fn.logit_scale],          "lr": 2e-5},  # learned temperature (Plan-6 carry-over)
], weight_decay=0.01)
```

Five param groups. The frozen backbone never appears. `loss_fn.logit_scale` is included verbatim from Plan-6 (`src/training/train_plan5.py:309-314`); omitting it freezes `tau_inv` at init, which §7.1 claims is logged as a trained quantity.

**Important — set `requires_grad=True` on ALL LoRA params after model construction.** PEFT's `add_adapter` activates only the newly-added adapter and deactivates the previous one. Before optimizer construction, do:

```python
for n, p in model.backbone.named_parameters():
    if "lora_" in n:
        p.requires_grad = True
```

Without this, the `_lora_params` filter may return an empty list for whichever adapter was deactivated by `add_adapter`. Same defensive pattern is needed at every `set_adapter` boundary during training (see §4.3 Option A grad-propagation footgun callout).

#### Option B

```python
q_lora, q_proj = model.query_tower.trainable_parameters()   # reuses ContrastiveQwen2VL helper
t_lora, t_proj = model.target_tower.trainable_parameters()

optimizer = torch.optim.AdamW([
    {"params": q_lora,                  "lr": 2e-5},
    {"params": q_proj,                  "lr": 1e-4},
    {"params": t_lora,                  "lr": 2e-5},
    {"params": t_proj,                  "lr": 1e-4},
    {"params": [loss_fn.logit_scale],   "lr": 2e-5},  # learned temperature (Plan-6 carry-over)
], weight_decay=0.01)
```

Five param groups, sourced from two separate model objects via the existing `ContrastiveQwen2VL.trainable_parameters()` helper. Same `logit_scale` parameter as Option A. Option B has **no** `set_adapter` grad-disable footgun — each tower's LoRA lives in a separate `PeftModel` and stays trainable throughout. Gradient cost is roughly twice Option A's (two backbones' worth of activations to free during backward), resident memory is doubled.

### 4. Key gotchas

**Option A gotchas**

1. **PEFT's `set_adapter` is a global switch on the wrapped model.** If you forget the second `set_adapter("target")` call before the target forward, you accidentally apply `lora_query` to the target inputs and the model trains garbage. Easy to add an assertion.
2. **Checkpoint format is non-obvious.** When saving, PEFT writes adapters to separate subdirectories: `ckpt_dir/query/adapter_model.safetensors` and `ckpt_dir/target/adapter_model.safetensors`. Loading requires `model.load_adapter(path, adapter_name="query")` then `model.load_adapter(path, adapter_name="target")`. Two of these calls per resume.
3. **Both adapters have to fit in `requires_grad=True` simultaneously.** PEFT supports this (it's the standard multi-adapter pattern), but worth a sanity check: `sum(p.requires_grad for p in model.parameters())` should be ~2× a single-adapter run.
4. **`accelerator.prepare()` wraps the whole model once.** No special handling needed for the two adapters — they're inside one `nn.Module`.

**Option B gotchas**

1. **Two backbones means two HuggingFace cache hits at startup.** Slower cold start; also blows through your local HF cache faster.
2. **`accelerator.prepare(model)` wraps both towers when they're attributes of a parent `nn.Module`.** No special handling.
3. **Per-step memory is dominated by activations, not weights.** Even though Option B has 2× the resident weights, the per-step compute is the same (two forward passes either way) — the difference is the static cost.
4. **Saving is two full PEFT adapter directories + two head state_dicts.** Roughly equivalent on-disk to Option A but in separate trees.

### Side-by-side comparison

| | Option A | Option B |
|---|---|---|
| Resident VRAM (model only) | ~15.3 GB | ~30.3 GB |
| Forward passes per step | 2 | 2 |
| Activations held during backward | 2× a query+target pass | 2× a query+target pass (same) |
| Realistic bs/GPU on A6000 (49 GB) | 24–32 | 8–16 |
| Effective negatives (× 8 GPUs, gather) | 192–256 | 64–128 |
| Code complexity | Adapter switching (1 line per forward) | Two model calls (trivial) |
| Hardware compatibility | A6000 + 3090 (with bs reduction) | A6000 only |
| V4 audio swap | **Query tower's modification input changes from text → audio** (`(reference_image, mod_text)` → `(reference_image, mod_audio)`); **target tower input unchanged** (`(target_image, "Describe this image in detail.")`). Backbone (speechQwen2VL) handles audio natively — no backbone swap. Target tower's behavior is **bit-exact identical** before and after V4. | Same query-side change (text → audio); same unchanged target input. Target-tower invariance is functionally equivalent but requires deliberate care — target lives in a separate `nn.Module` instance, so V4 finetuning has to explicitly freeze it (or accept independent drift). |
| Resemblance to DPR-shared | Yes (DPR with shared encoder + asymmetric adapters) | No |
| Resemblance to DPR-separate | No | Yes |

**Bottom line.** Both implementations are roughly the same number of lines of code. Real trade-offs:
- **VRAM headroom** (A is 2× cheaper → 3× larger effective batch on the same hardware; Plan-7 showed 128-vs-512 negatives matters).
- **Gradient isolation** (B's gradient paths are fully independent; A's are gradient-isolated *if* the PEFT `set_adapter` footgun is mitigated — see §4.3 / Appendix B walkthrough).
- **V4 target invariance** (A makes it the default; B makes it a discipline — see Appendix A Q1 follow-up).
