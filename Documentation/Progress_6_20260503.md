## 1. Context and Motivation

Plan-5 runs were corrupted by two bugs present throughout all training:

1. **EOS pooling bug** — `sum-1` indexed PAD tokens for left-padded sequences. Fixed in `contrastive_model.py` to `flip+argmax`. Same ckpt_epoch3 weights: 0.212 → **0.310** dev R@10 after fix.
2. **Duplicate target false negatives** — 53,686 train triplets / 28,252 unique targets (~47% collision). Diagonal-only InfoNCE penalizes same-target pairs as negatives in ~99% of batches.

Plan-6 trains from epoch 0 with both fixed. All Plan-5 checkpoints should be considered invalid for fair comparison against Plan-6 results.

## 2. Implementation (completed before first run)

### `src/training/loss.py` — Multi-positive InfoNCE

Replaced diagonal-only cross-entropy labels with a masked log-sum-exp formulation that treats all (i, j) pairs sharing the same `target_id` as co-positives.

**New functions:**

`_gather_tensor(x)` — all-gathers a tensor across ranks. Float tensors use `GatherLayer` (gradient flows back to originating rank). Integer tensors (`target_ids`) use plain `dist.all_gather` (no gradient needed).

`_multi_positive_nce(logits, pos_mask)` — for each row i:
```
loss[i] = logsumexp(logits[i, all j]) − logsumexp(logits[i, j ∈ positives])
```
Positives are masked-in; non-positives are filled with `-inf` before `logsumexp`. Returns row mean.

**`SymmetricInfoNCE.forward` new signature:**
```python
def forward(self, q, t, gather=False, target_ids=None) -> torch.Tensor
```
- `target_ids`: `(B,)` int64 gallery indices. When provided, gathers across ranks and builds `pos_mask = (target_ids.unsqueeze(1) == target_ids.unsqueeze(0))`. Loss = average of `_multi_positive_nce` in q→t and t→q directions.
- When `target_ids=None`: falls back to diagonal cross-entropy (Plan-5 behavior, used in smoke test).

### `src/training/train_plan5.py` — tid_to_idx plumbing

Added one line after `make_gallery_lookup`:
```python
tid_to_idx: dict[str, int] = {gid: i for i, gid in enumerate(gallery_ids)}
```

In the training loop, batch's `target_ids` (list of str) are converted to an int64 tensor and passed to the loss:
```python
tid_tensor = torch.tensor(
    [tid_to_idx[tid] for tid in target_ids], dtype=torch.int64, device=device
)
loss = loss_fn(q_embs, t_embs, gather=args.gather, target_ids=tid_tensor)
```

Smoke test is unaffected — it calls `loss_fn(q_embs, t_embs, gather=gather)` without `target_ids`, falling back to diagonal labels (correct since smoke items have distinct target_ids by construction).

## 6. Conclusions

### Server 10 (bs=64, 512 effective) — COMPLETE

**Best checkpoint: step 1664 (epoch 16.0), headline R@10 = 0.402, headline R@50 = 0.646.**

Plateau clearly reached: headline R@10 oscillated 0.381–0.402 from epoch 10 onward with no upward trend. Training loss continued to fall (1.45 at end) while eval metrics stopped responding — model has saturated the in-batch objective but generalization is maxed out for this configuration. No further training needed.

Phase-A gap: **0.402 vs 0.533 = −0.131** — cannot be closed by more epochs.

### Server 6 (bs=16, 128 effective) — COMPLETE

**Best checkpoint: ckpt_epoch10 (step 4190), dev R@10 = 0.368, headline R@10 = 0.387, headline R@50 = 0.625.**

Clear overfitting post-epoch 10: headline R@50 declined from 0.641 (epoch 6.5) to 0.582 (epoch 17.5), and headline R@10 from 0.387 (epoch 10) to 0.344 (epoch 17.5). Training loss continued falling (0.46→0.28) while eval metrics declined — smaller effective batch (128 vs 512) provides less regularization, leading to earlier and more severe overfitting compared to server 10.

Phase-A gap: **0.387 vs 0.533 = −0.146** — cannot be closed by more epochs.

Dev peak at step 3143 (dev R@10=0.378) has no saved checkpoint; nearest saved is ckpt_epoch10 (dev 0.368).

### Plateau vs. overfit comparison

- **Server 10 (512 eff)**: plateaued but did NOT overfit. Headline oscillated 0.381–0.402 from epoch 10 to 17.5 with no downward trend — eval stopped responding but never declined. Training loss kept falling while eval held flat.
- **Server 6 (128 eff)**: plateaued AND then overfit. Headline peaked at 0.387 (epoch 10), then declined steadily to 0.344 (epoch 17.5) while training loss continued falling — textbook train/eval divergence.

The larger batch on server 10 acts as a regularizer: each gradient step sees more diverse negatives, which slows overfitting. Server 6's 128-negative batches give a noisier, weaker training signal that the model eventually memorizes around.

### What Plan-7 needs

1. **Dev loss logging** — log multi-positive InfoNCE on held-out dev queries at every eval; primary overfitting signal missing from Plan-6.
2. **Hard-negative mining** — current in-batch negatives are random; hard negatives from the gallery would push the model harder past the current ceiling.
3. **LR scheduling** — constant LR throughout; cosine decay may allow finer convergence near the plateau.
4. **Cross-dataset eval** — add CIRR or FashionIQ zero-shot eval to distinguish in-distribution plateau from out-of-distribution overfit.
