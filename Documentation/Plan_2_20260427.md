# Plan — Phase 1 baseline scaffolding (local first, VLM on server)

## Context

The mentor's 2026-04-27 update (`Documentation/meeting_memo_20260420.md`)
sets the immediate scope: build a **text-modification caption-generation
retrieval baseline** end-to-end, look at metrics, *then* decide whether to
add contrastive learning and/or audio. The committed dataset exploration
already confirmed FACap is usable today; this plan turns that into running
code.

**Confirmed today:**
- Repo strategy: separate repos long-term. Private GitHub remote at
  `git@github.com:ZhuoyuanJiang/fashion-retrieval-agent.git` is now
  set up; both existing commits pushed to `main`. Public flip can happen
  later when the work is portfolio-ready.
- Default baseline VLM: **speechQwen2VL in text-only mode** (per user's
  Stage 1 → Stage 2 narrative continuity). Backend is configurable so
  vanilla Qwen2-VL, an oracle captioner, and a mock are all swappable
  via CLI for debugging and ablation.
- Local hardware: ~8 GB VRAM laptop.

**The hardware reality.** Qwen2-VL-7B (which speechQwen2VL is built on)
needs ~15 GB at fp16 and is still tight at int8 once image tokens are in
the mix. Running the real model locally on 8 GB VRAM is not a comfortable
fit. So the split is:

- **Locally** — build everything that surrounds the VLM call (dataset,
  embedding DB, retrieval, metrics, prompt templates, eval harness),
  using a tiny text encoder + a *mocked* VLM caption generator. Verify the
  end-to-end pipeline runs without errors and produces sane numbers on a
  small slice.
- **On your server** — swap the mock for the real speechQwen2VL inference
  call and run the actual baseline on a meaningful slice of FACap.

This split is genuinely productive: ~80% of the baseline code is
non-VLM plumbing that has nothing to do with VRAM, and that plumbing is
exactly where bugs live. Doing it locally first means the server time is
spent on the one thing the server is needed for — VLM inference.

  helper and caches to disk on first access.
- `src/baseline/text_encoder.py` — thin wrapper around a Sentence-BERT
  (`all-MiniLM-L6-v2`, 23 MB, CPU-friendly). Same encoder used to embed
  both database captions and VLM-generated captions. **The encoder is
  swappable — see "Encoder choice" note below.**
- `src/baseline/build_caption_db.py` — encodes a chosen subset of FACap
  target captions, saves `{embeddings.npy, metadata.jsonl, config.json}`
  to `runs/<run_name>/caption_db/`. The `metadata.jsonl` has one record
  per row (target_id, optional source category, optional caption length);
  `config.json` records encoder name + version, subset description, and
  the FACap commit SHA used so a result can be reproduced exactly.
- `src/baseline/vlm_caption.py` — generates target caption from
  `(reference_image, modification_text)`. Pluggable backend interface
  with **four** implementations:
  - `MockVLMCaptioner` — locally returns the FACap modification text
    rephrased into a caption template (just enough to verify retrieval
    plumbing); deliberately simple so the pipeline is testable without
    a GPU.
  - `OracleCaptioner` — returns the **ground-truth `target_caption`**
    from the dataset itself. The point: if oracle retrieval doesn't get
    Recall@1 ≈ 100%, the bug is in the encoder / index / retrieval /
    eval code, not the VLM. Decouples "is the model good?" from "is the
    plumbing correct?".
  - `Qwen2VLCaptioner` — calls vanilla `Qwen2-VL-7B-Instruct` (no audio
    extension). Useful as a clean text-only baseline reference number.
  - `SpeechQwen2VLCaptioner` — calls the user's Stage-1 model in
    text-only mode (audio path ignored). The user-chosen default for
    the headline baseline number, since it preserves the
    Stage 1 → Stage 2 narrative. Stub the model-loading code locally
    (`raise RuntimeError("server-only")`); fully wire it up when we
    move to the server.
- `src/baseline/prepare_images.py` — pre-fetches all reference images
  for the chosen evaluation slice into a local cache **before**
  inference starts. Server runs must not depend on surprise network
  fetches mid-experiment (HF rate-limit risk, latency variance,
  failure-mid-run risk).
- `src/baseline/retrieve.py` — given a generated caption embedding and
  the caption DB, return top-K target IDs by cosine similarity.
- `src/baseline/eval.py` — Recall@1/5/10/50 + **median rank + mean rank**,
  plus saves qualitative success/failure pairs to
  `runs/<run_name>/qualitative/results.jsonl`. Each JSONL row carries
  `{query_id, true_target, top10_predicted, generated_caption,
  failure_category}` — the `failure_category` field starts blank and is
  filled in later by hand from a small rubric: `caption_wrong /
  embedding_mismatch / dataset_ambiguity / visual_nuance_lost / null`
  (null = success).
- `src/baseline/run_baseline.py` — top-level driver that ties them
  together. CLI: `--n-eval <int>`, `--vlm
  {mock,oracle,qwen2vl,speechqwen2vl}`, `--run-name`. Default
  `--vlm speechqwen2vl` for the headline baseline.

## Reuse over new code

- **Sentence-BERT via `sentence-transformers`** for text embedding. Don't
  hand-roll an encoder. (See "Encoder choice" note below for *why this
  one* and when we'd swap.)
- **Marqo/fashion200k HuggingFace mirror** for on-demand image fetching;
  we already proved this works in `fetch_facap_sample.py`.
- **FACap's pre-computed image captions** are the retrieval database
  source — no need to caption images ourselves locally.
- For the mock VLM, generate the "target caption" by feeding a templated
  prompt (`"a fashion item that is {modification_text}, derived from
  {brief description of reference}"`) — keeps the local smoke test honest
  enough to surface real bugs in the retrieval and metric code.

## Encoder choice — why Sentence-BERT `all-MiniLM-L6-v2` *for now*

Picking `all-MiniLM-L6-v2` is a deliberate *baseline* choice, not a
quality claim. The reasons:

1. **CPU-friendly.** Our 8 GB VRAM is reserved for the (eventual) VLM.
   The text encoder must run on CPU. `all-MiniLM-L6-v2` is 23 MB and
   does hundreds of sentences/sec on a laptop CPU.
2. **Purpose-trained for retrieval.** Sentence-BERT models were trained
   on millions of sentence pairs with a contrastive objective so that
   cosine similarity ≈ semantic similarity. That is exactly what
   caption-to-caption retrieval needs.
3. **No fine-tuning required.** Out-of-the-box performance is "good
   enough" on general English, which is what FACap target captions are.
4. **Small embedding (384-dim).** Tiny on-disk database, fast cosine
   similarity, easy to inspect by hand.
5. **Strong baseline default.** Most retrieval papers compare against
   SBERT-style baselines, so the number is interpretable in context.

**When to consider swapping** (not now, but soon — once we have baseline
metrics):

| Encoder | Pros | Cons | When to try |
|---|---|---|---|
| `all-MiniLM-L6-v2` (current) | tiny, fast, CPU | 384-dim, general-domain | always — anchor baseline |
| CLIP text encoder (e.g. `openai/clip-vit-base-patch32`) | trained jointly with images, so embeddings align with visually-derived captions | larger, GPU preferred | if SBERT struggles to align VLM-generated captions with FACap captions |
| BGE-large / E5-large (2026 SOTA general retrievers) | top of MTEB leaderboards | ~1.3 GB, slower | if Recall@K plateaus and the bottleneck is text-side |
| FashionCLIP / MARQO-FashionCLIP | fashion-domain trained | smaller community, less tested | only after generic encoders are exhausted |

**The contract.** `text_encoder.py` exposes a single `encode(list_of_str)
-> np.ndarray` method. Swapping the encoder = changing one class in one
file; the rest of the pipeline doesn't care. The `config.json` saved by
`build_caption_db.py` records which encoder produced any given index, so
results are never silently mixed across encoders.

### A note on "FACap evaluation slice"

FACap ships only `*_train_triplets.json` files — there is **no formal
val/test split** in the repo. For evaluation we therefore *carve* a
held-out slice out of the train data ourselves:

- **Eval queries:** the last `N` triplets in `dress_train_triplets.json`
  (deterministic, easy to reproduce). Default `N = 1000`.
- **Candidate gallery:** the union of (a) the target image_ids of those
  N eval triplets — guaranteeing coverage — and (b) the remaining
  `~58,082 - N` image_ids as distractors.
- **Why this is honest:** because FACap modifications are
  auto-generated, the train/eval distinction here is mainly about *not
  testing on triplets the gallery construction is biased by*; it's not
  a true generalization test. Treat the resulting numbers as a
  same-distribution sanity check, not a benchmark. For a real
  cross-distribution number we want FashionIQ + enhFashionIQ in a
  later plan.
