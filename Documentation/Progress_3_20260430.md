### Issues encountered during setup

**Issue 1 — flash-attn build isolation**

`pip install -r requirements.txt` failed on `flash-attn==2.6.3` with:
```
ModuleNotFoundError: No module named 'torch'
```
Root cause: pip's isolated build environment for `flash-attn` doesn't inherit the
current env's torch. `flash-attn`'s `setup.py` tries to `import torch` during the
`get_requires_for_build_wheel` hook.

Fix: install torch first, then install everything else, then install flash-attn with
`--no-build-isolation` so it uses the current env's torch.

**Issue 2 — flash-attn cross-device rename (Errno 18)**

Even with `--no-build-isolation`, flash-attn failed with:
```
[Errno 18] Invalid cross-device link: '...flash_attn-2.6.3+cu123...' -> '/ssd1/...'
```
Root cause: pip downloads the prebuilt wheel to `/tmp` (on the root filesystem) and
then tries to rename it to the pip cache dir (on `/ssd1`, a different device). Rename
across devices is not allowed by the kernel.

Fix: set both `TMPDIR` and `--cache-dir` to `/ssd1/zhuoyuan/` so the download and
cache are on the same device:
```bash
mkdir -p /ssd1/zhuoyuan/tmp
TMPDIR=/ssd1/zhuoyuan/tmp pip install flash-attn==2.6.3 \
    --no-build-isolation \
    --cache-dir /ssd1/zhuoyuan/pip_cache
```

in `~/.bashrc` (lines 135–140). Duplicate block (lines 138–140) removed. All three

## Design issue identified: caption DB depends on n_eval unnecessarily

During the full run setup, it was noted that `build_caption_db.py` ties the DB
composition to `n_eval` via the `build_signature` stale-check. This causes an
unnecessary rebuild between the smoke (n_eval=50) and full (n_eval=1000) runs even
though the 59,082 embeddings are effectively the same content.

**Root cause:** `build_db` constructs the DB as:
- guaranteed eval targets (last n_eval triplets' targets)
- + randomly sampled distractors to fill db_size

When n_eval changes, the distractor sample changes (different seed draw), triggering
a rebuild. When `db_size == total targets` (the production case), this randomness is
meaningless — all targets end up in the DB regardless.

**Fix shipped during the full run** (the run was unaffected — old code path):

After discussion, the final design supports **two coexisting modes** rather than
removing the subset capability outright. Default is the new full mode; the old
subset behavior is preserved as opt-in for future debug iterations.

**1. `src/baseline/build_caption_db.py`** — split into two builders:
   - `build_db_full(out_dir, category, split, encoder_name)` — encodes ALL
     captions, no sampling. Production path.
   - `build_db_subset(out_dir, category, split, encoder_name, eval_n, db_size,
     seed)` — guaranteed eval targets + sampled distractors. Debug path.
   - Two signatures: `build_signature_full(...)` (3 keys) and
     `build_signature_subset(...)` (6 keys, includes `mode: "subset"`).
   - Shared `_write_db()` helper avoids duplicating encode + serialize logic.

**2. `src/baseline/run_baseline.py`**:
   - New `caption_db_dir(out_root, encoder_name, db_size, eval_n, seed)`
     resolver. Path scheme:
     - Full: `runs/caption_db/<encoder_slug>/`
     - Subset: `runs/caption_db_subset/eval{n}_db{m}_seed{s}/<encoder_slug>/`
   - `ensure_caption_db(...)` dispatches to the right builder based on whether
     `db_size` is `None`.
   - `run(...)` accepts optional `db_size` and `seed` kwargs.
   - CLI: added `--db-size` (default `None` → full mode) and `--seed` (default
     `42`, used only when `db_size` is set).

**3. `scripts/run_baseline_v1.sh`**:
   - `DB_SIZE` env var restored as opt-in (empty → full mode, default).
   - `SEED` env var added (default `42`).
   - Header documents both modes.
   - Args assembled into a bash array; `--db-size` / `--seed` only appended when
     subset mode is requested.

**4. Helper:** `encoder_slug(name)` in `build_caption_db.py` —
   `org/model` → `org__model`.

**Usage examples (after this commit):**
```bash
# Full mode (production, default — DB built once at runs/caption_db/<enc>/)
bash scripts/run_baseline_v1.sh

# Full mode with smaller eval slice (DB still reused, no rebuild)
N_EVAL=50 RUN_NAME=quick_check bash scripts/run_baseline_v1.sh

# Subset mode (debug — small DB at runs/caption_db_subset/eval50_db1000_seed42/<enc>/)
DB_SIZE=1000 N_EVAL=50 RUN_NAME=tiny_debug bash scripts/run_baseline_v1.sh
```

**Net effect for future work:**
- Change `n_eval` / `run_name` / VLM in full mode → DB reused, no rebuild.
- Change encoder → first time builds a new subdir, reused thereafter.
- FACap commit SHA mismatch still triggers a rebuild (safety net).
- Need a small DB for fast plumbing iteration → set `DB_SIZE=...` and you get
  the old subset behavior in a separate directory tree.

## M4 — Encoder ablation (12 encoders, replay over saved captions)

### Motivation

The `baseline_v1_speechqwen2vl` run uses `all-MiniLM-L6-v2` (22 M params,
2022). The R@1=8.4% / R@10=24.0% headline could plausibly be **encoder-bound**
rather than VLM-bound: a 22 M sentence encoder may simply lack the capacity
to embed clothing descriptions discriminatively against a 59 k gallery.

This milestone tests that hypothesis: hold the VLM-generated captions fixed
(reuse `runs/baseline_v1_speechqwen2vl/qualitative/results.jsonl` — the same
1000 generated captions) and **swap only the text encoder + recompute
retrieval**. If a 8 B SOTA encoder still gives R@1 ≪ 1, the bottleneck is the
captions, not the encoder, and Plan_4 (contrastive end-to-end) is justified.

### Replay design (no VLM rerun)

The 25-min VLM step is the most expensive part of a full run. Re-running it
12× (≈5 h) is wasteful when only the encoder changes. New script
`src/baseline/replay_with_encoder.py` skips VLM entirely:

1. Read saved `generated_caption` strings from the source run's
   `qualitative/results.jsonl` (1000 queries).
2. Build (or reuse) a per-encoder caption DB at
   `runs/caption_db/<encoder_slug>/` — encoded once, reused across runs.
3. Re-encode the 1000 generated captions as queries with the new encoder,
   retrieve top-K, score.
4. Write metrics + qualitative dump to `runs/baseline_v1_speechqwen2vl_<slug>/`.

Per-encoder cost: encode 59 k captions (depends on size — small SBERT models
~3 min CPU, 4–8 B LLM-encoders ~10 min on a 3090) + 1 k queries (seconds) +
ranking (seconds). Wall time per encoder: 1–15 min.

### Engineering issues encountered

**1. CLIP-ViT-L-14 — 77-token hard limit.** CLIP's
`max_position_embeddings=77` triggers a hard error if any input tokenizes to
>77 tokens, and the sentence-transformers `max_seq_length` attr does *not*
suffice (the wrapper still hits the model with full input). Fix: hard-truncate
inputs to `77 × 3 = 231` chars before tokenization in `_pre_truncate()`. CLIP's
tokenizer then truncates the rest. This drops trailing detail past ~250 chars,
but CLIP couldn't see it anyway. Same approach applied to SigLIP (64-token
limit, 192-char budget).

**2. Marqo Fashion CLIP/SigLIP — sentence-transformers loader fails.** Both
models throw `meta tensor` errors going through SBERT's transformers loader.
Fixed by adding a `loader: "open_clip"` field in `EncoderConfig` and a
duck-typed `_OpenClipWrapper` exposing `.encode(texts)` so the rest of the
replay pipeline doesn't care which loader was used. Also installed
`open_clip_torch`.

**3. Qwen3-Embedding-8B / NV-Embed-v2 — 24 GB OOM at fp32 load.** 7–8 B params
× 4 bytes = 28–32 GB at fp32, doesn't fit on a single 3090. Loading at bf16
(`model_kwargs={"torch_dtype": torch.bfloat16}`) halves it to 14–16 GB. NV-Embed
additionally OOM'd during the fp32→bf16 `.to()` conversion (peak transient
memory above 24 GB). Fixed by adding `low_cpu_mem_usage=True` so accelerate
streams weights directly into bf16 via the meta-init path, never materializing
fp32 on GPU.

**4. Multi-GPU orchestration.** `scripts/run_encoder_swap.sh` assigns each
slug to a specific GPU index, launches them in parallel as background jobs,
and waits. Heavy 4–8 B encoders get dedicated GPUs; small encoders share.
Idempotent: skips any slug whose `runs/<run_name>/metrics.json` already
exists, so a partial failure can be resumed by just re-running.

### Interpretation

**1. Encoder choice matters massively — MiniLM was a weak anchor.**
R@1 jumps from 0.084 (MiniLM-L6) to 0.258 (Marqo FashionCLIP) on the
*identical* set of VLM-generated captions. That's a **3.07× lift** with
zero VLM changes. Median rank drops from 155 to 8 in the same swap. The
Plan_3 baseline headline of "R@1=8.4%, weak baseline" is therefore
*partially encoder-bound* — the captions themselves carry materially more
signal than MiniLM-L6 can extract.

**2. Domain-specific pretraining beats general-purpose SOTA at R@1.**
Marqo FashionCLIP (~150M params, fashion-domain CLIP) tops every
general-purpose encoder including the 8B Qwen3-Embedding. FashionSigLIP
(64-token limit) trails its CLIP sibling but still beats every non-fashion
non-fine-tuned model except BGE/E5. Strong signal that domain-pretraining
is a high-leverage axis for this task.

**3. Scaling within Qwen3-Embedding family is non-monotone at R@1 but
monotone at R@10/R@50.** 0.6B → 4B → 8B gives R@1 = 0.120 → 0.192 →
0.174 (slight drop at 8B), but R@10 = 0.298 → 0.475 → 0.522 (clean
monotone scaling) and R@50 = 0.454 → 0.656 → **0.704** (best in the
table — only metric where Qwen3-8B wins outright). 8B finds the right
answer in the top-50 most reliably; the 4B is sharper at top-1.
Marqo FashionCLIP still beats Qwen3-8B on R@1/R@5/R@10. Likely
instruction-prompt sensitivity — Qwen3 family expects task-specific
prompts that we did not tune.

**4. CLIP-general (`clip-vit-l-14`) is the worst performer.** Dramatically
worse than MiniLM despite 5× the params, because CLIP's 77-token cap
truncates ~250-char dress captions. The fashion-domain Marqo CLIP has the
same 77-token cap but is fine-tuned on fashion text, so the surviving
tokens are far more discriminative. The architectural lesson: **CLIP token
length, not CLIP architecture, is the killer.**

**5. mpnet-base underperforms MiniLM-L6.** Counter-intuitive — bigger
model, worse retrieval. Reason: `all-mpnet-base-v2` is a *sentence
similarity* SBERT, not retrieval-tuned. BGE/E5/Qwen3-Embedding are
explicitly trained on query↔passage contrastive objectives with prefix
tokens. Architectural lesson: **retrieval training recipe matters more
than backbone size** at this scale.

### Headline takeaway for Plan_4

The R@1=8.4% MiniLM baseline overstates how weak caption-generation is.
With Marqo FashionCLIP, R@1=25.8% / R@10=53.3% / R@50=68.5% on a 59k
gallery — still not production-ready, but a much more interesting starting
point. Plan_4 (contrastive end-to-end) should be evaluated against the
**Marqo FashionCLIP encoder** as the new "strong" reference, not against
MiniLM-L6.

Open questions surfaced:
- Why does Qwen3-Emb-8B beat Qwen3-Emb-4B on R@10/R@50 but lose on R@1?
  Likely instruction-prompt sensitivity. Tuning the query/passage prompt
  could unlock the 8B model's full capacity.
- Can a fashion-domain BGE/E5 (fine-tuned BERT retriever on fashion text)
  match Marqo FashionCLIP? Not in the zoo, but worth a quick LoRA-ft
  ablation later.
- Is BGE/E5's R@1 ceiling (~0.23) the actual generated-caption ceiling,
  or can a fashion-tuned text encoder push past 0.30?
