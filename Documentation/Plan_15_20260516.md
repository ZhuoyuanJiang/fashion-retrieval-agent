## 1. Context & scope

The retrieval model is a two-tower contrastive model. Plan-13's best run
(shared backbone, bs=24) reached **headline R@10 = 0.6540** with a **text**
modification on the query side. Plan 14's research question: does the same
model reach competitive R@K when the modification is **spoken audio** entering
through speechQwen2VL's Whisper encoder?

Decided constraints (Plan 14 §1, unchanged):

- **Reuse Plan-13's architecture and recipe verbatim** — shared backbone + two
  PEFT LoRA adapters, multi-positive InfoNCE with cross-GPU `gather`, the
  Plan-12 PEFT-aware gradient-checkpointing `context_fn`, per-epoch gallery
  refresh, 18 epochs, standard AdamW. The **only** change is the query-side
  modification input: text → audio.
- **Train fresh from scratch.** No warm-start from any text checkpoint, no
  frozen target tower. Both towers co-train. The 0.6540 text result is a
  **comparison reference only** — not an initialization, not an alignment
  target.
- No ASR baseline — a sufficiently accurate ASR→text pipeline would just
  reproduce the text tower, so 0.6540 already proxies the perfect-ASR case.

This plan covers **M3** (code) and **M4** (run + eval). The image+audio
forward path is already proven (M1, Progress_14 §11); the audio data already
exists (M2, Progress_14 §12).

## 2. The data the code will consume (recap from M2)

`/tmp3/zhuoyuan/plan14_audio/manifest.json` has two sub-dicts, each keyed by
**FACap dress-train triplet index** (string):

- `in_dist` — **55,186** records, one per *used* triplet (53,686 train + 500
  dev + 1,000 headline), voiced by the 100-speaker training pool.
- `ood` — **1,500** records (the 500 dev + 1,000 headline triplets only),
  voiced by the 10 held-out speakers, for the separate OOD-voice eval.

Each record: `{wav, speaker, split, gender, accent}`. `wav` is a 16 kHz mono
`.wav`. `bank.json` holds the speaker → pool/gender/accent table.

The manifest key **is** the triplet index — so any code holding a triplet
index can look up its audio directly. This is the one new coupling M3 adds.

## 3. M3 — code changes

Five files. Each subsection below leads with the **concrete goal** — what the
code should produce after the change, shown as *before → after* — followed by
the edits that get there. The text path stays intact throughout:
`--query-modality text` must still reproduce Plan-13 byte-for-byte (it is the
control).

### 3.1 `src/data/contrastive_dataset.py` — the dataset class

**Goal:** make `FacapContrastiveDataset.__getitem__` hand the training loop the
modification as **audio** (a wav looked up from the M2 manifest by triplet
index), alongside everything it already returns — without dropping `mod_text`,
so the text path still works.

What `__getitem__` returns **now** (text-only — the class has no idea the audio
or `manifest.json` exist):

```python
{
  "cand_image": <PIL candidate image>,
  "mod_text":   "The dress is more fitted and shorter, with a vibrant red ...",
  "target_id":  "89255983_0",
  # "tgt_image": <PIL target image>    # only when load_target=True
}
```

What `__getitem__` returns **after M3**:

```python
{
  "cand_image":    <PIL candidate image>,
  "mod_audio":     "/tmp3/zhuoyuan/plan14_audio/audio/0.wav",  # NEW — manifest[idx]
  "mod_text":      "The dress is more fitted and shorter, ...",  # kept (text path)
  "target_id":     "89255983_0",
  "tgt_image":     <PIL target image>,
  "triplet_index": 0,                                          # NEW — see below
}
```

Edits:
- `__init__` gains `audio_manifest: Path | None` and `query_modality:
  {"text","audio"}`; when `audio`, load `manifest.json`.
- **Attach `triplet_index` to every item** — `__getitem__` items, `dev_items`,
  and `headline_items`. Today none of them carry their base index, and the
  eval needs it to look up audio. headline indices = `range(N-1000, N)`; dev
  indices = `clean_indices[perm[:500]]`; train = `self._train_indices[idx]`.
  Pure additive — the text path ignores the key.
- `__getitem__` adds `mod_audio` = `manifest["in_dist"][str(triplet_index)]
  ["wav"]`. (v1 returns the **path** — the M1 smoke test proved
  `process_vision_info` consumes paths; switch to a decoded 16 kHz mono array
  only if dataloader throughput needs it — a tuning detail, not a design
  change.)
- `contrastive_collate` collates `mod_audios` (mirrors `mod_texts`).
- Add `self.train_mod_audios` — per-training-item wav paths, mirroring
  `train_mod_texts` — for the shuffled sensitivity probe (§3.4).

### 3.2 `src/training/two_tower_model.py` — the query tower

**Goal:** make the query tower's per-sample chat message carry the spoken
modification instead of the typed one, so the modification reaches the model
as audio through the Whisper encoder. The **target tower is untouched**.

Query message built **now** (text):

```python
[{"role": "user", "content": [
    {"type": "image", "image": <candidate image>},
    {"type": "text",  "text": "Given this product image, find the item that
                               looks like the image but with the following
                               modification: <mod text>"},
]}]
```

Query message built **after M3** (audio):

```python
[{"role": "user", "content": [
    {"type": "image", "image": <candidate image>},
    {"type": "audio", "audio": <wav path>},      # NEW — carries the modification
    {"type": "text",  "text": INSTR},            # fixed, modification-free (§3.5)
]}]
```

Edits:
- `TwoTowerSharedBackbone` gains a `query_modality` attribute (set at
  construction).
- `_forward_one_side`, query side + `audio`: build the message above, run it
  through `process_vision_info(messages)` → `processor(text=, images=,
  videos=, audios=, return_tensors="pt", padding=True)` — the exact path M1
  verified. Pooling (last non-pad token), projection head, L2-norm — unchanged.
- `forward` / `encode_query` accept `mod_audios` and thread the modality;
  the `mod_texts` branch stays as-is.
- The audio encoder (Whisper `audio_encoder`, 637M) + `audio_projector` (17M)
  stay **frozen** — not LoRA targets, not projection heads — so the trainable
  surface is **unchanged** from Plan-13 (query LoRA + target LoRA + 2 heads +
  logit_scale, ≈32M). Unfreezing the projector is a noted V2 lever.

### 3.3 `src/training/train_plan10.py` — the training entry point

**Goal:** let the run be launched in audio mode and feed the audio batch into
the *same* training step the text run uses — loss, optimizer, gather, gallery
refresh, checkpointing all unchanged.

Training step **now** vs **after M3** (audio mode):

```python
model(cand_images, mod_texts,  tgt_images)   # now    — batch["mod_texts"]
model(cand_images, mod_audios, tgt_images)   # after  — batch["mod_audios"]
```

Edits:
- Add `--query-modality {text,audio}` (default `text`) and `--audio-manifest`
  (default the M2 `manifest.json`).
- Thread both into `FacapContrastiveDataset` + `TwoTowerSharedBackbone`.
- The loop reads `batch["mod_audios"]` when audio. W&B run name
  `plan14/audio_query_<descriptor>`.

### 3.4 `src/training/online_eval.py` — the eval

**Goal:** make every eval encode the query from **audio** (looked up per item
by `triplet_index`), and redefine the 3-way sensitivity probe in audio terms.
The gallery / target side is untouched (targets are images).

Query encoding **now** vs **after M3**:

```python
model.encode_query(images, texts)    # now
model.encode_query(images, audios)   # after — audios looked up by triplet_index
```

3-way sensitivity probe — what each arm feeds the query:

| probe arm | text run (Plan-13) | audio run (after M3) |
|---|---|---|
| `normal`   | the item's real modification text | the item's real audio |
| `stripped` | empty string `""` (image-only) | **image-only** query, no audio |
| `shuffled` | another item's text | another item's **audio** (`train_mod_audios`) |

The gap `R@10(normal) − R@10(stripped)` clearly positive = the model uses the
audio, not the image alone.

Edits:
- `run_retrieval_eval`, `run_dev_probe`, `run_dev_loss_two_tower` accept the
  query modality + an audio source; query encoding uses `mod_audios` instead
  of `modification_text`.
- `encode_gallery_with_tower` / `encode_target` — **unchanged**.

### 3.5 The fixed query instruction (`INSTR`)

**Goal:** one constant string that frames the retrieval task and carries **no**
modification content — the modification is entirely in the audio. Locked value
(same as the M1 smoke test):

> "Given this product image, find the item that looks like the image but with
> the modification described in the spoken audio."

System prompt left at default; system-prompt placement of the instruction is a
valid lever, unused in v1.

## 4. M4 — training & evaluation

### 4.1 Architecture & recipe (= Plan-13)

`--arch shared`: one speechQwen2VL backbone (frozen base + Stage-2 LoRA
merged) + PEFT LoRA adapters "query"/"target" + two projection heads.
Multi-positive symmetric InfoNCE, cross-GPU `gather`, PEFT-aware
gradient-checkpointing `context_fn`, end-of-epoch gallery refresh, 18 epochs,
AdamW (LoRA LR 2e-5, head LR 1e-4, weight decay 0.01), eval every 0.5 epoch,
`--first-eval-step 5`. **Trained fresh, both towers, no warm-start.**

### 4.2 Batch-size / memory scan (the one real unknown)

The audio query sequence is materially longer than the text one: a ~8 s clip
is ~400 audio tokens (~50 tokens/s) vs ~40 tokens for a typed modification,
and the frozen Whisper encoder adds ~1.3 GB of resident weights plus its own
activation memory. **Plan-13's bs=24 will almost certainly not hold.**

Run the same memory scan Plan-13 §11.2 did: scan bs ∈ {8, 12, 16, 24} with a
1-epoch stability gate (model load + step-0 backward + end-of-epoch gallery
refresh + post-refresh eval, all without OOM), pick the largest stable bs.
Target GPU class: an 8-GPU host of the same 48 GB class Plan-13 used. Record
peak per-rank memory per bs. Effective negatives = bs × 8.

### 4.3 Single-epoch sanity gate

Before the full run: one epoch, then assert end-of-epoch `dev R@10 > 0.10`
(reuses the existing `--first-eval-step 5` + 0.5-epoch eval path — no parallel
harness). A pass means the audio→retrieval signal is flowing; a fail stops the
run for a wiring audit.

### 4.4 Full run

`run_plan10.sh --arch shared --query-modality audio --batch-size <scan result>
--max-epochs 18 --first-eval-step 5`. W&B `plan14/audio_query_<descriptor>`.
Standard robust launch (tmux + unbuffered + tee). Per-epoch checkpoints.

### 4.5 Evaluation — in-distribution voice (primary)

Every 0.5 epoch (unchanged cadence): dev (500) + headline (1000)
R@{1,5,10,50}, **dev InfoNCE loss** (the primary overfitting signal — logged
every eval), and the 3-way sensitivity probe. Primary headline R@10 uses
**in-distribution-voice** audio (training-pool speakers) — the eval triplets
are already held-out *content*; the voice stays in-distribution so a
speaker-OOD effect does not confound the headline number.

### 4.6 Evaluation — OOD voice (separate number)

At the **peak** and **final** checkpoints only (not every 0.5 epoch — it is a
robustness check, not a training signal): re-run the dev + headline eval using
the `ood` manifest audio (the 10 held-out speakers). Report as its own
R@{1,5,10,50}. The gap (in-dist − OOD) quantifies sensitivity to unheard
voices.

### 4.7 Decision matrix (headline R@10, in-distribution voice, peak ckpt)

| R@10 | Read |
|---|---|
| ≥ 0.60 | Audio-native two-tower is competitive with the 0.6540 text tower. Proceed. |
| 0.45–0.60 | Works with a real gap — diagnose TTS quality / batch size / epochs; one re-run. |
| < 0.45 | Structural problem — audit the image+audio wiring, audio intelligibility, training. |

### 4.8 Write-up

`Progress_15` (or a Progress_14 §13 follow-up — match the repo's convention at
that point): R@K trajectory, peak vs final, dev-loss curve, sensitivity gap,
in-dist vs OOD-voice numbers, and the comparison vs 0.6540.

## 6. Risks

1. **Batch size / memory** — the audio sequence + Whisper encoder shrink the
   feasible bs vs Plan-13's 24; fewer effective negatives can cost R@K
   (Progress_12 showed batch size drove ~72% of one architecture gap).
   Mitigation: the §4.2 scan picks the largest stable bs; the bs-vs-R@K
   relationship is a known, documented quantity.
2. **Synthetic → real speech gap** — mitigated by the 100-speaker VCTK
   diversity + the held-out OOD-voice eval; real-recorded-speech eval is a
   future plan.
3. **Throughput** — audio dataloading + the Whisper encoder forward slow each
   step; if disk I/O dominates, move audio decode into the dataloader workers
   (§3.1).
4. **Eval-item indexing** — the eval must look up audio by `triplet_index`;
   the §3.1 index-attachment is load-bearing and is covered by the
   text-control check (§5).
