# Plan 9 — Detailed-prompt VLM rerun + oracle upper bound + encoder re-ablation

**Date:** 2026-05-04
**Follows:** Progress_4_20260501.md (caption analysis)
**Phase:** A (Phase A is finally finishing; this is the last caption-side
ablation before we close the book on Phase A)

---

## 1. Context

Plan 4 (`notebooks/caption_analysis.py`) characterized the speechQwen2VL
captioner against the FACap ground-truth long captions. Two findings
drive this plan:

- **Length gap is huge.** Median generated caption = **92 chars**, median
  FACap target caption = **554 chars** — **~6× compression** by the VLM.
  The VLM's current `"concise"` prompt is the obvious culprit: it asks
  for a *concise* summary and the model obliges.
- **Caption agreement vs rank correlates but is far from sufficient.**
  Spearman ρ = −0.306 between rank and `token_overlap`. Real signal
  but only ~30% effect — suggests both VLM-side and encoder-side have
  headroom, not one or the other alone.

This plan attacks the **caption quality** axis directly: change the
prompt to elicit FACap-length detailed captions and re-measure across
the same encoder zoo as Plan 3 M4. Also adds an **oracle upper bound**
on the full 59k production DB — a measurement we have not done before
(the Plan 2 M3 oracle was on a tiny 1000-row plumbing DB).

## 2. The two experiments

### Experiment A — Oracle on the full 59k production DB ("encoder ceiling")

For each of the 1000 eval queries, use the **target's own FACap
ground-truth caption** as the retrieval query (instead of the
VLM-generated caption), against the full 59,082-row dress gallery.
Replay against all 11 encoders.

**This is not the Plan 2 M3 plumbing test.** That test used a tiny
1000-row hand-built DB (50 eval-targets force-included + 950 random
distractors), in which any cosine-1.0 self-match trivially hits R@1 = 1.0.
The full 59k production gallery contains *near-duplicate dresses* —
"same silhouette, different fabric", "same color, different pattern" —
whose long captions are textually similar. Even with the target's exact
caption as the query, the encoder may rank a near-duplicate above the
true target.

What this oracle measures is the **encoder's discrimination ceiling on
the real gallery** — the best R@K achievable assuming the VLM produces
a perfect FACap-style caption. It bounds Experiment B from above:

| Oracle R@10 (Marqo FashionCLIP) | Interpretation | Implication for Experiment B |
|---|---|---|
| > 0.85 | Encoder discriminates near-duplicates fine; VLM is the real bottleneck | Experiment B has substantial room — push prompting hard |
| 0.55 – 0.75 | Encoder is near its ceiling on this gallery | Experiment B will plateau early; Plan 5 (contrastive) is the right next move |
| < 0.50 | Encoder cannot resolve gallery ambiguity even with perfect text | Caption-side work is futile; pivot to image-side or contrastive |

### Experiment B — Detailed prompt + full encoder re-ablation

Replace the current `"concise"` prompt with a `"detailed"` variant
(see §4 for the actual prompt — minimal one-word change), rerun the
VLM on the same 1000 queries, then replay across **all 11 encoders**
from Plan 3 M4.

End deliverable: a side-by-side encoder ablation table comparing
**concise prompt** (current) vs **detailed prompt** (new), with a
`Δ R@10 = new − old` column to surface which encoders benefited and
which regressed (see §3 — token-cap interaction).

## 3. Why CLIP/SigLIP token caps make this a real ablation, not a
       monotone improvement

This is the load-bearing technical observation. **Encoder context windows
are hardware-constrained**, not soft suggestions:

| Encoder family | `max_position_embeddings` | ~chars at 3-4 chars/token |
|---|---:|---:|
| CLIP-ViT-L-14 | 77 | ~250 |
| Marqo FashionCLIP | 77 | ~250 |
| Marqo FashionSigLIP | **64** | ~200 |
| MiniLM-L6 | 256 | ~900 |
| BGE-large / E5-large / mpnet-base | 512 | ~1800 |
| GTE-ModernBERT-base | 8192 | enormous |
| Qwen3-Embedding (0.6 / 4 / 8B) | 8192+ | enormous |

**Current regime (~92-char captions):** every encoder sees 100% of the
caption. The Plan 3 M4 ablation is a fair comparison.

**Detailed-prompt regime (if outputs grow to ~300-500 chars, in line
with FACap ground-truth length):** BERT-/LLM-based encoders see all
of it; **CLIP / SigLIP truncate ~25–40% of the caption**. Marqo
FashionCLIP currently leads R@1 = 0.258 partly because the current
short-caption regime suits its narrow window. A detailed-prompt rerun
may flip rankings — bge-large / qwen3-emb-8b could overtake Marqo
FashionCLIP because they finally see content the CLIP family can't.

The actual length the VLM produces under the new prompt is itself an
empirical question — it could plateau at 200 chars (in which case
CLIP/SigLIP still see everything and the interaction doesn't kick in)
or hit 500+ chars (token-cap interaction matters a lot). Section 4
addresses this with a tiered fallback.

**This is exactly why the new ablation table needs a `Δ` column.** A
single absolute number per encoder hides whether the change helped or
hurt; the per-encoder delta is the analysis-relevant quantity.

## 4. Prompt design — minimal intervention with a soft inspiration hint

The current `_Qwen2VLLikeCaptioner.PROMPT_TEMPLATE` literally asks for
a *concise* caption — and the VLM is obeying it (median 92 chars). The
new prompt flips that to *detailed* and adds a deliberately soft
"for example" hint that the VLM can use as inspiration, **not** as a
checklist:

```python
PROMPT_TEMPLATE_DETAILED = (
    "Given the reference fashion image and the modification instruction, "
    "write a detailed caption describing the target fashion item after "
    "applying the modification. For example, you could consider aspects "
    "like silhouette, neckline, sleeves, fabric, color, pattern, or "
    "distinctive details."
)
```

**Two changes from v1:** `concise` → `detailed`, plus an `"For example,
you could consider..."` clause listing fashion attributes.

### Phrasing choices, on purpose

- **"For example, you could consider"** is the *softest* level of
  prescription that still provides a hint. "For example" frames the
  list as illustrative; "could consider" is weaker than `consider` /
  `mention` / `describe` / `cover`. Together they signal *brainstorming
  hint*, not *checklist*.
- **No `"if visible"`, no `"mention whichever applies"`, no `"skip any
  that don't"`.** All of these phrasings — even when meant softly —
  carry an implicit "if it's there, you should write it" obligation.
  Pure example framing avoids the trap.
- **No bulleted format.** A markdown bullet list reads as a
  requirement spec to most LLMs/VLMs; inline prose with "or" reads as
  examples. Matters more than it sounds.
- **No example vocabulary inside parentheses** (e.g. `silk / linen /
  lace / knit`). Earlier draft had these; risk is *anchoring bias* —
  the VLM may pick from the listed options even when the actual
  attribute (sequin, satin, etc.) is out-of-list. Naming axes is
  fine; naming sub-values is over-prescriptive.

### What we're trying to avoid (and why)

Two failure modes a more prescriptive prompt would invite:

1. **Hallucination from checklist coverage.** Forcing the VLM to cover
   all named axes when the reference image is e.g. a plain solid dress
   invites it to invent "subtle floral accents" or "delicate button
   details" to fill slots. That degrades alignment with the actual
   target image — strictly worse than a short caption.
2. **Loss of generalizability.** The named axes are
   dress-leaning ("neckline" / "sleeves" don't apply to pants;
   "silhouette" means something different for outfits). Soft-hint
   framing limits the cost — the model can ignore inapplicable axes
   without violating the prompt — but if we ever add cross-category
   eval, the prompt may need a category-aware variant. For now,
   Phase A is dress-only and this trade-off is acceptable.

### Fallback ladder (if the minimal prompt isn't enough)

Run the minimal version first, look at the length distribution, then
escalate only if needed:

| Tier | Prompt | When to use |
|---|---|---|
| **1 (default)** | "detailed" + soft "for example" hint — the version above | Always start here |
| 2 | + "Be specific and concrete about what is clearly visible in the reference." | If Tier 1 outputs still median ≤ 200 chars |
| 3 | + explicit bulleted axes (still no example vocabulary), dress-only | If Tier 2 still under-describes; accept the loss of generalizability |

`max_new_tokens` is already 256 — generous headroom (~750-1000 chars)
for any tier. No change needed.

## 5. Implementation — minimal changes, reuse existing infra

### 5.1 Code changes

- **`src/baseline/vlm_caption.py`** — add `PROMPT_TEMPLATE_DETAILED`
  alongside the existing concise template. Plumb a constructor flag
  `prompt_variant: Literal["concise", "detailed"] = "concise"` through
  `_Qwen2VLLikeCaptioner` so we can pick at runtime without forking a
  subclass. Default stays `"concise"` for back-compat.
- **`src/baseline/run_baseline.py`** — add a CLI flag `--prompt-variant
  {concise,detailed}` (default `concise`) and pass it to the captioner
  constructor. Stamp the variant into `metrics.json["context"]` so
  results are self-describing.
- **`src/baseline/replay_with_encoder.py`** — no change needed.
  Replays the saved `results.jsonl` regardless of which prompt produced
  it.

### 5.2 Oracle helper

We *could* extend `OracleCaptioner` to be invokable through
`run_baseline.py --vlm oracle`, but `OracleCaptioner` predates the
59k full-DB design and may have stale wiring. Simpler:

- New script `src/baseline/run_oracle.py` (≤50 lines): for each of the
  1000 eval triplets, set `generated_caption = target_caption` and dump
  a `results.jsonl` to `runs/oracle_v1_speechqwen2vl/`. No VLM, no GPU,
  ~5 seconds. Then `replay_with_encoder.py` runs against it like any
  other source run.

Why "speechqwen2vl" in the oracle path even though no VLM ran: keeps
the file-naming pattern consistent with the rest (`baseline_v1_<vlm>_<encoder>`)
so the aggregator scripts don't need special-casing.

### 5.3 Run names

| Experiment | VLM source run | Per-encoder runs |
|---|---|---|
| Oracle (Exp A) | `runs/oracle_v1_speechqwen2vl/` | `runs/oracle_v1_speechqwen2vl_<slug>/` |
| Detailed prompt (Exp B) | `runs/baseline_v2_speechqwen2vl/` | `runs/baseline_v2_speechqwen2vl_<slug>/` |

`baseline_v2` (not `baseline_v1_detailed`) keeps the naming aligned
with how prompt changes are tracked elsewhere — it's a new baseline
version, not a sub-variant of v1.

### 5.4 Orchestration

- **Exp A (oracle):** run `run_oracle.py` → 11× `replay_with_encoder.py`
  in parallel via the existing `scripts/run_encoder_swap.sh` pattern
  (point it at the new source run). ~5 min source + ~30-60 min total
  for all encoders.
- **Exp B (detailed prompt):** `bash scripts/run_baseline_v1.sh
  PROMPT_VARIANT=detailed RUN_NAME=baseline_v2_speechqwen2vl` (add the
  env var pass-through to the script). ~25 min VLM. Then 11×
  `replay_with_encoder.py` in parallel. ~30-60 min total.

### 5.5 Aggregation

`src/baseline/aggregate_encoder_results.py` already produces

## 6. Verification

1. **Prompt sanity check**: run the detailed prompt on 5 hand-picked
   eval queries first (`N_EVAL=5 bash scripts/run_baseline_v1.sh
   PROMPT_VARIANT=detailed RUN_NAME=detailed_smoke`). Eyeball the
   captions:
   - length materially longer than the v1 baseline (≥ 200 chars,
     ideally 300+ — but no hard target; the prompt no longer specifies
     a length)
   - actually applies the modification, not just describing the
     candidate
   - no obvious hallucinated attributes (be alert: if the dress is
     plain solid and the caption mentions floral patterns, the prompt
     is over-eliciting and we need to back off)
   - if length still ≤ 200 chars across all 5 samples, escalate to
     prompt Tier 2 (see §4) before committing to a full run
2. **Full Exp B run**: `runs/baseline_v2_speechqwen2vl/metrics.json`
   exists with N=1000.
3. **Length distribution check**: rerun
   `notebooks/caption_analysis.py` with `RUN_DIR =
   runs/baseline_v2_speechqwen2vl`. Section 4 should now show median
   gen length 300-500 chars (vs 92 before). Section 7's Spearman ρ
   tells us whether the longer captions correlate with better rank.
4. **Encoder ablation completeness**: 11 of 11 encoders have
   `runs/baseline_v2_speechqwen2vl_<slug>/metrics.json`. Same for
   `runs/oracle_v1_speechqwen2vl_<slug>/metrics.json`.
5. **Δ table generated**: `Documentation/encoder_swap_table.md`
   updated with the detailed/concise/oracle comparison.

## 7. Decision points after results

(Not for execution; for after we have the Δ table.)

- **If detailed > concise across all encoders:** prompt change is a
  pure win, fold into baseline.
- **If detailed > concise for BERT/LLM but ≤ for CLIP family:**
  confirms the token-cap interaction. The "best" encoder choice now
  depends on prompt — record both as Phase A's final story.
- **If detailed ≈ concise everywhere:** speechQwen2VL is prompt-saturated
  — bigger leverage is swapping the VLM (Plan 4.5 was vanilla Qwen2-VL
  / GPT-4V; we never ran it). Promote that to the next plan.
- **If detailed < concise widely:** suggests the VLM hallucinates more
  on long-form prompts than it adds detail. Audit a sample of detailed
  captions and decide whether to revert or iterate the prompt.
- **If oracle R@10 is below detailed R@10 + 0.10:** encoder is the
  bottleneck regardless of prompt. Phase A is done; pivot to encoder
  fine-tuning or contrastive end-to-end (Plan 5+ already in progress).
