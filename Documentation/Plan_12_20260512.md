## 1. Context

Plan-10 V1 produced two architectural variants:

| Variant | per-GPU batch | effective negatives (8 GPU gather) | peak head R@10 |
|---|---|---|---|
| Two-tower **separate backbones** (Plan-10 V1 Option B) | 8 | 64 | **0.6370** (ckpt_epoch11) |
| Two-tower **shared backbone** (Plan-10 V1 Option A, in progress at ep 13) | 4 | 32 | 0.5830 (ep 7.5 peak, plateauing) |

The shared variant trained with `--batch-size 4` because Option A had to
**disable HuggingFace's gradient checkpointing** as a safety measure against a
PEFT footgun: HF's gradient-checkpointing re-runs each transformer block's
forward during the backward pass to reconstruct activations, and that
recompute reads `model.active_adapter` **at recompute time**, not at the
original forward time. With two named adapters (`"query"`, `"target"`)
toggled per forward, the recompute of the query branch silently uses
`lora_target` weights, corrupting gradients into garbage. See Progress_11
§Appendix B / C for the full deliberation.

Disabling gradient checkpointing avoids the bug but doubles activation memory,
forcing `bs=4` on A6000-class GPUs. The result is a confounded comparison:

> Did the shared-backbone variant under-perform because of the **architecture**
> (two adapters competing for one frozen substrate), or because of the
> **batch size** (32 vs 64 effective negatives)?

Without isolating one variable, we cannot say. This plan removes the
batch-size confound.

## 2. The single research question for this plan

> **With PEFT-aware gradient checkpointing enabled (no silent gradient
> corruption) and `bs=8` matching Option B, does the shared-backbone variant
> reach ≥0.62 head R@10 on the 1000-query FACap dress slice — i.e., close
> to Option B's 0.6370 peak — or does it still plateau near 0.58?**

Outcome interpretation:

| Re-run peak head R@10 | Interpretation |
|---|---|
| ≥ 0.62 | Batch size was the dominant factor. Shared backbone is competitive with separate when given the same effective negatives. Architecture choice is then driven by VRAM / audio-swap considerations (Option A wins). |
| 0.58 - 0.62 | Partial effect from both batch size and architecture. Worth a third datapoint at bs=12 or bs=16 to confirm the trend. |
| < 0.58 | Architecture is the dominant factor. Two adapters sharing one substrate has a real ceiling below independent backbones. Plan 13 (V2 ablations) should bias toward the separate-backbone variant for the audio extension. |

## 4. The fix — PEFT-aware checkpointing via `context_fn`

`torch.utils.checkpoint.checkpoint(use_reentrant=False, ...)` accepts a
`context_fn` argument: a callable returning a tuple of two context managers,
applied to the forward and recompute respectively. HF Transformers'
`gradient_checkpointing_enable(gradient_checkpointing_kwargs=...)` forwards
arbitrary kwargs to the underlying `torch.utils.checkpoint.checkpoint` call,
so we can plumb a custom `context_fn` through without monkey-patching.

The fix captures `model.active_adapter` **at forward time** (via closure) and
restores it inside a custom context manager that wraps the recompute:

```python
import contextlib

def make_adapter_restoring_context_fn(peft_model):
    """
    Returns a context_fn for torch.utils.checkpoint.checkpoint that captures
    the active PEFT adapter at forward time and restores it during the
    backward-pass recompute.
    """
    def context_fn():
        # Called at forward time — capture the adapter that's active *now*.
        adapter_at_forward = peft_model.active_adapter

        # Forward context: no-op (the adapter is already set correctly by
        # the caller in TwoTowerSharedBackbone.forward).
        fwd_ctx = contextlib.nullcontext()

        # Recompute context: re-set the adapter to what it was at forward,
        # then restore the previous adapter on exit so we don't leak state
        # across recomputes of sibling blocks.
        @contextlib.contextmanager
        def recompute_ctx_mgr():
            prev = peft_model.active_adapter
            peft_model.set_adapter(adapter_at_forward)
            try:
                yield
            finally:
                peft_model.set_adapter(prev)

        return fwd_ctx, recompute_ctx_mgr()

    return context_fn
```

In `TwoTowerSharedBackbone.__init__`, change the existing line that disables
checkpointing into an `enable` with this `context_fn`:

```python
# OLD (Plan-10 V1 Option A):
# vlm.gradient_checkpointing_enable(...) intentionally NOT called.

# NEW (Plan-12):
vlm.gradient_checkpointing_enable(
    gradient_checkpointing_kwargs={
        "use_reentrant": False,
        "context_fn": make_adapter_restoring_context_fn(vlm),
    }
)
```

Everything else in `TwoTowerSharedBackbone` stays the same: the
`requires_grad=True` reset after both forwards (mitigation for footgun #1)
remains required.

### Why this is safe

- At forward time of the query branch, `peft_model.active_adapter == "query"`.
  `context_fn()` is called by `torch.utils.checkpoint.checkpoint`, captures
  `"query"` into the closure, and returns `(nullcontext, recompute_ctx_mgr)`.
- The checkpoint wrapper runs the forward inside `nullcontext` (we don't
  need to do anything — the adapter is already correct), then **discards the
  intermediate activations** and stores only the inputs + the closure.
- Later, at backward time, the user has already done the target-branch
  forward, so `model.active_adapter == "target"`.
- The checkpoint wrapper enters the **recompute** context, which calls
  `peft_model.set_adapter("query")` — restoring the adapter to what it was
  at the original forward — runs the forward again to reconstruct
  activations, then exits the context and resets to `"target"`.
- Gradients flow through the correct adapter weights. No silent corruption.

### Edge cases to be aware of

1. **Nested checkpoints**: Qwen2VL has many transformer blocks, each
   independently checkpointed. They all share the same `peft_model`
   reference, so each one independently captures and restores. The
   `try/finally` in `recompute_ctx_mgr` keeps state local.
2. **DDP synchronization**: gradient checkpointing + DDP + `use_reentrant=False`
   is supported (we already verified this works for the separate-backbone
   variant in Plan-5/6). No changes needed.
3. **PEFT version dependency**: PEFT's `active_adapter` property and
   `set_adapter` API have been stable since PEFT 0.7. Verify version in
   `pip show peft` is ≥ 0.7 before launching (it currently is at 0.13).
4. **HF Transformers version dependency**: the
   `gradient_checkpointing_kwargs={"context_fn": ...}` plumbing requires
   Transformers ≥ 4.41 (the version where arbitrary kwargs were forwarded
   to `torch.utils.checkpoint.checkpoint`). Verify before launching.

### VRAM expectation

Plan-10 Option A's smoke at bs=8 OOM'd because activations doubled when
checkpointing was off (47.5/47.5 GB resident per GPU). With checkpointing
back on, the expected resident VRAM is ~25-30 GB at bs=8, leaving headroom
to scale to bs=12 or bs=16 if we want to push the negatives further. For
this plan we stay at bs=8 to keep the apples-to-apples comparison with
Option B clean.

### Smoke test before full run

Before launching the 18-epoch run, do a 5-step smoke at `bs=8` on a single
GPU and verify:

1. Loss is finite at step 0 (~`log(8*8)` = ~4.16 for diagonal multi-positive
   on 64 unique-target items, since we gather 8 GPUs in production but the
   smoke is single-GPU so the matrix is 8×8).
2. Loss decreases over 5 steps.
3. Insert a temporary assertion or counter in
   `make_adapter_restoring_context_fn` that records (adapter_at_forward,
   adapter_at_recompute_entry). Confirm:
   - During forward, recompute_ctx hasn't been entered yet (counter = 0).
   - During backward, recompute_ctx is entered with `active_adapter ==
     <whatever was set by the most recent set_adapter call>` and exits with
     `active_adapter` restored to `adapter_at_forward`.
   - At least one recompute is entered with adapter ≠ adapter_at_forward,
     which is the case the fix is preventing corruption in.
4. Compare a 5-step gradient norm trajectory at bs=4 (no checkpointing) vs
   bs=4 (with checkpointing + context_fn). They should match within
   numerical noise — same forward, same gradients, just different memory
   strategy.

If any check fails, do not launch the full run. Iterate on the fix.

## Appendix A — Why the bs=4 vs bs=8 comparison is the load-bearing one

In multi-positive InfoNCE with `gather=True`, the effective number of
negatives per query equals (per-GPU batch × number of GPUs - 1). With 8
GPUs, Option A had 31 negatives and Option B had 63. Doubling the effective
negatives ≈ doubling the discriminative signal per gradient step. The
expected lift from the negative-count change alone, by Plan-6's logged
trajectory (where we have data at bs=16, 32, 64, 128, 256), is roughly
+0.04 to +0.07 head R@10 going from 32 → 64.

Option A's peak (0.583) vs Option B's peak (0.637) is a gap of 0.054 —
suspiciously close to the expected size of the batch-size effect alone.
That's the entire motivation for this plan: the gap might be the
architectural cost, the negatives cost, or both. We can't tell from
Plan-10's data.

If Plan-12's shared+bs=8 lands at ~0.62-0.63, the architectural cost is
nearly zero — the cheap-VRAM shared variant is just as good when given the
same training signal. That's the result we'd most like to see, because it
opens the V4 audio-extension path with the cleanest swap (one shared
backbone, swap query adapter only, target adapter reused).
