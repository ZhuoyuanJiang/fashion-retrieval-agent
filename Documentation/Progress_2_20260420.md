  `runs/<run_name>/caption_db/` if it doesn't exist, instead of
  requiring a separate `build_caption_db` step before each run. This

  description of reference}"`). The longer template would have needed
  a vision pass to describe the reference, which defeats the point of
  a CPU-only mock. Result: mock numbers are slightly higher than a
  true random baseline (see above) but this didn't affect the
  exit-criterion check.
- `qualitative/results.jsonl` rows include extra fields beyond
  Plan_2's spec: `query_idx`, `modification_text`, `top10_scores`,
  `rank`. They cost nothing to write and make hand-classification of
  failures easier later.
- **`max_new_tokens` bumped 128 → 256** in `_Qwen2VLLikeCaptioner.GENERATION_KWARGS`
  (2026-04-30, post-M3, after the prompt-design analysis below).
  Rationale: FACap's longest dress caption is ~170 tokens
  (130 words × ~1.3 tokens/word); a 128-token cap would silently
  truncate any prompt that successfully elicits a FACap-length
  caption. With greedy decoding (`num_beams=1`, `do_sample=False`),
  raising the cap is free on outputs that finish early under
  "concise" — it only matters when the model wants to keep going.
  Pure safety margin; doesn't change behavior on the current
  zero-shot prompt.

### Prompt design considerations for Plan_3

The current prompt is zero-shot — class constants at
`vlm_caption.py:78–88`:

```python
PROMPT_TEMPLATE = (
    "Given the reference fashion image and the modification instruction, "
    "write a concise caption describing the target fashion item after "
    "applying the modification."
)
GENERATION_KWARGS = {"max_new_tokens": 128, "num_beams": 1, "do_sample": False}
```

> **Update (2026-04-30, post-analysis).** `max_new_tokens` was
> subsequently bumped to **256** — see the corresponding deviations
> bullet above. The analysis below references the original `128`
> because the gap argument concerns the binding *prompt* constraint
> (`"concise"`), not the generation cap. The bump is a pure safety
> margin against silent truncation if a future prompt elicits longer
> outputs; under the current zero-shot prompt + greedy decoding it
> changes nothing.

A real query is `[image] + PROMPT_TEMPLATE + "\n\nModification: {mod}"`.
Detailed treatment lives in `Documentation/notes_baseline.md` Q9; the
parts that bear on Plan_3 specifically:

**FACap caption-length distribution (empirical, n=1000 dress
targets, measured from `runs/smoke_oracle/caption_db/metadata.jsonl`).**

| Stat | chars | ~words |
|---|---|---|
| min | 201 | 40 |
| p10 | 427 | 85 |
| median | 566 | 113 |
| mean | 538 | 108 |
| p90 | 615 | 123 |
| max | 651 | 130 |

FACap dress captions are **~110 words median**, range mostly 85–125
words — longer than a "80-word" rule-of-thumb estimate.

**Why length matters for SBERT retrieval.** `all-MiniLM-L6-v2` is a
**symmetric** encoder using mean-pooling — the 384-d vector reflects
all topics in the input, weighted by frequency. The right framing is
**topical-scope mismatch**, not length per se. A long caption with
extra topics not in the target *dilutes* the relevant topics; a short
caption misses topics the DB caption has. The current `"concise"`
instruction + `max_new_tokens=128` (~95-word ceiling) will likely
yield 30–60 word outputs against the ~110-word-median DB above — a
real gap.

**Should we match DB style?** Mostly yes — both sides are
fashion-item captions (symmetric retrieval). Match length/scope,
fashion-specific vocabulary (silhouette, neckline, hemline), and
attribute order. Don't match generic boilerplate (*"This is a..."* /
*"The image shows..."*) which cancels out, and don't hallucinate
content to fill space (lands embeddings randomly).

**Standard moves in IR / RAG.**

| Move | Idea | When to use |
|---|---|---|
| **Symmetric encoder + style-matched prompt** | Few-shot or explicit style instructions to nudge VLM output toward DB style; encode both with the same SBERT-style model | Our case: DB and queries are same content type |
| **Asymmetric encoder** | Use a model trained with separate "query:" / "passage:" prefixes (BGE, E5, msmarco-MiniLM) that learns to bridge query↔doc style differences | When queries are short / questions but docs are long — not our case |
| **Two-stage rerank** | Dense first-stage retrieves top-50; a cross-encoder reranks them | When first-stage Recall@50 is good but Recall@1 is poor |
| **Hybrid (BM25 + dense)** | Sparse (BM25) + dense (SBERT) scores combined | When key attributes are rare/specific words (brand names, exact colors) |
| **Few-shot prompting** | Show 2–3 (image, modification, ideal_caption) triplets in the prompt | Standard cheap fix to nudge output style |

The professional default for composed image retrieval via captions
is roughly: (1) symmetric SBERT-family encoder both sides ✅,
(2) style-matched prompt, (3) few-shot if needed,
(4) cross-encoder reranker, (5) swap encoder only if 1–4 exhausted.

**Decision (2026-04-30): zero-shot first, evidence-driven
iteration.** Hardcoding a style instruction now (e.g. *"~80 words,
mention color, silhouette, fabric, length, neckline"*) feels too
prescriptive *before we have a number*. For Plan_3:

1. Run zero-shot (current `PROMPT_TEMPLATE`) → first metrics +
   `qualitative/results.jsonl`.
2. Read ~5 generated captions. If they're 30-word generic vs.
   FACap's 110-word fashion-rich, the style mismatch is real and
   visible in the data.
3. Iterate the prompt with style guidance only if the numbers
   warrant it.

Principle: don't optimize a prompt against an *imagined* problem.
Run the experiment, look at the failures, then prescribe.

`run_baseline.py` gates against stale-DB reuse on two axes:
1. **CLI args.** Rerunning with the same `--run-name` but a different
   `--encoder`, `--n-eval`, `--db-size`, `--seed`, `--category`, or
   `--split` raises `RuntimeError` with a diff of the offending args.
2. **FACap data revision.** If the local `data_exploration/datasets/facap-repo/`
   HEAD changes between when the DB was built and the current run, the
   gate also raises (the recorded `facap_commit_sha` no longer matches
   the current checkout).

In either case, delete the run dir or pick a fresh `--run-name` to
proceed.
