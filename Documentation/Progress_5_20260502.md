## 2. Implementation Completed

### New files

**`src/training/contrastive_model.py`** — `ContrastiveQwen2VL`
- Loads speechQwen2VL base with Stage-2 LoRA merged, then attaches a fresh Plan-5 LoRA (rank 32, alpha 64, targets: q/k/v/o_proj only)
- Query embedding: last hidden state at EOS position (dim 3584) → 2-layer MLP projection head (3584 → 1024 → 512, GELU + LayerNorm, L2-normalized output)
- Target dim 512 matches frozen FashionCLIP image tower — no adapter needed on gallery side
- 32.2 M trainable params total (LoRA ~28 M + proj head ~4.2 M)

**`src/training/loss.py`** — symmetric InfoNCE with learned temperature
- Temperature τ is a learned scalar: τ⁻¹ = exp(logit_scale), initialized at 14.29, clamped to ≤ 100
- With `--gather` flag: all-gathers query and target embeddings across all GPUs before computing the full NxN similarity matrix → effective batch = per_gpu_batch × n_gpus negatives

**`src/training/dataset.py`** — `FacapContrastiveDataset`
- 53,686 train / 500 dev / 1,000 headline / 2,663 excluded IDs
- Returns `(cand_image, mod_text, target_id)` per sample
- Target embeddings looked up from pre-built cache at runtime (not re-encoded each step)

**`src/training/online_eval.py`** — three-condition retrieval probe
- **normal**: real modification text + candidate image (primary metric)
- **mod_stripped**: modification text removed — measures image-only nearest-neighbor baseline
- **mod_shuffled**: modification text word-order shuffled — tests bag-of-words vs. semantic understanding
- Reports R@1/5/10/50 and median rank for each condition

**`src/training/train_plan5.py`** — custom Accelerate DDP training loop

*Optimizer:* Three param groups with separate LRs — standard `torch.optim.AdamW` (no 8-bit):
- LoRA params: lr=2e-5
- Projection head: lr=1e-4
- logit_scale: lr=2e-5

*Eval cadence:* epoch-based — every 0.5 epoch (`--eval-epochs 0.5`). Dev (500 queries) and headline (1000 queries) run together at every eval trigger so every checkpoint has both diagnostic and authoritative metrics at the same step. Uses `_should_eval(step)` which checks epoch boundary crossings, so it fires correctly regardless of batch size. First eval fires at ~0.25 epoch (alive-bar check).

*W&B logging (every 20 steps):* `train/loss`, `train/logit_scale`, `train/epoch`, `train/lr_lora`, `train/lr_proj`, `train/grad_norm`

*W&B logging (every 0.5 epoch, dev set):* `dev/r1_normal`, `dev/r5_normal`, `dev/r10_normal`, `dev/r50_normal`, `dev/median_rank`, `dev/r10_stripped`, `dev/r10_shuffled`, `dev/sensitivity_gap`

*W&B logging (every epoch end, headline set):* `headline/r1`, `headline/r5`, `headline/r10`, `headline/r50`, `headline/median_rank`

*Checkpointing:* saves LoRA weights (`vlm_lora/`) and projection head (`proj_head.pt`) at end of each epoch to `--run-dir/ckpt_epochN/`

**`scripts/run_plan5.sh`** — launch wrapper (sets env vars, calls accelerate launch)

## 5. DDP Debugging Log (6 failed runs before first clean run)

All debug runs are in `runs/plan5/run_20260502_0703xx` through `071114`.

### Bug 1 — Wrong conda env (`run_20260502_070302`)
**Error:** `ModuleNotFoundError: No module named 'peft'`  
**Cause:** `accelerate` resolved to the base Python 3.13 environment which has no `peft`.  
**Fix:** Use full path: `/home/zhuoyuan/miniconda3/envs/fashion_retrieval/bin/accelerate`

### Bug 2 — All 8 DDP ranks loaded to GPU 0 (`run_20260502_070408`)
**Error:** `CUDA out of memory` on GPU 0 (process showing 18.5+ GB); all other GPUs had <1 GB used.  
**Cause:** `device_map="cuda"` in HuggingFace triggers model parallelism across ALL GPUs — not what we want in DDP where each rank owns one full model copy.  
**Fix:** Pass `device_map=f"cuda:{accelerator.local_process_index}"` so each rank loads to its own GPU.

### Bug 3 — PeftModel LoRA adapter always loaded to GPU 0 (`run_20260502_070919`)
**Error:** OOM on GPU 0 even after device_map fix.  
**Cause:** `PeftModel.from_pretrained(model, lora_repo, device_map=device_map)` — the `device_map` kwarg triggers HF's `dispatch_model` (parallelism) rather than controlling the load target. Internally `infer_device()` always returns `"cuda"` (GPU 0) regardless.  
**Fix:** Use `torch_device=device_map` instead:
```python
model = PeftModel.from_pretrained(model, lora_repo, torch_device=torch_device)
```
where `torch_device = device_map if isinstance(device_map, str) else None`.

### Bug 4 — DDP bucket mismatch (`run_20260502_071114`)
**Error:** `RuntimeError: Expected to have finished reduction in the prior iteration before starting a new one. ... find_unused_parameters=True`  
**Cause:** Reentrant gradient checkpointing creates a separate autograd graph per checkpoint segment; some LoRA parameters appear unused in DDP's bucket tracking even though they receive gradients.  
**Fix:** Two changes in `train_plan5.py`:
```python
# In contrastive_model.py:
vlm.gradient_checkpointing_enable(gradient_checkpointing_kwargs={"use_reentrant": False})

# In train_plan5.py:
from accelerate.utils import DistributedDataParallelKwargs
ddp_kwargs = DistributedDataParallelKwargs(find_unused_parameters=True)
accelerator = Accelerator(log_with=log_with, kwargs_handlers=[ddp_kwargs])
```

## 12. server 10 Launch Debugging

Several attempts were needed to get server 10 running. Summary of issues and fixes:

### Issue 1 — HF cache in wrong location (first rsync)
The initial rsync copied the model to `/ssd1/zhuoyuan/hf_cache/models--DanJZY--...` on server 10. Because `HF_HOME=/ssd1/zhuoyuan/hf_cache`, HuggingFace's `huggingface_hub` layer looks in `$HF_HOME/hub/models--...` = `/ssd1/zhuoyuan/hf_cache/hub/models--...`. The files were not in `hub/`, so HF showed `Fetching 4 files: 0%` and started downloading to `~/.cache/` (NAS). This ran silently for ~7 hours before being caught.

**Root cause:** `TRANSFORMERS_CACHE=/ssd1/zhuoyuan/hf_cache` (how the model was found on server 6) vs `HF_HOME/hub/` (where `huggingface_hub` / PeftModel expects it). The two cache lookup paths are different: `transformers` uses `TRANSFORMERS_CACHE` directly for the base model, while `huggingface_hub` / PeftModel uses `HF_HOME/hub/`. These only agree when `TRANSFORMERS_CACHE` = `HF_HOME/hub/`.

**Fix:** Set all three env vars explicitly in every SSH launch command (not relying on profile inheritance):
```bash
HF_HOME=/ssd1/zhuoyuan/hf_cache \
TRANSFORMERS_CACHE=/ssd1/zhuoyuan/hf_cache \
HF_DATASETS_CACHE=/ssd1/zhuoyuan/hf_cache \
```
And keep the model at `TRANSFORMERS_CACHE/models--...` (no `hub/` subdir), matching how server 6 is set up. Deleted 8.3 GB of partial NAS downloads each time they appeared.

### Issue 2 — bs=256 OOM on A6000
**Error:** `torch.OutOfMemoryError: CUDA out of memory. Tried to allocate 4.50 GiB. GPU 6 has total capacity 47.50 GiB of which 1.03 GiB is free. Process has 46.47 GiB in use.`

Qwen2-VL-7B produces ~1000 visual tokens per fashion image at full resolution. At bs=256 per GPU × ~1100 tokens/sample, the MLP intermediates during gradient checkpointing recompute (gate_proj + up_proj × intermediate_dim=18944) overflow the 47.5 GB A6000.

The practical limit on A6000 is bs=64 per GPU, not the ~bs=160 you might naively expect from 2× VRAM.

## 13. Design Decision: Drop 8-bit AdamW

8-bit AdamW saves ~200 MB of optimizer state for our 32.2M trainable params (fp32 → 8-bit: ~258 MB → ~64 MB). That is negligible relative to total VRAM and not worth the bitsandbytes dependency or potential quantization noise. **Future runs will use standard `torch.optim.AdamW`**. The current bs=16 and bs=256 runs (launched 2026-05-02) still use 8-bit and are left as-is; change takes effect on the next relaunch.

## 14. W&B Metrics Reference

### What is R@K?

The gallery has ~59,000 product images. For each query (a candidate image + modification text), the model ranks all 59,000 images by cosine similarity to the query embedding. R@K = fraction of queries where the true target image appears in the top-K results.

- **R@1 = 0.22** → the correct target is ranked #1 for 22% of queries
- **R@10 = 0.22** → the correct target is in the top 10 for 22% of queries
- **R@50** → correct target in top 50; useful for checking if the model is "in the right neighborhood"

Phase-A baseline (FashionCLIP image tower, frozen): R@1=0.258, R@10=0.533. Plan-5 trains a VLM query encoder to beat this using text+image queries against the same frozen gallery.

---

### Logging cadence

**Every 20 steps (`train/`):** loss, logit_scale (τ⁻¹), epoch, lr_lora, lr_proj, grad_norm.

**Every 0.5 epoch (`dev/`):** R@1/5/10/50 and median_rank on normal queries; R@10 on stripped and shuffled queries; sensitivity_gap. (500-query dev set.)

**Every epoch end (`headline/`):** R@1/5/10/50 and median_rank on the full 1000-query headline set. Authoritative number to report.

There is no "validation loss" — this is a retrieval task. The eval metric is R@K (did the true target appear in the top-K ranked results?).

---

### How to interpret each dev metric

**`dev/r10_normal`** — primary metric. Should increase monotonically. If it plateaus before epoch 1, something is wrong (mode collapse or bad learning rate).

**`dev/r10_stripped`** — what a model that ignores text entirely would score (pure visual nearest neighbor). We want this to stay low and grow slowly. If it grows as fast as `normal`, text is not helping.

**`dev/r10_shuffled`** — what a bag-of-words model scores (word order doesn't matter). If this is close to `normal`, the model treats text as a bag of keywords, not as a sentence. Ideally `shuffled` < `normal`.

**`dev/sensitivity_gap = r10_normal − r10_stripped`** — must be > 0 at all times. A gap ≤ 0 means the modification text is actively hurting retrieval (model is confused by text). A growing gap is the clearest sign the model is learning to use text. If it ever goes negative mid-training, check the loss curve for signs of collapse.

**`dev/median_rank`** — lower is better (1 = always ranked first, 59048 = always ranked last). Starts near 30000 (random), should drop to under 500 by epoch 1 if training is working.

**`train/logit_scale`** (τ⁻¹) — the InfoNCE temperature is learned. τ⁻¹ = exp(logit_scale). If it goes above 80–100, similarities saturate and gradients die. We clamp it at 100. Healthy value: 10–30 early training, potentially rising to 30–50 at convergence.

**`train/grad_norm`** — should be smooth (roughly 0.5–5.0 range). Sudden spikes (>50) → numerical instability, check LR. Trending to zero → gradient vanishing.

---

### Key charts for mentor presentations

1. **`train/loss` over steps** — proves training is working (loss decreasing from ~log(N_neg) at init)
2. **`dev/r10_normal` over steps** — the retrieval accuracy curve; compare to Phase-A baseline (0.533)
3. **`dev/sensitivity_gap` over steps** — proves the model uses modification text, not just image similarity

Set W&B x-axis to **Step** (not Relative Time). The default "100m" axis is wall-clock minutes, not steps.

## 16. CUDA OOM Bug: lm_head Allocates 2.31 GB

### Problem

Three bs=16 runs on server 6 (v1–v3) crashed at backward pass:

```
torch.OutOfMemoryError: Tried to allocate 2.31 GiB.
GPU 1 has a total capacity of 23.68 GiB of which 1.17 GiB is free.
```

Traceback: `contrastive_model.py:154` → `lm_head` forward in `modeling_qwen2_vl.py:1480`.

### Root Cause

Qwen2-VL-7B's `lm_head` projects `(B, seq_len, 3584)` → `(B, seq_len, 152064)` for token generation. At bs=16 with ~509 tokens/sample:

```
16 × 509 × 152064 × 2 bytes (bf16) ≈ 2.31 GB
```

Plan-5 never uses these logits — only `hidden_states[-1]` at the EOS position is needed. The full lm_head projection was being computed and cached through the backward pass.

### Fix (`src/training/contrastive_model.py`)

Replace lm_head with a 1-output stub that eliminates the allocation entirely:

```python
_lm_head = vlm.base_model.model.lm_head
vlm.base_model.model.lm_head = nn.Linear(
    QWEN2VL_HIDDEN_DIM, 1, bias=False, dtype=torch.bfloat16
).to(next(_lm_head.parameters()).device)
del _lm_head
```

`lm_head` is not a LoRA target module so no trainable parameters are affected. The stub is re-created from scratch at each launch and is not saved in checkpoints. VRAM savings: **≈2.3 GB** — enough to run bs=16 cleanly on a 24 GB RTX 3090.

```bash
--resume-from <ckpt_dir>   # e.g. runs/plan5/run_.../ckpt_epoch3
--start-epoch <N>           # 0-based epoch index (N=3 when resuming ckpt_epoch3)
```

`global_step` is initialized to `start_epoch × steps_per_epoch` so W&B x-axis aligns when overlaid with the original run.

### Temperature save/restore fix

v3 checkpoints (epoch 1/2/3) did not save `loss_fn.pt` — only `vlm_lora/` and `proj_head.pt`. The learned temperature (τ⁻¹ ≈ 14.39 at epoch 2.5) reset to initialization (14.29) on resume, causing R@10 to drop from 0.154 → 0.114.

Fix: `train_plan5.py` now saves loss function state at every checkpoint:

```python
torch.save(
    accelerator.unwrap_model(loss_fn).state_dict(),
    ckpt_dir / "loss_fn.pt",
)
```

The `if loss_fn_pt.exists()` guard in the resume block allows graceful loading from old checkpoints that lack it.

**Note on W&B continuity:** Resume does not continue the original W&B run (no run ID was saved). The continuation run starts fresh at the correct global_step so curves align when overlaid manually in W&B.
