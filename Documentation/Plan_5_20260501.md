# Plan 5 — Single-GPU contrastive fine-tune (design)

## Context

Phase A is closing. Plan_3 baseline (caption-then-retrieve, MiniLM-L6)
landed at R@1 = 0.084. The Phase-A encoder ablation (`Progress_3` / M4)
lifted that to **R@1 = 0.258, R@10 = 0.533** with frozen Marqo
FashionCLIP — that is the bar Phase B has to beat.

This plan is the *design* (not yet execution) for the **first
single-GPU contrastive end-to-end run** — Option 3 from the project
proposal. It was originally numbered Plan_4 in earlier `Plan_overview`
revisions; on 2026-05-01 the conversation forked into two parallel
chats — one for caption-quality ablation, one for contrastive
training — and the caption-quality track took the Plan_4 slot.
Everything from "old Plan_4" downwards now shifts by +1; this plan is
**Plan_5**.

Audio modality is OFF for Plan_5 (text + image only). Audio extension
is deferred to Plan_8+, per the proposal's "text first" ordering.

**Key decisions carried in:**
- VLM: `speechQwen2VL` (Qwen2-VL-7B base + Stage-2 LoRA). Audio path
  unused at this stage.
- Frozen target encoder: Marqo FashionCLIP **image** tower (the same
  encoder that gave Phase A's R@1 = 0.258).
- Eval slice: same 1000-query / ~58 k-target slice as Plan_3 — head-to-
  head comparable.
- **Hardware split (single-GPU sanity check, not multi-GPU yet):**
  - **Training**: single A6000 (48 GB) or equivalent (A100 40/80 GB,
    H100 80 GB). Drops the 3090's 8-bit-AdamW gymnastics and lets us
    bump micro-batch.
  - **Eval-only sweeps + (optional, deferred) Step 0 debug runs**:
    any modest GPU (3090, A6000, A100 — they're inference-only
    forwards over a frozen tower; whichever is free is fine).
  - Plan_5 deliberately stays **single-GPU per step** to keep the
    debug surface small. Multi-GPU is Plan_7.

### Why this section exists

Earlier drafts of this plan included a Step 0 that was meant to gate
training. It went through two redesigns under cross-review:

1. **Original**: image→image retrieval using the eval *target* image
   as both query and a gallery row. **Broken** — that's self-retrieval
   (eval target IDs all live in the gallery), so any encoder scores
   ≈ R@1 = 1.0. Not a meaningful ceiling.
2. **First revision**: candidate-only retrieval *floor* — encode the
   reference image, retrieve against the gallery, ignore mod text.
   "How well does pure visual similarity find the target?" Reframed
   as a floor, not a ceiling.
3. **Second revision (proposed)**: caption→image cross-modal sanity —
   use Phase A's generated captions as queries, FashionCLIP-text →
   FashionCLIP-image retrieval. "Is FashionCLIP's text/image space
   well aligned in the fashion domain?"

### Why none of these drives a decision for Plan 5

After working through the candidate uses of a Step 0 number, none of
them actually changes Plan 5's training plan:

| Candidate use | Why it doesn't actually inform a decision |
|---|---|
| Choose target encoder | Already locked to FashionCLIP-image based on Phase A's text-side ablation; no Step 0 result would override that |
| Trigger a pause / redesign | No better target tower available — even a poor result wouldn't stop training |
| Inform Plan 6 (alternate target tower) | Plan 5's own training results are far more informative than Step 0; that decision can wait |
| Calibrate the alive bar (R@10 ≥ 0.35 by 0.25 epoch) | The 0.35 threshold is a defensible mid-point between Phase A's MiniLM (0.24) and Marqo (0.53) — doesn't need empirical calibration |
| Detect mode collapse (model just doing visual NN) | Verification #2 (zero out mod-text mid-training, R@10 must stay ≥ 0.10) tests exactly this in a more actionable way |
| Cross-modal alignment sanity (caption→image) | FashionCLIP's text and image towers are jointly trained — alignment is expected to be reasonable; even if it's not, we'd still run Plan 5 with FashionCLIP-image as the target |

The proposed selection of encoders to sweep was Marqo FashionCLIP-image,
Marqo FashionSigLIP-image, DINOv2-large, and original SigLIP/CLIP-image.

### When to revisit

Run Step 0 (one of the variants above) **only if**:
- Plan 5 doesn't converge → run caption→image cross-modal sanity to
  check whether the FashionCLIP text/image alignment is the problem
- Plan 5 converges but R@10 ≈ candidate-only level → run the
  candidate-only floor sweep to confirm mode collapse
- Plan 6 needs to decide on swapping target tower → run the full
  4-encoder sweep to compare DINOv2 / FashionSigLIP / etc. against

## Design decisions (committed)

### 1. Loss + target encoder — symmetric InfoNCE, image-side targets

Train the speechQwen2VL query embedding against a **frozen Marqo
FashionCLIP image-tower** target. Symmetric InfoNCE, in-batch negatives
only (no hard-negative mining in v1).

**In other words:** adjust Qwen2-VL's weights so that, given a query
`(ref_image, mod_text)`, it produces a 512-dim vector that lands
close to the 512-dim vector that frozen FashionCLIP's image-half
produces for the correct target dress image. After training, those
two vectors live in the same space — that's what makes
nearest-neighbor retrieval work. (See `notes.progress5_contrastive_learning.md`
Q14 for the full unpacking of "against ... target.")

Why image-side targets, not the FashionCLIP-text route from Phase A:
the encoder ablation showed FashionCLIP-text already ceilings the
system at R@1 ≈ 0.26 — using its text tower as the *target* would
replicate that ceiling. Image-side targets sidestep the captioning
bottleneck entirely.

The alternative (distillation-flavored, MSE against frozen FashionCLIP
*text* embedding) is parked as a Plan_6 ablation — simpler, but with
exactly the cap above.

### 2. Embedding source from Qwen2-VL

Last hidden state at the EOS position
(`outputs.hidden_states[-1][:, -1, :]`, dim 3584) followed by a
**2-layer MLP projection head** (3584 → 1024 → `D_target`,
GELU + LayerNorm, L2-normalized output).

`D_target` is **set at construction time from the chosen target
encoder's output dimension** (queried from the target embedding cache
metadata, not hard-coded). Defaults: 512 for Marqo FashionCLIP-image
(v1's choice); 768 for FashionSigLIP-image; 1024 for DINOv2-large.
`contrastive_model.py:ContrastiveQwen2VL.__init__` reads
`target_cache.embedding_dim` from the cache metadata (written
alongside `target_emb_cache.npy`) and builds the projection head's
final layer accordingly. No target-side adapter is ever added.

Mean-pool / attention-pool ablations deferred to Plan_6.

### 3. Trainable parameters

**Stage-2 LoRA inheritance** — `speechQwen2VL = Qwen2-VL-7B base + Stage-2
LoRA adapter` (rank/alpha 64/128 on `q/k/v/o_proj` + `gate/up/down_proj`,
trained for the audio path). Plan 5 does **not** want a second co-existing
PEFT adapter (double-wrapping is brittle and the Stage-2 audio adapter
isn't relevant when the audio modality is off). Decision:
**(a) Load base Qwen2-VL → load Stage-2 adapter → `merge_and_unload()`
to fold Stage-2 weights into the base → attach a fresh, named Plan 5
LoRA adapter on top.** This preserves whatever Stage-2 contributed to
general representations and gives Plan 5 a clean single-adapter
training surface. Implemented by the `_load_qwen2vl_base()` helper
extracted from `vlm_caption.py` (see §Modify); Plan 5 calls it with
`merge_stage2=True`. The merge runs at every training startup
(~30s overhead).

**Possible future optimization (deferred):** a one-time prep script
that does the Stage-2 merge once and saves the result to a local
`stage2_merged_base/` checkpoint, so subsequent runs skip the merge.
Mathematically equivalent — pure engineering convenience. Worth
revisiting once Plan 6 starts queuing many ablation runs and the
per-startup merge cost becomes annoying.

- **Plan 5 LoRA on LLM decoder**: rank 32, alpha 64, target =
  `q/k/v/o_proj` only (drops `gate/up/down_proj` from Stage-2's
  recipe — embeddings don't need MLP-projection capacity for this
  task). Estimate ~30 M trainable.
- **Projection head**: ~4.2 M trainable, fp32.
- **Frozen**: vision tower, FashionCLIP image tower, all merged
  base + Stage-2 weights.
- **Optimizer**: plain AdamW (fp32 momentum + variance) on the
  trainable params. No `bitsandbytes` 8-bit optim needed — the
  A6000's 48 GB is plenty of headroom.

VRAM budget on an A6000 (48 GB): 14–15 GB base bf16 + ~150 MB LoRA +
head + ~600 MB FashionCLIP-image + ~280 MB AdamW state (fp32 for
~35 M trainable) + bf16 activations w/ gradient checkpointing on the
LLM decoder. Per-step **true** micro-batch is what matters for the
contrastive loss (see §4 below) — not the gradient-accumulated
effective batch.

### 4. Batch construction (revised after cross-review)

> **Cross-review correction (2026-05-01).** An earlier draft claimed
> "micro-batch 8 + grad-accum 8 → effective batch 64 (63 in-batch
> negatives per anchor)". That is **wrong**. With a standard custom
> loop, the InfoNCE softmax is computed inside each micro-batch — so
> grad-accum gives 7 in-batch negatives per anchor, not 63.
> Accumulated gradients do not back-propagate cross-micro-batch
> contrastive structure. Plan revised below.

The contrastive loss needs *true* batch size — only embeddings present
in the same forward pass create negatives.

**v1 plan**:
1. **Profile first.** First implementation step (Step 0 is now
   skipped from the default path — see the Step 0 section): time +
   VRAM-profile a single forward+backward at micro-batch 16, 24, 32, 48,
   64 with grad-checkpointing on, on the actual A6000 we'll train on.
   Pick the largest that fits comfortably (≤ ~40 GB to leave headroom
   for fragmentation + the FashionCLIP image tower co-resident).
2. **Use that micro-batch as the *actual* contrastive batch.** No
   grad-accum for the contrastive loss in v1. (Grad-accum *can* be
   layered on top later for stable optimizer steps, but it does not
   add negatives.)
3. **Expected outcome**: a true batch in the 24–48 range on a single
   A6000. Each anchor sees `(B - 1)` in-batch negatives. Document the
   actual number that lands in `Progress_5_<date>.md`.
4. **If true batch < 32**, this is borderline-small for InfoNCE.
   Options at that point:
   - (a) Implement **gradient cache** (Chen et al. 2021 — two-pass:
     compute all embeddings without grad first to get the full softmax
     denominator, then re-forward in chunks with the loss attached
     per-chunk).
   - (b) Accept smaller batch and revise the §7 thresholds downward.
   - (c) Add a momentum-encoder negative queue (MoCo-style) — but
     that's Plan 6 territory.
   - (d) **Multi-GPU `all_gather` from day 1** — see §8 Compute target.
   - Decision deferred until profile numbers land; gradient cache is
     the preferred fallback because it preserves the contrastive geometry
     without changing the algorithm.

   **Multi-GPU readiness (design now, decide later):** the symmetric
   InfoNCE loss is implemented with an optional `gather: bool = False`
   flag. When `True` (multi-GPU run), each GPU's `(query, target)`
   embeddings are `all_gather`-ed across the world before computing the
   softmax denominator — so anchor i on GPU 0 sees negatives from all
   GPUs' targets, not just its own GPU's batch. When `False`
   (single-GPU run), the gather is a no-op. **No code-path divergence**:

   ```python
   if gather and dist.is_initialized():
       q = torch.cat(GatherLayer.apply(q), dim=0)
       t = torch.cat(GatherLayer.apply(t), dim=0)
   # ... rest of the loss is the same
   ```

   Single-GPU runs pay zero cost; multi-GPU runs get true cross-GPU
   negatives. This is the only loss-side concession to Plan 7.
5. **No hard-negative mining in v1.**
6. **Target embedding cache.** Image embeddings pre-computed once for
   the full ~58 k gallery (frozen tower → deterministic) and cached at
   `runs/plan5/target_emb_cache.npy` plus `target_emb_cache.meta.json`
   (fields: `encoder_id`, `embedding_dim`, `gallery_ids` in row order,
   `image_hash`). Loaded into RAM at train time; same cache reused as
   the eval gallery.
7. **Sampler check**: redraw on duplicate `target_id` within a batch.

### 5. Training loop — custom PyTorch

Custom loop, **not** HF Trainer or `sentence-transformers` Trainer.
Reasons:
- HF Trainer's `compute_loss` assumes one model + per-sample labels.
  Symmetric InfoNCE wraps two encoders and a batch-level loss.
- `sentence-transformers`'s contrastive Trainer assumes a
  `SentenceTransformer` backbone wrapper around the model — wrapping
  Qwen2-VL multimodal in that costs more than it saves.
- Mentor's explicit guidance: don't use HF Trainer blindly here.

A ~250-line custom loop with manual bf16 + grad-checkpoint is the
smallest debuggable surface, and reuses cleanly into Plan_7's
`accelerate launch` for multi-GPU. (No grad-accum for the contrastive
loss in v1 — see §4.)

**Accelerate-readiness from day 1**: write the loop using
`accelerate.Accelerator()` from the start. Single-GPU runs (`python
train_plan5.py`) work normally; multi-GPU runs (`accelerate launch
train_plan5.py`) work without code changes. The only multi-GPU-specific
piece is the `gather=True` flag passed to the InfoNCE loss (see §4).
No-op when single-GPU, no extra dependency cost.

### 6. Online eval (revised after cross-review)

> **Cross-review correction (2026-05-01).** The original draft used
> the Plan 3 1000-query slice as *both* the online validation set
> (checked every 500 steps for early-stop / hyperparameter decisions)
> *and* the final headline benchmark. That turns the headline number
> into a fitted quantity. Split into two slices below.

Two slices, populated under L2 exclusions (see Resolved Q2):

1. **Online dev slice** — ~500 fresh queries carved out of the train
   split *after* L2 exclusion, never seen by training. Used every
   500 optimizer steps for the §7 **alive bar** and any
   hyperparameter / early-stop decisions. Cheap to evaluate (~1 min
   on the cached gallery).
2. **Headline benchmark slice** — the original Plan 3 1000-query
   slice. **Touched only at end of epoch and at the final
   convergence point** for the §7 **promising bar** (end of
   epoch 1) and **headline-win bar** (convergence). Never used to
   make in-training decisions.

Both slices use the same ~58 k image-tower gallery (cached). Reuses
`src/baseline/retrieve.py:top_k` and `src/baseline/eval.py:compute_metrics`.

The dev slice is constructed once at startup by `dataset.py` and
its `target_id` set is also added to the L2 exclusion set (so train
never sees dev targets either). Slice composition is deterministic
from the random seed; logged to `runs/plan5/<run_name>/dev_slice.json`.

### 7. Sanity-check criteria

Each bar binds to a specific slice and a specific schedule. Note:
under L2 train filtering (~56k triplets) and the §4 profile-chosen
true-batch protocol, one epoch is roughly 1.2k–2.3k optimizer steps
depending on the chosen batch size — so the alive bar is sized as a
*fraction of an epoch*, not in raw steps.

| Bar | Slice | Threshold | When | Action |
|---|---|---|---|---|
| **Alive (warning)** | dev (~500 queries) | R@10 ≥ 0.20 | by 0.25 epoch | **Warning only** — log a flag, keep training. Used to surface "this run looks anemic, you may want to look" without auto-stopping. |
| **Promising** | headline (Plan 3 1000-query) | R@10 ≥ 0.55 | at end of epoch 1 | Reference target; used in Progress write-up. |
| **Headline win** | headline (Plan 3 1000-query) | R@1 ≥ 0.28 | at convergence (or end of training) | The actual goal. |
| **Sensitivity gap** | dev | R@10(normal) − R@10(mod-stripped) > 0 | by 0.25 epoch | **Warning only** — log a flag if violated. |

**Hard stops** (training auto-aborts) are reserved for actual numerical
failures: NaN/Inf in loss, NaN/Inf in `logit_scale`, OOM, dataloader
errors. The bars above are "is the run healthy?" indicators — they
warn but never auto-kill, so a slow-warmup run isn't accidentally
terminated. Dev R@K is logged every 500 steps from step 0 onward so
trends are visible well before the formal checkpoints.

If a warning fires, the human looks at the trajectory:
- R@10 = 0.18 at 0.25 epoch but **rising** (e.g., 0.05 → 0.12 → 0.18) → keep going, the model is learning, just slower than expected
- R@10 = 0.18 at 0.25 epoch and **flat** (0.17 → 0.18 → 0.18) → debug
- Sensitivity gap negative or flat-zero → debug regardless of absolute R@10

### 8. Compute target (decision deferred to post-smoke-test)

Two viable paths depending on which hardware is available when training
is ready:

**Path A — single A6000-class GPU (48 GB)**: Plan 5's original design.
True micro-batch 24–48, single-GPU, simplest debug path. Defers Plan 7
multi-GPU work entirely.

**Path B — multi-GPU on 8× 3090-class GPUs (24 GB each)**: per-GPU
micro-batch ~4 (based on the original 3090 budget), 8 GPUs × 4 =
effective contrastive batch 32 with `all_gather`. Roughly comparable
batch to Path A. Requires `accelerate launch` + the `gather=True`
flag in §4. Pulls some of Plan 7's work forward.

**Decision rule**: pick at smoke-test time. If a single A6000-class box
is free *and* smoke test fits comfortably at micro-batch 24+, take
Path A. If A6000-class hardware is contested or the model needs more
batch than Path A gives, take Path B. The training code is written to
support both without modification — single-GPU runs use
`python train_plan5.py`, multi-GPU runs use
`accelerate launch train_plan5.py`.

**Why design for both rather than committing now**:
- Contrastive learning is batch-sensitive; Plan 5 Risk #5 already flags
  "true batch < 32" as a potential problem. Multi-GPU with `all_gather`
  is the cleanest mitigation.
- The marginal code cost (1 flag in InfoNCE + accelerate-readiness in
  the loop) is small and pays off whether we use multi-GPU now or in
  Plan 7.

**Time estimate honest:** "5–8 hr / epoch" is a guess based on A6000
single-GPU. Multi-GPU on 8× 3090-class should be in the same ballpark
for total wall-clock (more GPUs but slower per-GPU compute roughly
cancels). Real estimate comes from the smoke test + first half-epoch
on whichever hardware we land on.

## Risks (ranked by severity)

1. **Composed-retrieval collapse.** LoRA + contrastive can collapse
   into one of two degenerate regimes: (a) text-bag-of-words encoder
   that ignores the reference image, or (b) visual nearest-neighbor
   that ignores the modification text. The second is more dangerous
   for this project — a model that "works" on visual similarity alone
   would still score reasonably but is not actually doing composed
   retrieval. *Mitigation*: the Verification #2 sensitivity probe
   (normal vs mod-stripped vs mod-shuffled R@K, every dev eval) tests
   both directions. If `R@10(normal) − R@10(mod-stripped) ≤ 0` by
   0.25 epoch, stop and debug.
2. **Temperature blowup / mode collapse.** Learned inverse temperature
   drifts → similarities saturate at one-hot. *Mitigation*: parametrize
   as `logit_scale = log(1/τ)` (CLIP convention), clamp `exp(logit_scale) ≤ 100`
   (i.e. effective τ ≥ 0.01) on every optimizer step, log
   `exp(logit_scale)` and the loss every step, symmetric loss helps.
3. **Train/eval leakage.** The Plan 3 1000-query headline slice is the
   *last* 1000 train triplets, and Codex measured 770 / 836 unique
   eval target IDs leak into earlier train triplets under L1.
   *Mitigation*: L2 train filtering committed (Resolved Q2) — the
   exclusion set is `eval_target_ids ∪ eval_candidate_ids ∪
   dev_target_ids ∪ dev_candidate_ids`; assert at startup that no
   training triplet contains any excluded ID.
4. **OOM at long mod-text.** P99 mod-text + image tokens at the chosen
   micro-batch can blow the activation budget. *Mitigation*: cap
   `max_seq_length=128` on mod text; grad-checkpoint already on; the
   §4 profile-first step picks the micro-batch with measured headroom.
5. **Small-batch InfoNCE.** If §4's profile shows true micro-batch < 32,
   the contrastive loss has < 31 negatives per anchor, which is on the
   low end for in-batch InfoNCE. *Mitigation*: gradient cache (§4
   option a) preserves the loss geometry without enlarging the
   per-pass compute graph. Decision deferred until profile lands.
6. **Slow convergence.** Estimated ~5–8 hr per epoch on a single
   A6000 at the profile-chosen micro-batch (faster compute than 3090
   plus larger per-step batch). *Acceptable for v1*; multi-GPU is
   Plan_7.
7. **FashionCLIP image-tower bias.** Targets inherit its visual biases.
   *Acceptable for v1*; alternate target tower (DINOv2 / SigLIP) is
   Plan_6 ablation.

   gallery, so any encoder scores ~1.0). The first revision (a
   candidate-only retrieval *floor*) was fixed but, on a deeper look
   with the user, didn't actually inform any Plan 5 decision either —
   so Step 0 is now Optional / deferred. See the Step 0 section.

   That leaves the target-tower decision driven entirely by Phase A's
   clear text-side win for Marqo FashionCLIP:
   - **v1 commits to frozen Marqo FashionCLIP-image.**
   - Joint target-tower training, siamese (Qwen2-VL on both sides),
     and alternate target towers (DINOv2 / FashionSigLIP) all stay as
     Plan 6 ablations.

2. **Eval slice — L2 train filtering + separate dev slice for online
   eval.** Cross-review (Codex, 2026-05-01) endorsed L2 over L1 and
   L3, and additionally flagged that using the Plan 3 1000-query slice
   for *both* online eval (every 500 steps, drives early-stop /
   hyperparameter decisions) and the headline benchmark turns the
   headline number into a fitted quantity. Both points adopted.

   | Level | Definition | Cost vs Plan 3 head-to-head |
   |---|---|---|
   | **L1** | Exclude eval *triplets* from training only. | Same as Phase A. Allows the model to learn embeddings for eval target images via other triplets — Codex measured 770 / 836 unique eval target IDs leak under L1. |
   | **L2** ✅ | Also exclude eval `target_id`s and `candidate_id`s entirely — drop any training triplet whose target or candidate ID is in the eval set. | Stricter than Phase A. Codex measured 2,059 / 58,082 train triplets dropped (3.54%, not the 5–10% the earlier draft estimated). |
   | **L3** | Eval target images do not appear in the gallery during training either. | Breaks the shared-59k-gallery comparison with Phase A. Overkill. |

   **Decision (committed):** L2 train filtering, plus a **separate
   ~500-query dev slice** carved from train (after L2 exclusions)
   for online R@K every 500 steps. The Plan 3 1000-query slice
   becomes the **final headline-only** slice — touched at end of
   epoch and at convergence, never used for in-training decisions.
   See §6 (Online eval) for the implementation.

   Implementation: `dataset.py` builds an `eval_target_ids` ∪
   `eval_candidate_ids` ∪ `dev_target_ids` ∪ `dev_candidate_ids`
   exclusion set applied as a dataset filter; asserts at startup
   that no training triplet leaks any eval or dev ID.

## Verification — Plan_5 done end-to-end

1. **Plumbing smoke**: `train_plan5.py --smoke` runs 50 steps on 64
   fixed triplets; loss decreases monotonically (windowed avg);
   no NaN / Inf in loss or τ.
2. **Composed-retrieval sensitivity probe** (replaces the earlier
   single-direction "image-grounding probe"). At every dev-slice
   eval, run two retrievals on the same dev queries:
   - (a) **Normal**: full `(ref_image, mod_text)` query
   - (b) **Mod-stripped**: `(ref_image, "")` — empty modification
   - (c) **Mod-shuffled**: `(ref_image, mod_text_from_a_random_other_triplet)`
     — same ref image, *wrong* mod

   Three signals matter, all logged:
   - `R@10(normal)` — the headline trajectory
   - `R@10(normal) − R@10(mod-stripped)` — **must be positive and
     growing**. If this gap is ≤ 0 by 0.25 epoch, the model is doing
     pure visual NN and ignoring the modification text. Stop and debug.
   - `R@10(normal) − R@10(mod-shuffled)` — should be positive too;
     wrong mods should hurt more than no mod.

   This catches the failure mode the earlier "R@10(mod-stripped) ≥ 0.10"
   bar would have missed: a model that scores 0.50 on normal *and*
   0.50 on mod-stripped is a visual-NN collapse, not a working
   composed-retrieval system. The proposal is a composed retrieval
   project — the modification has to matter for the result to count.
3. **Alive bar (warning only)**: R@10 ≥ 0.20 on the **dev slice** by
   0.25 epoch (matches the §7 schedule). If below threshold, log a
   warning but **do not auto-stop** — human reviews trajectory and
   decides. Hard stops are reserved for NaN/Inf, OOM, and dataloader
   failures.
4. **Eval-harness sanity** (code check, not a ceiling): encode the same
   image as both query and gallery row through `online_eval.py`,
   confirm near-perfect retrieval modulo numerical noise — proves
   the gallery indexing and `top_k` plumbing are correct.
5. **Reproducibility**: re-run with same seed → dev R@10 at the 0.25
   epoch checkpoint matches within ±0.01.
6. **Headline** (touched only at end of epoch and convergence):
   `runs/plan5/<run_name>/metrics.json` with R@1 / 5 / 10 / 50 + median
   rank on the **Plan 3 1000-query slice**; `Progress_5_<date>.md`
   compares Plan_3 (R@1 = 0.258, R@10 = 0.533) vs. Plan_5 head-to-head
   on the identical gallery. The §7 "promising R@10 ≥ 0.55" and
   "headline R@1 ≥ 0.28" bars still apply, evaluated on the headline
   slice only at the prescribed checkpoints.

If 1–4 hold but 6's headline doesn't beat Phase A, Plan_5 still
"succeeded as a sanity check" — debate moves to Plan_6 design.
