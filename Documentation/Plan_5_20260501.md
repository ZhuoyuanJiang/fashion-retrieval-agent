# Plan 5 — Single-GPU contrastive fine-tune (design)

## Context

Phase A is closing. Plan_3 baseline (caption-then-retrieve, MiniLM-L6)
landed at R@1 = 0.084. The Phase-A encoder ablation (`Progress_3` / M4)
lifted that to **R@1 = 0.258, R@10 = 0.533** with frozen Marqo
FashionCLIP — that is the bar Phase B has to beat.

This plan is the *design* (not yet execution) for the **first
single-GPU contrastive end-to-end run** — Option 3 from the project
proposal. It was originally numbered Plan_4 in earlier `Plan_overview`
revisions; on 2026-05-01 the conversation forked into two parallel
chats — one for caption-quality ablation, one for contrastive
training — and the caption-quality track took the Plan_4 slot.
Everything from "old Plan_4" downwards now shifts by +1; this plan is
**Plan_5**.

Audio modality is OFF for Plan_5 (text + image only). Audio extension
is deferred to Plan_8+, per the proposal's "text first" ordering.

**Key decisions carried in:**
- VLM: `speechQwen2VL` (Qwen2-VL-7B base + Stage-2 LoRA). Audio path
  unused at this stage.
- Frozen target encoder: Marqo FashionCLIP **image** tower (the same
  encoder that gave Phase A's R@1 = 0.258).
- Eval slice: same 1000-query / ~58 k-target slice as Plan_3 — head-to-
  head comparable.
- **Hardware split (single-GPU sanity check, not multi-GPU yet):**
  - **Training**: single A6000 (48 GB) or equivalent (A100 40/80 GB,
    H100 80 GB). Drops the 3090's 8-bit-AdamW gymnastics and lets us
    bump micro-batch.
  - **Step 0 + eval-only sweeps**: any modest GPU (3090, A6000, A100
    — they're inference-only forwards over a frozen tower; whichever
    is free is fine).
  - Plan_5 deliberately stays **single-GPU per step** to keep the
    debug surface small. Multi-GPU is Plan_7.

## Goal

Prove the contrastive recipe is alive on a **single A6000-class GPU
(48 GB)**, with a clear go/no-go criterion, reusing as much Phase-A
infra as possible. Plan_6 handles the deferred ablations (loss
alternative, embedding-source variants, alternate target tower,
hard-negative mining).

## Step 0 — multi-encoder image-only retrieval ceiling sweep (gate before training)

Before any training run, measure the **image-only retrieval ceiling**
on the same 1000-query / ~58 k-target FACap dress slice we used in
Plan 3. Procedure: for each candidate target encoder below, encode
every gallery image and every eval-query target image with the frozen
encoder; do nearest-neighbor retrieval purely image → image (no VLM,
no captions, no training).

The result: a small table of R@K per encoder. The encoder with the
highest R@10 becomes Plan 5's frozen target tower. The R@10 number
itself becomes the **ceiling** Plan 5's trained query embedding can
never exceed under the frozen-target design.

### Candidate target encoders to sweep

| Encoder | Why include |
|---|---|
| **Marqo FashionCLIP-image** (current pick) | Strongest fashion-domain ceiling expected; Phase A best on the text side |
| **Marqo FashionSigLIP-image** | Same family, different objective; cheap to compare |
| **DINOv2 (large)** | General visual SSL, no fashion bias — tests whether fashion-specialization is actually needed |
| **Original SigLIP-image** (or original OpenAI CLIP-image) | Non-fashion-tuned sanity floor |

### Action thresholds (based on the best encoder's R@10)

| Best ceiling R@10 | Action |
|---|---|
| ≥ 0.70 | Plenty of headroom over Phase A (R@10 = 0.533). Frozen-target is fine; proceed with the design as-is. |
| 0.60 – 0.70 | Some headroom. Frozen still defensible for v1; flag alternate target tower as a Plan_6 priority. |
| < 0.55 | Frozen-target was a bad bet across all candidates. **Pause** before committing to a long training run. Reconsider one of: (a) joint target-tower training (unfreezing), (b) siamese / Qwen2-VL on both sides, (c) caption-side multi-positive supervision (notes Q8). |

Cost: ~5 minutes of compute per encoder. **Runs on any GPU we have
free** — even a 3090 is plenty, since this is inference-only over
frozen towers. No need to occupy the A6000 for this. Should be the
**first command** run when implementation begins, before any training
infra is wired up.

This same operation (with the chosen target encoder) serves as
eval-harness sanity (Verification item 4) — running it through
`online_eval.py` confirms the eval code is correct *and* establishes
the ceiling in one shot.

## Design decisions (committed)

### 1. Loss + target encoder — symmetric InfoNCE, image-side targets

Train the speechQwen2VL query embedding against a **frozen Marqo
FashionCLIP image-tower** target. Symmetric InfoNCE, in-batch negatives
only (no hard-negative mining in v1).

**In other words:** adjust Qwen2-VL's weights so that, given a query
`(ref_image, mod_text)`, it produces a 512-dim vector that lands
close to the 512-dim vector that frozen FashionCLIP's image-half
produces for the correct target dress image. After training, those
two vectors live in the same space — that's what makes
nearest-neighbor retrieval work. (See `notes.progress5_contrastive_learning.md`
Q14 for the full unpacking of "against ... target.")

Why image-side targets, not the FashionCLIP-text route from Phase A:
the encoder ablation showed FashionCLIP-text already ceilings the
system at R@1 ≈ 0.26 — using its text tower as the *target* would
replicate that ceiling. Image-side targets sidestep the captioning
bottleneck entirely.

The alternative (distillation-flavored, MSE against frozen FashionCLIP
*text* embedding) is parked as a Plan_6 ablation — simpler, but with
exactly the cap above.

### 2. Embedding source from Qwen2-VL

Last hidden state at the EOS position
(`outputs.hidden_states[-1][:, -1, :]`, dim 3584) followed by a
**2-layer MLP projection head** (3584 → 1024 → 512, GELU + LayerNorm,
L2-normalized output). Target dim **512** to match Marqo FashionCLIP's
image-tower output (no extra adapter on the target side).

Mean-pool / attention-pool ablations deferred to Plan_6.

### 3. Trainable parameters

- LoRA on LLM decoder: rank 32, alpha 64, target = `q/k/v/o_proj`
  only (drop `gate/up/down_proj` from speechQwen2VL Stage-2's recipe;
  embeddings don't need MLP capacity). Estimate ~30 M trainable.
- Projection head: ~4.2 M trainable, fp32.
- Vision tower frozen. FashionCLIP image tower frozen.
- Plain AdamW (fp32 momentum + variance) on the trainable params.
  No `bitsandbytes` 8-bit optim needed — the A6000's 48 GB is
  plenty of headroom.

VRAM budget on an A6000 (48 GB): 14–15 GB base bf16 + ~150 MB LoRA +
head + ~600 MB FashionCLIP-image + ~280 MB AdamW state (fp32 for
~35 M trainable) + bf16 activations w/ gradient checkpointing on the
LLM decoder → fits comfortably below 40 GB at **micro-batch 8**.
Gradient accumulation 8 → effective batch 64. (If activation
profiling shows headroom, micro-batch 16 / accum 4 keeps the same
effective batch and halves the wall clock per step.)

### 4. Batch construction

- Micro-batch 8 triplets, grad-accum 8 → **effective batch 64**
  (63 in-batch negatives per anchor, symmetric). Bump to micro-16 /
  accum-4 once VRAM is profiled, if it fits.
- No hard-negative mining in v1.
- Target image embeddings pre-computed once for the full ~58 k gallery
  (frozen tower → deterministic) and cached at
  `runs/plan5/target_emb_cache.npy`. Loaded into RAM at train time;
  same cache reused as the eval gallery.
- Sampler check: redraw on duplicate `target_id` within a batch.

### 5. Training loop — custom PyTorch

Custom loop, **not** HF Trainer or `sentence-transformers` Trainer.
Reasons:
- HF Trainer's `compute_loss` assumes one model + per-sample labels.
  Symmetric InfoNCE wraps two encoders and a batch-level loss.
- `sentence-transformers`'s contrastive Trainer assumes a
  `SentenceTransformer` backbone wrapper around the model — wrapping
  Qwen2-VL multimodal in that costs more than it saves.
- Mentor's explicit guidance: don't use HF Trainer blindly here.

A ~250-line custom loop with manual bf16 + grad-checkpoint + grad-accum
is the smallest debuggable surface, and reuses cleanly into Plan_7's
`accelerate launch` for multi-GPU.

### 6. Online eval

Every 500 optimizer steps, run R@K on the same 1000-query / ~58 k-
gallery slice from Plan_3, with image-tower embeddings as the gallery
(not caption embeddings). Reuses `src/baseline/retrieve.py:top_k` and
`src/baseline/eval.py:compute_metrics`. The gallery is built in-memory
(parallel to `CaptionDB` but image-side).

### 7. Sanity-check criteria

| Bar | Threshold | When |
|---|---|---|
| Alive | R@10 ≥ 0.35 | by step 2k |
| Promising | R@10 ≥ 0.55 | by end of 1 epoch |
| Headline win | R@1 ≥ 0.28 | at convergence |

If "alive" fails by step 2k, stop and debug — do not run a full epoch.

## Risks (ranked by severity)

1. **Catastrophic forgetting of image grounding.** LoRA + contrastive
   can collapse the LLM into a text-bag-of-words encoder that ignores
   the reference image. *Mitigation*: image-only probe every 500 steps
   (zero out modification text); R@K must stay above chance.
2. **Temperature blowup / mode collapse.** τ drifts → similarities
   saturate. *Mitigation*: clamp τ ≤ 100, log every step, symmetric
   loss helps.
3. **Train/eval leakage.** The 1000 eval queries are the *last* 1000
   train triplets per Plan_3, and the eval target images may also
   appear in other training triplets. *Mitigation*: separation level
   to be picked from L1/L2/L3 in Resolved Q2 — defaults to L2 unless
   cross-review pushes back; assert no leakage at startup.
4. **OOM at long mod-text.** P99 mod-text + image tokens at micro-batch
   4 can blow the activation budget. *Mitigation*: cap
   `max_seq_length=128` on mod text; grad-checkpoint already on.
5. **Slow convergence.** Estimated ~5–8 hr per epoch on a single
   A6000 at micro-batch 8 / grad-accum 8 (faster compute than 3090
   plus 2× the per-step batch). *Acceptable for v1*; multi-GPU is
   Plan_7.
6. **FashionCLIP image-tower bias.** Targets inherit its visual biases.
   *Acceptable for v1*; alternate target tower (DINOv2 / SigLIP) is
   Plan_6 ablation.

## Resolved questions (decided 2026-05-01 with user)

1. **Target tower — decision deferred to Step 0 results; frozen is
   the implementation default.** Rather than picking frozen vs joint
   training in the abstract, Plan 5 measures the multi-encoder
   image-only ceiling first (Step 0). The empirical ceiling drives
   the call:
   - If best ceiling R@10 ≥ 0.65 → **frozen** (with the winning
     encoder) for v1. The complexity / debug-surface argument and
     the anti-collapse anchor argument both favor frozen when the
     ceiling is comfortable.
   - If best ceiling R@10 ∈ [0.55, 0.65] → frozen but flag joint
     training as a high-priority Plan 6 ablation.
   - If best ceiling R@10 < 0.55 → pause the design; consider one
     of (a) unfrozen / joint target tower in Plan 5 itself,
     (b) siamese (Qwen2-VL on both sides, with stop-gradient /
     momentum-encoder tricks for collapse control), or
     (c) caption-side multi-positive supervision (notes Q8).

   Not deciding now keeps us empirical instead of speculative.

2. **Eval slice — three separation levels on the table; decision
   deferred to cross-review feedback.** The user wants the eval set
   excluded from training but is not yet sure how strictly to draw
   the boundary. Three concrete options:

   | Level | Definition | Cost vs Plan 3 head-to-head |
   |---|---|---|
   | **L1** | Exclude eval *triplets* from training only. Eval target/candidate images may still appear in other training triplets. | Same as Phase A. Trivial. Possible leakage path: model learns embeddings for eval target images via other triplets. |
   | **L2** | Also exclude eval `target_id`s and `candidate_id`s entirely — drop any training triplet whose target or candidate ID is in the eval set. Eval target images are unseen by the model during training. | Stricter than Phase A. Drops ~5–10% of train triplets. Plan 5 winning under stricter conditions makes the result *more* credible. |
   | **L3** | Eval target images do not appear in the gallery during training either — full held-out partition. | Breaks the shared-59k-gallery comparison with Phase A. Rare in retrieval research; usually overkill. |

   *My read* (not committed): L2 is the right tradeoff — stricter
   than Phase A's L1 (so the head-to-head is even more credible) but
   still uses the same gallery so head-to-head numbers stay
   comparable. L3 is overkill. **Asking cross-reviewer (Codex) for a
   second opinion before locking this in.**

   Implementation if L2 wins: an `eval_target_ids` ∪
   `eval_candidate_ids` exclusion set applied as a dataset filter;
   assert at startup that no training triplet leaks any eval ID.

3. **Logging — W&B with meaningful run names.** Decided: Weights &
   Biases for the loss / τ / R@K traces. Run-name convention:
   `plan5_<target_encoder>_<lora_rank>_<eff_batch>_<date>`, e.g.
   `plan5_fclip-img_lora32_eb64_20260502`. Project name: `plan5-cir`.
   Tags: `phase-b`, `single-gpu`, `<frozen-target|joint-target>`,
   `infonce`.
