## Pipeline recap — which encoder, how R@K is computed

The notebook analyzes the original `runs/baseline_v1_speechqwen2vl/`
run, whose configuration is:

- **VLM** (caption generator): `speechqwen2vl`
- **Encoder** (gallery + query embedding): `sentence-transformers/all-MiniLM-L6-v2`
  — 22 M params, 384-dim, the **weakest** anchor from the encoder
  zoo. Plan 3's M4 ablation showed Marqo FashionCLIP beats this 3.07×
  on R@1 (0.084 → 0.258) on the same captions, but caption_analysis
  uses MiniLM-L6 deliberately — see "Why MiniLM" below.

**End-to-end retrieval, per query (1 of 1000 dress triplets):**

```
1. (reference image, modification text)  →  speechqwen2vl
   → "generated_caption" (1 string, ~92 chars median)
2. MiniLM-L6.encode(generated_caption)
   → query_emb  ∈  ℝ^384
3. Gallery built once: MiniLM-L6.encode(every dress caption in FACap)
   → gallery_embs  ∈  ℝ^{59082 × 384}
4. cosine_similarity(query_emb, gallery_embs)
   → 59082 scores
5. Sort descending, find true_target's position  →  rank ∈ [1, 59082] ∪ {None}
```

After the per-query rank is known:

```
R@1  = #{rank == 1 } / 1000  =  84 / 1000  = 0.084
R@10 = #{rank ≤ 10} / 1000  = 240 / 1000  = 0.240
R@50 = #{rank ≤ 50} / 1000  = 384 / 1000  = 0.384
```

Two design properties of this pipeline are worth pinning down because
they're easy to forget later:

1. **Text-only after step 1.** Steps 2–5 never see a pixel. The image
   is funneled into a text bottleneck by the VLM, and the encoder /
   ranker only operate on strings. This is why the Phase A bottleneck
   can be cleanly attributed to either the VLM (caption quality) or
   the encoder (text→embedding mapping) — they're separated by the
   string-shaped interface.

2. **Captions are encoder-independent.** The 1000 generated captions
   are written by the VLM and saved into `qualitative/results.jsonl`
   *before* MiniLM ever sees them. Swapping the encoder (the M4
   ablation, or `replay_with_encoder.py`) re-runs only steps 2–4 — the
   captions don't change. This is what makes caption_analysis's
   conclusions portable across encoder runs.

### Why MiniLM-L6 (the weakest encoder) for this analysis

It's a research-style choice, not "we forgot to swap encoders":

- **Captions are the same** under any encoder, so any conclusion about
  caption quality (length gap, token-overlap distribution,
  per-failure-mode rubric) transfers verbatim to the Marqo run.
- **More failure samples** → easier to characterize. With MiniLM,
  84 / 1000 = rank=1 success and 472 / 1000 = rank≥201 deep miss; with
  Marqo FashionCLIP, success rises to 258 and deep miss falls
  proportionally. The MiniLM stratification gives the rank-stratified
  gallery a richer set of bad cases to look at.
- **Anchor stays the anchor.** The Plan 3 baseline report headline is
  the MiniLM run; analyzing the same artifacts keeps the analytical
  story aligned with the report rather than introducing a separate
  pair of numbers.

If the analysis ever needs to be repeated against the strong
(Marqo FashionCLIP) run, change `RUN_DIR` to
`runs/baseline_v1_speechqwen2vl_marqo-fashionclip` in cell 2 and
re-execute; only Sections 3, 6, 7, 8 will produce different numbers
(rank-stratified bins shift). Sections 4 and 5 (length stats, per-query
gallery captions) are byte-identical.

## Per-section intent — the message each cell is designed to deliver

The table above describes *what* each section does. This section
describes *why* — the question we wanted answered before writing the
cell. Recording this here because section headers alone don't explain
the reasoning, and the reasoning is what's reusable for future
analysis notebooks.

The notebook is a **funnel from broad to specific**:

```
S3-4: aggregate stats        (one number per metric, all 1000 queries)
   ↓
S5-6: visual examples        (~30 queries with full image+caption context)
   ↓
S7-8: quantitative agreement (per-query metrics, aggregated by bucket)
   ↓
S9:   forward-compatible labeling scaffold
```

A reader can stop at any level and have a complete (if shallower)
picture. Below: per-cell goal and what we learned.

### S1 — Imports + paths
**Goal:** make `RUN_DIR` the single point of redirection so reanalyzing
a different encoder run is a one-line change, not a notebook-wide
find/replace.

### S2 — Load data
**Goal:** confirm we're looking at the right run. Print row count +
first-row keys so a misconfigured `RUN_DIR` fails loudly here, not
five cells later.

### S3 — Quick stats (R@K + rank histogram)
**Goal:** situate the reader in the run. The log-x rank histogram
shows the long failure tail at a glance — "most queries are *not* at
rank 1, and the deep-miss tail is fat." This is what the rest of the
notebook is going to characterize.

### S4 — Caption length histograms
**Goal:** surface the **single most striking fact** about the VLM
output before any deeper analysis: it's ~6× shorter than the
ground-truth caption (median 92 vs 554 chars). If you only read one
cell, this is it. It motivates the "VLM under-describes" hypothesis
and Plan 4.5 (VLM swap).

### S5a — Per-query gallery, 10 random (seed=42)
**Goal:** give the reader a *representative* cross-section. Because
472/1000 queries are at rank ≥ 201, a uniformly random sample is
mostly failures — that's intentional. The reader calibrates "what
does typical failure look like" rather than seeing only cherry-picked
successes.

### Why split queries by rank — the core analytical move (frames S5b–S8)

Everything from S5b through S8 stratifies the 1000 queries by
retrieval rank. This is the load-bearing design choice; the cells
below only make sense once you accept this framing.

#### The core question: why bucket by rank at all?

The notebook's central question:

> **Are good captions and bad captions actually different?**

You can only answer this by splitting the 1000 queries into "the
ones that retrieved successfully" and "the ones that didn't," then
comparing what the captions look like in each group.

Without splitting, all you have is "the average generated caption
is 92 characters" — that single number tells you nothing about
where the good-vs-bad difference lies.

#### `rank` is a proxy for "success vs failure"

We don't have ground-truth labels saying "this caption is well-written
or not." But we do have retrieval `rank`:

- `rank = 1` → retrieval succeeded → this caption is **likely**
  high-quality.
- `rank = 500` → retrieval failed → this caption **likely** has a
  problem.

So **rank is a proxy for caption quality.** Bucketing the 1000
queries by rank is equivalent to using retrieval performance to
auto-label captions as "good" or "bad" — without needing human
annotation.

#### Why not just two groups (success vs failure)? Why 5 buckets?

A binary split (rank=1 vs rank>1) shows you a binary difference,
but you don't know whether that difference is continuous or
step-shaped. Five buckets (rank=1 / 2-10 / 11-50 / 51-200 / 201+)
let you see *how* caption properties change as rank changes:

```
Expected: rank gets worse → caption quality gets worse → monotone correlation
Actual:   0.238 → 0.229 → 0.217 → 0.208 → 0.190   ✓ clean monotone decline
```

A monotone decline is much stronger evidence than a binary
difference — it says caption quality and retrieval are *continuously*
correlated, not "rank=1 is the outlier and the other 999 are all
about the same" (a threshold artifact).

#### S5b vs S6 in plain language

- **S5b (1 per bucket):** "Grab one rank=1 example and one rank=500
  example, eyeball the difference — for demo."
- **S6 (3 per bucket):** "To avoid the risk that the one rank=1
  example I happened to pick was a fluke, randomly draw 3 per
  bucket — for statistical robustness."

The "3" in S6 isn't a magic number — it's the minimum sample size
that keeps you from being fooled by a one-off example. Five would
be more robust, but 5 × 5 = 25 large images makes the notebook too
long. 3 is the readability-vs-robustness compromise.

#### The full reasoning chain

```
1. We want to know: does caption quality determine retrieval
   success or failure?
2. But we have no ground-truth "caption quality" label.
3. → Use rank as a proxy: better rank = caption is likely better.
4. → Bucket the 1000 queries by rank into 5 buckets.
5. → Measure caption properties per bucket
   (length, token overlap).
6. → If properties change monotonically with rank, that means
   caption quality really is correlated with retrieval.
7. → ρ = −0.306 + monotone decline = "correlated, but only
   explains part of the variance."
8. → Conclusion: caption quality is one bottleneck, not the
   only one.
```

**Without step 4 (bucketing), steps 5–8 cannot be derived.** That's
what makes the rank split the load-bearing analytical move.

### S5b — 5 hand-picked across the rank spectrum
**Goal:** side-by-side compare rank=1 / 2-10 / 11-50 / 51-500 /
500+ / missing-from-DB, **one** example per bucket (the first found).
Quick read on "are these qualitatively different?" — but only one
sample per bucket, so don't generalize from it. Section 6 fixes that.

### S6 — Rank-stratified gallery, 3 random per bucket (seed=2026)
**Goal:** upgrade S5b from "1 example per bucket" to "3 random
examples per bucket". One example can be cherry-picked (consciously
or not); three random samples are the minimum to ask "is the rank=1
caption *typically* more detailed, or was that one example a fluke?"
This cell is the **visual scaffold** for Section 7's quantitative
claim — Section 7 numbers without Section 6 visuals would be hard to
trust.

### S7 — Caption-vs-target programmatic metrics
**Goal:** quantify what S6 showed visually. The visual gradient
("rank=1 captions look more detailed") gets a number:
**Spearman ρ = −0.306, p = 4e−23**. The `p`-value confirms the trend
is real; the magnitude (~0.3, not ~0.7) confirms it's only weakly
explanatory — most of the rank variance is *not* literal token
overlap. This is the cell that splits the failure-mode story into
two: VLM-side (caption quality) *and* encoder-side (semantic
mapping).

The two scatter plots exist because correlation alone hides
distribution shape. The scatter shows whether the relationship is
monotone, has clusters, or has outlier behavior.

### S8 — Per-bucket caption-agreement summary
**Goal:** one-line-per-bucket numeric summary that maps back to S6's
buckets. S7's Spearman tells you "there's a trend"; S8's table tells
you "the trend looks like this." The clean monotone
(0.238 → 0.229 → 0.217 → 0.208 → 0.190) is what we wanted to see — a
non-monotone result here would have been a flag that token overlap is
the wrong agreement metric.

### S9 — Failure annotation scaffold
**Goal:** don't lock in a hand-labeling tool now, but ensure that
if/when we want to label 100 failures with the Plan_2 rubric, the
file format is decided and the helper exists. Append-only is
deliberate: cells can be re-run safely without overwriting prior
labels. Lowest-priority cell — a forward-compatibility hook, not an
analysis result.

### Why two galleries (S5 and S6) instead of one
S5 is "what does the system do on average" (random + a few specific
ranks). S6 is "what does the system do *conditional on rank*" (3
random examples per rank bucket). Both views are needed because:

- S5 alone would over-represent failures (since failures dominate the
  random sample) — the reader might leave thinking "this system never
  works."
- S6 alone would obscure the base rate — every bucket has 3 examples
  regardless of how big the bucket is, so the reader couldn't tell
  whether rank=1 happens for 10 or 500 queries.

Together they give both the marginal distribution (S5) and the
conditional comparison (S6) without the reader having to compute
either.

## Key findings (from the executed notebook)

### Caption length: VLM massively under-describes

| | median (chars) | mean | range |
|---|---|---|---|
| VLM-generated caption | 92 | 97 | 29 – 218 |
| FACap target caption | 554 | 526 | 252 – 666 |
| len_ratio (gen / target, median) | **0.167** | | |

The VLM-generated caption is **~6× shorter** than the FACap
ground-truth target caption. The encoder side of the pipeline is being
given a substantially compressed signal compared to what FACap's own
captioning produces.

### Caption agreement vs rank correlates, but weakly

Spearman rank-correlation on 1000 ranked queries:

| Pair | ρ | p |
|---|---|---|
| rank vs token_overlap | **−0.306** | 4.0e-23 |
| rank vs len_ratio | −0.216 | 5.5e-12 |

Both negative as expected (better rank ↔ higher overlap / less
under-describing), and both highly significant. But ρ ≈ −0.3 is a
**weak-to-moderate** effect — far from 1.0. This means caption-target
agreement explains only a small fraction of rank variance. Most of what
determines whether a query lands at rank 1 vs rank 500 is **not**
literal token overlap with the ground-truth caption.

### Per-bucket overlap gradient (clean monotone)

Mean token_overlap by rank bucket (from Section 8):

| Bucket | n | mean overlap | median overlap |
|---|---|---|---|
| rank=1 | 84 | 0.238 | 0.235 |
| rank 2-10 | 156 | 0.229 | 0.228 |
| rank 11-50 | 144 | 0.217 | 0.206 |
| rank 51-200 | 144 | 0.208 | 0.204 |
| rank 201+ | 472 | 0.190 | 0.180 |
| missing-from-DB | 0 | — | — |

Clean monotone trend, ~25% relative drop from rank=1 to rank=201+.
Confirms the Spearman direction without imposing a parametric
assumption.

### Even successful queries have low literal overlap

The headline rank=1 bucket sits at **23.8% mean token overlap** —
meaning even the queries that got the answer perfectly right shared
less than a quarter of their tokens with the ground-truth target
caption. The encoder is doing real **semantic** matching, not literal
token matching. This is good news for the embedding-side approach but
weakens any naive "make the VLM output more target-like words"
intervention.

### Gallery is complete: 0 missing-from-DB

`n_unranked = 0 / 1000`. Every triplet's target was successfully
encoded into the DB. The "missing-from-DB" bucket exists in Section 6
purely as defensive scaffolding for future configurations (DB
down-sampling, cross-split eval, cross-category eval — see
`data_exploration/notes_facap_dataset.md`); it is empty for this run.

## Interpretation

**Two independent failure modes are visible in the data.**

1. **VLM under-describes** (length ratio 0.167 — 6× compression). A
   stronger captioner (Plan 4.5: vanilla Qwen2-VL or a better
   instruction prompt) would directly attack this. Worth running.

2. **Encoder ceiling is real even with perfect captions.** Rank=1
   queries only show 23.8% token overlap on average; rank=201+ queries
   show 19.0%. That ~25% relative gap is what caption quality buys you,
   but the *absolute* level (≈ 0.2 even at rank=1) means the
   text-encoder is pattern-matching on a small set of discriminative
   tokens, not on overall caption similarity. Plan 5 (contrastive
   end-to-end) addresses this side: train the embedding so that
   "compressed VLM caption" maps close to "long FACap caption" of the
   same garment.

**These two interventions stack.** A better VLM doesn't subsume
contrastive training, and contrastive training doesn't subsume a
better VLM — Spearman ρ = −0.30 says caption quality matters but is far
from sufficient.

## Cross-conversation Q&A — surfaced understanding

Reading-pass dialogue clarified one point worth recording in the
project record:

- **FACap is a within-category benchmark.** Every triplet's candidate
  and target are the same category (dress / skirt / etc.); no
  cross-category modifications exist in the data. This shapes the
  task scope: the system as built does not attempt "turn this dress
  into a skirt" queries — the dataset doesn't supply training or eval
  signal for that, and the DB only contains dress captions.
  Cross-category retrieval would require a different dataset
  (Fashion200k, CIReVL, LasCo) and is out of Phase A/B scope.

  See `data_exploration/notes_facap_dataset.md` for the full breakdown
  (counts per category, why the missing-from-DB bucket exists,
