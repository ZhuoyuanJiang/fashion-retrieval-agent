## Summary

Before writing Option A (shared backbone + two PEFT LoRA adapters), we surfaced a silent-correctness interaction between PEFT's `set_adapter` API and PyTorch's gradient checkpointing.

Plan_10 §4.3 calls out **one** footgun: PEFT's `set_adapter(name)` flips `requires_grad=False` on the inactive adapter, so without an explicit reset before `backward()`, the inactive adapter never receives `.grad`. Mitigation: the `for n,p in named_parameters(): if "lora_" in n: p.requires_grad = True` reset in the parent `forward`.

A **second** footgun, not in the plan, exists when gradient checkpointing is enabled: `use_reentrant=False` checkpoint **recomputes** the forward during `backward()`, reading `model.active_adapter` at recompute time. Since `active_adapter` is the last value set in the current step (always `"target"` after our query→target sequence), the query branch's recompute applies `lora_target` instead of `lora_query` — silently wrong query-side gradients. Loss still decreases (target branch's gradients are correct), grad-receipt assertion still passes (`.grad` is non-zero), only a side-by-side comparison with Option B would reveal it.

**Three options** to resolve this — see Appendix B for the full trade-offs.

**Recommendation:** Option 1 — disable gradient checkpointing on Option A's V1 backbone. Costs bs=24 → bs=8 (matching Option B's smoke), gains zero silent-gradient risk. Pending the user's call before implementation begins.

## Implementation Log

### 2026-05-12 — Code written

Added `TwoTowerSharedBackbone` to `src/training/two_tower_model.py` following the design in Appendix B Option 1 + Appendix D:

- One Qwen2VL backbone loaded via `_load_qwen2vl_base(..., merge_stage2=True)` (same as `ContrastiveQwen2VL`). Stage-2 LoRA is merged into base weights before any contrastive LoRA is attached.
- PEFT: `get_peft_model(vlm, lora_cfg, adapter_name="query")` then `vlm.add_adapter("target", lora_cfg)` so both adapters share the same backbone weights.
- Manual `for n,p in vlm.named_parameters(): if "lora_" in n: p.requires_grad = True` at init time — both adapters trainable simultaneously (PEFT's add_adapter sets target's requires_grad=False initially).
- **Deliberately did NOT call `vlm.gradient_checkpointing_enable(...)`**. Kept `vlm.enable_input_require_grads()` (needed for PEFT under DDP) and `vlm.config.use_cache = False`.
- Same lm_head 1-output stub as `ContrastiveQwen2VL` (we never use logits).
- Two `_ProjectionHead` instances (fp32) imported from `contrastive_model.py`.
- `forward(cand_images, mod_texts, tgt_images)` calls `set_adapter("query")` → query forward → `set_adapter("target")` → target forward → `requires_grad=True` reset on all `lora_*` params → return `(q_emb, t_emb)`. The reset undoes PEFT's set_adapter-flips-inactive-False footgun (Plan_10 §4.3).
- `encode_query` / `encode_target` `@torch.inference_mode` eval helpers on the unwrapped model.
- `trainable_parameters()` filters by parameter name (`.query.` / `.target.` substring in name) rather than requires_grad, because set_adapter mid-step can flip requires_grad on whichever adapter is inactive. The optimizer holds references to all four groups; gradient flow per step is controlled by `forward()`'s reset.

Wired into `train_plan10.py` with `--arch shared|separate` flag (default `separate` for back-compat with the Option B path that landed in `a10edb7`). W&B run name auto-includes the arch tag (`v1_shared_...` vs `v1_separate_...`).

Launcher `scripts/run_plan10.sh` forwards `--arch` transparently via the existing wildcard catch — added `--arch shared` to the usage block at the top of the file.

```bash
ssh <server-10> 'TS=$(date +%Y%m%d_%H%M%S); RUN_DIR=...; LOG=...; mkdir -p "$RUN_DIR"
  tmux kill-session -t plan10_a 2>/dev/null
```

Three independent layers of protection:
1. **tmux**: training survives SSH disconnects, cron-style terminal closures, network blips
2. **PYTHONUNBUFFERED=1**: stdout streams to the log in real time
3. **tee**: log goes BOTH to file AND to the tmux pane (visible if you attach via `tmux attach -t plan10_a`)

**Root cause:** the checkpoint-save block (`train_plan10.py:567–580`) was written exclusively for Option B's class shape:

```python
for tower_name in ("query_tower", "target_tower"):
    tower = getattr(unwrapped_model, tower_name)
```

`TwoTowerSharedBackbone` has a different attribute shape: one `.vlm` (a PeftModel with two adapters) + two heads (`.head_query`, `.head_target`). When `getattr(model, "query_tower")` ran, it raised `AttributeError`.

**Why this wasn't caught earlier:**
- The smoke was set to `--max-epochs 1`, so its end-of-epoch checkpoint code DID run — but the smoke crashed earlier (the /ssd1 disk-full incident) before reaching it.
- Once we moved to `/ssd2`, the 4th smoke "appeared dead" at step 820 due to log buffering, and we killed it instead of letting it complete.
- Static `py_compile` cannot catch attribute errors on runtime objects.
- The grad-receipt assertion validates training step shape, not checkpoint save shape.

**Fix:** added `if args.arch == "shared": … else: …` branch in the checkpoint-save block:

```python
if args.arch == "shared":
    unwrapped_model.vlm.save_pretrained(str(ckpt_dir / "shared_backbone"))
    torch.save(unwrapped_model.head_query.state_dict(), ckpt_dir / "head_query.pt")
    torch.save(unwrapped_model.head_target.state_dict(), ckpt_dir / "head_target.pt")
else:
    for tower_name in ("query_tower", "target_tower"):
        # ... existing Option B pattern
```

PEFT's `save_pretrained` on a `PeftModel` with two adapters writes both adapters into subdirectories of the destination (`shared_backbone/query/`, `shared_backbone/target/`).

`py_compile` clean. Relaunched in tmux as `plan10_a` with new timestamp.

### Lessons learned

1. **Always check disk space on ALL relevant paths before launching a long training run.** We saw `/ssd1` at 100%, `/` at 96%, NAS home at 101 GB / 100 GB simultaneously. Multiple disks can be tight on shared lab servers; one path being writable doesn't mean the broader IO path (wandb, tmp, HF) will succeed.

2. **HuggingFace auto-caches to `~/.cache/huggingface/` if `HF_HOME` is missing.** Non-interactive SSH does NOT source `.bashrc`. Always re-export `HF_HOME`, `TRANSFORMERS_CACHE`, `HF_DATASETS_CACHE` inside the launcher script — already done in `scripts/run_plan10.sh:24-26`. The lesson generalizes to any new script.

3. **Use `PYTHONUNBUFFERED=1` for any long-running Python job piped via `tee` / `>` / `nohup`.** Without it, the log file goes silent even though the process is alive — leading to false "the run died" diagnoses.

4. **Use `tmux` (or `screen`) for long jobs on remote servers**, NOT bare `nohup ... &` from a one-shot SSH command. tmux gives: (a) a place to attach later for live debugging, (b) survival against SSH session cleanup, (c) explicit "is the session still alive?" via `tmux ls`.

5. **End-to-end smoke tests must include at least one end-of-epoch boundary.** Our smoke killed at step 820 of 1678 before reaching end-of-epoch. The checkpoint-save bug was downstream of training and would only have surfaced at step 1678. Future smokes should hit the end-of-epoch path: either use a tiny dataset with `steps_per_epoch < 50`, or `--max-epochs 1` AND let it complete.

6. **Architecture-conditional code paths need both branches tested.** I added `--arch shared` for model construction + adapter naming, but missed the analogous branching needed for checkpoint save. Any new conditional on architecture class must exercise both branches in smoke.

7. **Process being alive ≠ logs being updated.** Cross-check with: `ps -ef | grep PROCESS`, `nvidia-smi --query-compute-apps`, and the log file. Two-of-three healthy ⇒ "alive + buffered"; only one ⇒ "stuck or dying".

8. **`setproctitle("CIR")` makes `ps`/`pgrep` cleaner for other lab users but harder for self.** Use `pgrep CIR` (not `pgrep train_plan10`) when looking for our own processes.

## Appendix A — Q&A: Why does the recompute use the wrong adapter?

### User question
> 我们不是一次只激活一个 active adapter 吗？然后不是 loss 直接可以一次性训两个 adapter 么？为什么会出错？

两个直觉都对：
1. ✅ 我们确实一次只激活一个 adapter
2. ✅ 一个 loss 确实可以同时训两个 adapter

那为什么会出错？**答案藏在 gradient checkpointing 的一个细节里**。

### 先看：没有 checkpointing 时，怎么训两个 adapter？

```
Step 1: set_adapter("query")       → active_adapter = "query"
Step 2: q_emb = forward(query输入)
        ├─ PEFT 层读 active_adapter="query" → 用 lora_query 算
        ├─ 每一层的中间值（activations）都被 PyTorch **保存**起来
        └─ autograd 图记录：q_emb depends on lora_query (这一层用了 lora_query)

Step 3: set_adapter("target")      → active_adapter = "target"
Step 4: t_emb = forward(target输入)
        ├─ PEFT 层读 active_adapter="target" → 用 lora_target 算
        ├─ 中间值再次被保存
        └─ autograd 图记录：t_emb depends on lora_target

Step 5: loss = f(q_emb, t_emb)
        autograd 图现在长这样：
          loss
          ├── 经 q_emb ─── 经那些保存的 activations ─── lora_query (叶子)
          └── 经 t_emb ─── 经那些保存的 activations ─── lora_target (叶子)

Step 6: backward()
        ├─ 对 lora_query 叶子：用保存的 query 分支 activations → 算出正确梯度 ✅
        └─ 对 lora_target 叶子：用保存的 target 分支 activations → 算出正确梯度 ✅
```

**关键**：activations 在 forward 时已经**保存好了**，每一份 activation 都是用当时正确的 adapter 算出来的。backward 直接用这些保存的值，不再问"现在 active_adapter 是什么"。

所以**没有 checkpointing 时，一个 loss 训两个 adapter 完全没问题**。这跟直觉一致。

### 加上 checkpointing 后，问题在哪？

Gradient checkpointing 的核心动作：**为了省显存，forward 时不保存 activations，backward 时再重新算一遍**。

```
Step 1: set_adapter("query")
Step 2: q_emb = forward(query输入)
        ├─ PEFT 层读 "query" → 用 lora_query 算
        ├─ activations 算出来了**但被丢掉**，只保存输入
        └─ autograd 图记录："这一段以后要重新算 forward 才能拿到 activations"

Step 3: set_adapter("target")      ← active_adapter 变了
Step 4: t_emb = forward(target输入)
        ├─ PEFT 层读 "target" → 用 lora_target 算
        ├─ activations 同样被丢掉
        └─ autograd 图记录："这一段以后要重新算"

Step 5: loss = f(q_emb, t_emb)
        现在 model.active_adapter = "target"（最后一次设的是它）

Step 6: backward() 开始
        ├─ 反向到 target 分支：
        │  需要 activations → 调用 forward 重新算
        │  PEFT 层读 active_adapter = "target" → 用 lora_target ✅ 对的
        │  （因为最后一次 set_adapter 就是 target，恰好运气好）
        │
        └─ 反向到 query 分支：
           需要 activations → 调用 forward 重新算
           PEFT 层读 active_adapter = "target" → 用 lora_target ❌
           （但这里应该用 lora_query！原始 forward 用的是 lora_query！）
```

### 一个生活化的类比

想象做菜：
- 第 1 步：拿出 **A 牌面粉**（`lora_query`），和水混成面糊
- 第 2 步：把面糊做成煎饼（`q_emb`）
- 第 3 步：拿出 **B 牌面粉**（`lora_target`），和水混成面糊
- 第 4 步：把面糊做成华夫饼（`t_emb`）
- 第 5 步：煎饼 + 华夫饼 = 一道最终菜（loss）

**没 checkpointing**：两份面糊我们都留了样本（保存的 activations）。等会儿要"反向推导"，可以直接看样本。

**有 checkpointing**：为了省冰箱空间，我们把面糊扔了，只记着"用 X 牌面粉和水"。回头要重新做。

但问题是——面粉罐里现在装的是 **B 牌**（最后一次拿的是 B）。当我们重做"煎饼面糊"时，配方说"拿面粉和水"，我们抓起面粉罐——里面是 B，于是煎饼面糊**被错做成了 B 牌面糊**。

autograd 图（菜谱）说得对：煎饼的梯度要更新到 A 牌面粉。但用错面糊算出的梯度本身是错的——它告诉我们"怎么改 A 牌面粉才能让 B 牌面糊做出来的煎饼更好吃"——逻辑都是歪的。

### 总结这个 bug 的本质

| | 没 checkpointing | 有 checkpointing |
|---|---|---|
| activations 怎么来 | forward 时保存好的 | backward 时重新算一遍 |
| 算 activations 时 PEFT 用哪个 adapter | forward 时设的（query 分支用 lora_query，target 分支用 lora_target，都对的） | **当前 `model.active_adapter`**（永远是最后一次 set 的那个） |
| autograd 图 | 对的 | 对的（图本身没问题） |
| 梯度计算 | ✅ 正确 | ❌ query 分支用错 adapter 算 activations 算梯度 |

**根本原因**：`active_adapter` 是 model 上的一个可变变量（mutable state），PEFT 层在 forward 时**动态读取**它。Checkpointing 的 recompute 重跑 forward，读到的是当前值，而不是原来 forward 时的值。

### 为什么这个 bug 是 silent

- `loss.item()` 还在下降 — target 分支梯度是对的，整体 loss 还是会优化
- Grad-receipt assertion (Plan_10 §10.1.b) — 通过。它只检查 `.grad` 非零，不检查方向/大小是否正确
- 训练指标 (train loss / dev loss / R@K) — 都会算出看起来合理的数字。Query tower 学到了**东西**，只是不是对的东西
- 唯一信号：跟 Option B 对比 — 如果 Option A 的 query LoRA 学习曲线、最终 R@K 跟 B 明显不同（且更差），可能是这个 bug。但要等 full run 跑完才看得出，10 小时之后

Plan §4.3 的 `requires_grad=True` reset 解决的是相关但**不同**的症状——`set_adapter` 翻转 `requires_grad=False` 导致 `backward` 直接跳过 `.grad` 累加（"没梯度"）。reset 解决"没梯度"，但解决不了"梯度算错"。两个 footgun 同根：`set_adapter` 改动 model 的全局可变状态。

## Appendix B — Three options compared

### Option 1 — Disable gradient checkpointing on Option A

```python
# In TwoTowerSharedBackbone.__init__:
vlm = get_peft_model(vlm, lora_cfg, adapter_name="query")
vlm.add_adapter("target", lora_cfg)
vlm.enable_input_require_grads()  # still needed for PEFT
vlm.config.use_cache = False
# NO gradient_checkpointing_enable() — activations stay saved
```

| | |
|---|---|
| Correctness | Provably fine — activations were saved at forward time with correct adapter; backward doesn't recompute |
| VRAM | ~2× activations during backward. Qwen2VL-7B at bs=8 seq~2k ≈ 10–15 GB activations. Total per-GPU ~25–30 GB / 49 GB on A6000 → fits |
| Batch size cap | bs=8–12 per GPU. Plan §6.1's bs=24 not reachable without checkpointing |
| Compute speed | Faster than checkpointed (no recompute) but smaller batch → fewer negatives per step |
| Code complexity | Minimal change vs Plan §Appendix B walkthrough |
| Implementation risk | Lowest |

### Option 2 — Keep gradient checkpointing, trust PEFT / empirical verification

```python
# Same init as Option B: checkpoint enabled
vlm.gradient_checkpointing_enable(gradient_checkpointing_kwargs={"use_reentrant": False})
```

| | |
|---|---|
| Correctness | UNCERTAIN — depends on whether installed PEFT version guards this specific scenario. Recent PEFT issues mention multi-adapter + checkpoint as a known interaction; no guaranteed fix that I can verify |
| VRAM | Allows Plan §6.1 bs=24 (per-GPU). 8×24 = 192 effective negatives |
| Detection signal | Diff-based comparison against Option B's `lora_query` learning curve / final R@K — but only surfaces after a multi-hour run |
| Implementation risk | Moderate — if wrong, lose a full run's worth of compute |

### Option 3 — Custom checkpointed forward that re-sets `active_adapter` on recompute

```python
def forward_query(self, images, texts):
    def _checkpointed_segment(*args):
        self.backbone.set_adapter("query")  # ← re-set on every (re)call
        return self._raw_forward(*args)
    return torch.utils.checkpoint.checkpoint(
        _checkpointed_segment, *inputs, use_reentrant=False
    )
```

Wrap each tower's forward in a Python function whose **first instruction** is `set_adapter(name)`. Both the original forward AND the recompute then start by re-establishing the correct adapter.

| | |
|---|---|
| Correctness | Not "provably correct" by itself. `set_adapter()` inside the recompute fn fixes the active-adapter problem, but `set_adapter()` *also* mutates `requires_grad` on the inactive adapter (footgun #1). Mid-backward, that means `lora_query.requires_grad` can be flipped to False *after* the query branch has already passed through gradient accumulation — silent skip on `.grad` accumulation. To be fully correct, the fn must also re-reset `requires_grad=True` on both adapters after every `set_adapter` call |
| VRAM | Same as Option 2 — full checkpointing benefits |
| Code complexity | ~40+ lines once the `requires_grad` thrashing is also managed; closure-captured tensors must thread correctly through `torch.utils.checkpoint.checkpoint` |
| Implementation risk | Moderate-to-high — the obvious version is incomplete; the complete version is fiddly and bugs here would also be silent |

### Option 4 — Use PyTorch's `checkpoint(context_fn=...)` hook (cleanest correct approach)

PyTorch 2.1+'s non-reentrant checkpoint accepts a `context_fn` argument that returns a pair `(forward_ctx, recompute_ctx)` of context managers. The forward context runs during the original forward; the recompute context runs during backward's recompute. This is the official extension point for "snapshot some mutable state at forward, restore it at recompute."

```python
def make_adapter_context(name: str):
    def context_fn():
        forward_ctx = nullcontext()
        recompute_ctx = _SetAdapterAndRestore(self.backbone, name)
        return forward_ctx, recompute_ctx
    return context_fn

# In the tower forward:
return torch.utils.checkpoint.checkpoint(
    self._raw_forward, *inputs,
    use_reentrant=False,
    context_fn=make_adapter_context("query"),
)
```

| | |
|---|---|
| Correctness | Provably correct, using PyTorch's official mechanism — no need to restructure the forward to include `set_adapter()` inside the checkpointed segment |
| VRAM | Same as Option 2 — full checkpointing benefits, enables bs=24 per Plan §6.1 |
| Code complexity | Cleaner than Option 3 (the forward stays natural; only the checkpoint *wrapping* changes). However, HuggingFace transformers' `gradient_checkpointing_enable()` uses its own checkpoint wrapper internally (`model._gradient_checkpointing_func`) — plumbing `context_fn` through to that path requires either overriding `_gradient_checkpointing_func` or monkey-patching `torch.utils.checkpoint.checkpoint` |
| Implementation risk | Low conceptually, but the integration path with PEFT + Qwen2VL + accelerate's specific versions needs empirical verification before a real run |
| When to use | V1.1 optimization, AFTER Option A V1 (no-checkpoint) has proved the architecture works AND batch size is identified as a real bottleneck |
