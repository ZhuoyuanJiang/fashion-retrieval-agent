What you need to understand and master from Nima’s teaching
1) Structured multi-output generation
You do not need to panic about “two outputs.” Nima’s point was that bounding box and intent can be emitted together in a structured schema. The fastest practical test is probably:

{
  "bbox": [x1, y1, x2, y2],
  "intent": "find_cheaper"
}
with the intent constrained to a fixed set of labels. This lets you test the idea without immediately changing the architecture.

2) Zero-shot / few-shot prompting before expensive training
A smart first experiment is:

train localization,
then prompt the model to also emit intent,
then add a few in-context examples if zero-shot is unstable.
The reason this may work is that the intent is often explicitly stated in the user’s query.

3) Audio is the “delta”; image is the base
This was the biggest conceptual lesson of the meeting. The spoken request does not fully specify the desired product. It only specifies the change relative to the reference item. The reference image contains the base clothing semantics; the audio describes the modification. That is exactly why this problem is a good use case for an audio+vision model.

4) The safest strong baseline is caption-generation + text retrieval
This is the most realistic “good resume but still feasible” version:
reference image + spoken request -> VLM-generated modified caption -> text embedding -> retrieve best catalog item
This is much stronger than a basic API wrapper, but much safer than immediately training a new multimodal embedding model end-to-end.

5) Contrastive learning is the stronger extension
The more advanced version is:

query side = reference image + audio modification,
target side = target product image,
train positive pairs close and negative pairs far,
retrieve by nearest neighbor in embedding space.
This is more research-worthy, but also more fragile.

6) Why large batch size matters
Contrastive learning benefits from many positive/negative comparisons in each batch. If the batch is too small, the model gets fewer meaningful negative examples and the embedding space may separate items poorly. That is why Nima specifically brought up batch size and positive/negative construction.

7) Why Hugging Face code changes may be necessary
Your current setup is optimized as a language model with next-token loss. Retrieval learning needs a different objective. Nima’s advice was to treat the model as a feature extractor, extract a final representation, and optimize that representation with contrastive loss rather than standard next-token prediction.

Proposed Method Section
Here is a method section you can directly adapt into your proposal.

Method
We formulate the project as audio-conditioned composed fashion retrieval. Given a reference garment image (I_r) and a spoken modification request (a), the system retrieves from a catalog the target garment (I_t) that best matches the visual content of (I_r) after applying the requested modification. This formulation follows the composed image retrieval setting used by FACap and FashionIQ, except that we replace the modification text with speech so the query is naturally aligned with our audio-extended VLM. (arXiv)

Data
We will use FACap as the main training corpus because it provides 227,680 fashion-domain composed retrieval triplets of the form <reference image, modification text, target image>, as well as image-caption pairs. FACap is constructed from Fashion200k and DeepFashion-MultiModal, which provide large-scale fashion images, product descriptions, and textual annotations. We will use FashionIQ as the downstream benchmark for fine-tuning and evaluation, following prior CIR work. FashionIQ contains 18,000 training triplets and 6,016 validation triplets across dresses, shirts, and tops/tees. (arXiv)
Because FACap and FashionIQ are text-based rather than audio-based, we will convert the modification texts into synthetic speech using a text-to-speech pipeline. For each modification request, we will generate multiple audio variants with different speakers, speaking rates, and light background noise. This lets us train the audio projector on raw speech while preserving the original CIR supervision. To reduce train-test mismatch, we will also record a small real-speech evaluation subset with spontaneous spoken shopping requests from multiple speakers.

Main model: caption-generation retrieval baseline
Our primary model will fine-tune the current audio-extended VLM to generate a target-oriented description from the combination of a reference image and a spoken modification request. Concretely, the model input is (reference image, spoken query), and the output is a short textual description of the item we want to retrieve. The supervision target will be the target-image caption from FACap or a normalized catalog description derived from the target item.
This design follows Nima’s core intuition: the speech does not describe the whole desired item, but only the delta relative to the reference image. The reference image provides the base garment identity, while the speech specifies what should be changed. The model therefore learns to combine the visual base and the spoken modification into a single retrieval-ready description.
After caption generation, retrieval is performed in a separate catalog index. We will assign each catalog image a caption, encode all captions offline using a frozen text embedding model, and store them in a vector index. At inference time, we will encode the generated target description in the same embedding space and retrieve the top-(K) nearest items by cosine similarity. This is the safest implementation path because it uses your existing audio-VLM as the multimodal reasoning front end while keeping the retrieval engine simple and stable. Nima’s text-based retrieval idea is also consistent with CIReVL, which uses captioning plus language recomposition for composed retrieval. (OpenReview)

Optional reranking stage
To improve precision, we will add a reranking module on the top-(K) retrieved candidates. For each candidate image, we will feed (reference image, spoken query, candidate image) to the VLM and ask it to score whether the candidate satisfies the requested modification. This second-pass validation is cheaper than full end-to-end retrieval training, but it gives the system a way to recover from noisy first-stage text retrieval.

Research extension: direct shared-embedding retrieval
As a second-stage extension, we will replace the caption-generation retrieval pipeline with a direct retrieval model. In this version, the query branch encodes (reference image, spoken query) into a dense retrieval embedding, and the candidate branch encodes the target image into the same embedding space. We will train the model with an in-batch contrastive loss so that matched triplets have high cosine similarity and mismatched items have low similarity. This extension is directly inspired by the CIR training paradigm used in FashionBLIP-2 and by Nima’s suggestion to learn an embedding space for retrieval. (arXiv)
In the actual implementation, I would make one technical adjustment to Nima’s quick verbal sketch: instead of using raw vocabulary logits as the retrieval representation, I would use a pooled hidden representation or a small learned projection head on top of the model’s final hidden states. That keeps the spirit of his idea, but makes the retrieval formulation technically cleaner.

Training schedule
We will train in two stages.
First, we will train on FACap with synthetic speech to adapt the audio-VLM to the fashion CIR domain.
Second, we will fine-tune on FashionIQ using the same audio-converted setup, so the model adapts to human-written relative modifications from a standard benchmark. FACap’s own training results suggest that large-scale fashion-domain CIR pretraining improves downstream retrieval, and that contrastive CIR training is the right core learning signal; the paper also shows that auxiliary text-retrieval objectives can help in some settings but are not guaranteed to help after downstream fine-tuning, so we will keep auxiliary objectives optional rather than central. (arXiv)
Evaluation
We will evaluate retrieval quality using Recall@K on FashionIQ, following standard CIR evaluation. If accessible from the FACap release, we will also evaluate on enhFashionIQ, the refined validation set introduced by the FACap paper, because it contains more detailed annotations and is better suited for fine-grained retrieval analysis. In addition, we will run a small real-speech robustness study using recorded spoken queries to compare performance under synthetic versus natural audio input. (arXiv)

Demo system
The final demo will present a catalog-style interface. A user selects or uploads a reference garment image, speaks a modification such as “make it black,” “same style but longer,” or “find a more formal version,” and the system returns the top retrieved fashion items. The first demo should focus on style, color, sleeve, fit, and formality changes, because these are naturally aligned with FACap/FashionIQ. Price-based queries such as “cheaper” can be added later only if the demo catalog has explicit price metadata.
