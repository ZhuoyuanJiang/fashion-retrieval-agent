## 4. fetch_artifacts.sh — the reproducibility entry point

New script: **`scripts/fetch_artifacts.sh`**. Two modes:

**Default (`--public`, no flag):** for outsiders who clone the repo.
1. `huggingface_hub.snapshot_download` of `DanJZY/audio-composed-fashion-item-retriever`
   (audio + text checkpoints) into `runs/hf_models/`.
2. Snapshot the base model and LoRA into `$HF_HOME`.
3. `bash scripts/setup_datasets.sh` to clone the third-party annotation repos.
4. Print an explicit checklist of what's still needed (FACap images not in
   the script; demo preset cache shipped with the repo via `demo_assets/`).

**Owner mode (`--from-drive`):** for restoring on a fresh machine if the
training server dies. `rclone copy` everything from the Drive backup folder;
untar `plan14_audio.tar` / `facap-images.tar` / `caption_db.tar` /
`baselines.tar` into their target dirs.

Design points:

- **Repo-local default paths.** Defaults to `$REPO_ROOT/_artifacts/` so a
  fresh server with no `/ssd1` or `/tmp3` just works; the original training
  server's layout (`/ssd1/zhuoyuan/...`, `/tmp3/zhuoyuan/...`) is the "dev
  machine example" in the comment header, reproducible by exporting
  `RUNS_LOCAL` / `AUDIO_LOCAL` / `FACAP_LOCAL` env vars.
- **`/ssd1/` symlinks only created if `/ssd1` already exists.** The script
  prints a clear "edit AUDIO_2T_CKPT in src/demo/config.py or symlink it
  here" note on machines that lack the original SSD layout, rather than
  silently creating phantom top-level directories.
- **Dangling `runs/` symlink handled.** A fresh clone gets a `runs/` symlink
  pointing to the training server's `/ssd1` path that doesn't exist on the
  new machine; the script detects and replaces it with a real directory
  (public mode) or relinks it to `$RUNS_LOCAL` (`--from-drive`).
- **`_artifacts/` in `.gitignore`.** Default landing zone shouldn't pollute
  git.

Open follow-up: a `--with-images` flag that wraps facap-repo's own
`download_datasets.py` to pull the 59 K FACap images from upstream HF mirrors
(Fashion200k, DeepFashion-MultiModal) — deferred this session.

## 5. demo_assets/ relocation (out of the symlink, into the repo)

The demo at `src/demo/config.py` pointed `PRESET_CACHE_JSON` /
`PRESET_AUDIO_DIR` / `PRESET_THUMBS_DIR` at `runs/demo/<...>` — a path that
resolves through the dev-machine `runs/` symlink to `/ssd1/.../runs/demo/`,
**outside the repo**. Consequence: the 2.4 MB demo cache that the v0.1 cached
demo absolutely needs (`preset_cache.json` + 8 TTS `.wav` clips + a small
`survey.jsonl`) was not committable to git.

Decision: copy the **safe, project-owned** 2.4 MB into a new repo-local
directory **`demo_assets/`** and update `src/demo/config.py` to point there:

```
fashion-retrieval-agent/
├── demo_assets/
│   ├── preset_cache.json      (80 KB) — 8 preset queries + cached top-K per pipeline
│   ├── preset_audio/          (2.3 MB, 8 wav clips) — our own TTS synthesis
│   └── survey.jsonl           (8 KB) — minor
└── src/demo/config.py         — paths now point at demo_assets/
```

Rationale for going to git directly (not HF dataset): HF Hub datasets connote
"complete, training-worthy data"; our demo cache is small demo trinkets that
ship with the demo code. GitHub side-by-side with the code is the right
semantic fit.

**Per-file fallback to the original `runs/demo/` location.** `src/demo/config.py`
doesn't hard-code the `demo_assets/` paths — a small `_demo_path(rel)` helper
prefers `demo_assets/<rel>` but falls back to `runs/demo/<rel>` if the primary
is missing. This way the original developer's dev server doesn't need to
duplicate the 16 MB `preset_thumbs/` into `/home`: the thumbs stay on `/ssd1`
(reachable via the `runs/` symlink), while the small 2.4 MB cache + audio
ship from `demo_assets/`. A stderr warning fires if a file is missing from
both locations.

Crucially **excluded** from git (16 MB legal gray area): `preset_thumbs/` —
1085 thumbnails derived from FACap-source images. We don't redistribute
those; users regenerate them locally (see §6). `.gitignore` carries a
`demo_assets/preset_thumbs/` exception so git tracks the cache + audio but
not user-generated thumbs.

## 6. make_demo_thumbs.py — user-side thumbnail generation

The 16 MB demo thumbnail set is derived from FACap images (Fashion200k /
DeepFashion-MultiModal source). FACap's upstream license is unclear — strict
reading says we shouldn't repackage and redistribute even thumbnails.
Practical solution: **users generate thumbnails on their own machine from the
FACap images they fetched themselves**. No redistribution involved; it's
purely local PIL `Image.thumbnail()` on their data.

New script: **`scripts/make_demo_thumbs.py`**.

What it does:
1. Read `demo_assets/preset_cache.json`, collect every unique item ID
   (~1085 across 8 presets × 4 pipelines + references).
2. For each ID, locate the matching `<id>.jpeg` in the user's local
   `facap-images/` directory (`rglob` fallback for nested subdirs).
3. PIL `thumbnail()` resize to ~300 px on the longer side, save to
   `demo_assets/preset_thumbs/<id>.jpeg`.

Idempotent (skips already-generated thumbs), reports missing IDs at the end.
~75 lines of Python with argparse + Pillow.

Expected user flow:
```
git clone ...
bash scripts/fetch_artifacts.sh --with-images   # model + FACap originals
python scripts/make_demo_thumbs.py              # ~1 minute, fills demo_assets/preset_thumbs/
bash scripts/run_demo.sh                        # demo with full visuals
```

## 7. Decisions and their rationale

A few small judgment calls drove the shape of this phase:

1. **Personal Gmail Drive over Yale Workspace** — durability after
   graduation. Yale Workspace accounts get deprovisioned; personal Gmail
   doesn't.
2. **Don't redistribute FACap-derived thumbnails** — upstream license is
   unclear (already noted in README §Licenses). Use the
   user-generates-locally pattern instead.
3. **Demo cache in git, not HF dataset** — semantic fit; HF datasets imply
   training-worthy data, this is demo trinkets.
4. **Don't publish Plan-6 to HF** — it's a known-failed first attempt
   (R@10 0.402); putting it next to the headline 0.624 dilutes the message.
   Preserved in Drive backup; easy to add later if the demo goes live.
5. **Repo name without `qwen2vl` suffix** — clean, simple, professional;
   Qwen2-VL surfaced via base_model metadata + tags. Easier on a resume.
6. **`research-only` license** — FACap upstream license unclear; safer to
   constrain commercial use until that's resolved.
7. **Repo-local default paths in fetch_artifacts.sh** — works on any fresh
   machine; original training-server layout is the "dev machine example"
   override.
8. **Don't auto-publish 16 MB thumbnails** — go halfway (publish 2.4 MB
   cache + audio so demo loads; thumbnails optional, user-generated). The
   "publish 2.4 MB but not thumbnails" middle ground is awkward (demo loads
   but shows ID strings instead of images), so the make_demo_thumbs.py
   bridge is what closes the visual loop.
