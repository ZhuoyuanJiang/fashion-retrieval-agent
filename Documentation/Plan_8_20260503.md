**Date:** 2026-05-03

## 1. Context

We have two trained pipelines ready to demonstrate (Phase A caption-based; Plan-6 contrastive embedding) and a future audio path that has no model yet. The need is a simple browser interface so reviewers (mentors, advisors, eventual app users) can try the system: input a candidate fashion image plus a modification (text now, audio later) and see what each pipeline retrieves. Building this MVP is the deliverable that lifts the project out of "github-only code" into something we can show.

**MVP requirements:**

- Same UI, three backends, each clearly labelled.
- Accept image + text (mandatory) and image + audio (Whisper ASR transcribes; same text feeds Pipelines 1 & 2).
- Run all pipelines side-by-side on the same query, showing top-K retrieved images, per-stage latency, and (for curated examples) the rank of the true target.
- Reachable from a personal laptop with minimal deployment work.
- No new model training.

## 2. Demo readiness stages (v0.1 → v0.3)

The MVP ships in three stages. Each stage is itself demoable; later stages flip pieces from "cached / mocked" to "live" without changing the UI.

| Stage | What's live | What's cached / mocked | Where it runs |
|-------|-------------|------------------------|---------------|
| **v0.1 — Scripted demo** | UI flow, mic widget records audio, three-column layout, presets, P3 placeholder | All retrieval results read from a precomputed JSON; **mock ASR** returns the preset's predefined prompt regardless of audio content; latencies are baked into the JSON | Personal laptop. No GPU needed. |
| **v0.2 — Live Pipeline 2** | UI; Whisper ASR (real); Pipeline 2 inference on preset & free-form input | Pipeline 1 results still cached (only runs on presets); P3 still placeholder | Any 24 GB+ GPU host (3090 / 4090 / A6000). |
| **v0.3 — Full live** | Everything except P3 | Only P3 still placeholder | A6000-class host (≥49 GB) for both VLMs concurrent. |

**Glossary:**

- **Cached**: result was computed offline once and stored in JSON. The app does a JSON lookup at request time — no model loaded, no GPU needed at demo time. **Real result, just precomputed.**
- **Live**: the model is resident in GPU memory; the app actually runs inference when the user clicks Run. Real-time, input-dependent, costs GPU.
- **Mocked**: result is hard-coded regardless of user input. Used in v0.1 *only* for ASR (returns the preset's predefined text regardless of what the user said). Not used for retrieval results — those are real (precomputed).

The v0.1 scripted mode is honest as long as we don't quote its latencies as "real inference time" — they're recorded.

## 3. Three pipelines (same UI, three backends)

1. **Caption-based** (Phase A, R@10 = 0.533) — VLM `(image, mod_text) → caption`, then text encoder → cosine vs caption DB.
2. **Contrastive embedding** (Plan-6, headline R@10 = 0.402) — fine-tuned Qwen2-VL `(image, mod_text) → 512-d query embedding`, then cosine vs FashionCLIP target cache.
3. **Native audio (future, placeholder column)** — audio → VLM embedding directly, *skipping ASR*. The whole point of this pipeline is that prosody and non-lexical cues are preserved for retrieval. Model not yet trained; the UI shows a placeholder column with an architecture sketch and a "coming soon" caveat. When the native-audio model lands, this column flips to live with no other UI change.

**Audio is the user's input modality, not a separate pipeline.** A shared Whisper ASR front-end (see §9) transcribes audio → text once; the same text feeds Pipelines 1 & 2 (which are inherently text-based). Pipeline 3 is the path that *skips* ASR — that is what reviewers should anchor on, not "Pipeline 3 = Whisper".

Common contract returned by every pipeline:

```python

    target_ids: list[str]          # top-K gallery ids
    scores: list[float]            # cosine similarity
    image_paths: list[Path]        # resolved /ssd1/.../<id>.jpeg paths

    intermediate: dict[str, str]   # caption text (P1); empty for P2; transcript shown separately at the audio frontend
```

Single new dependency (`pip install gradio`). Native widgets cover every need: `gr.Image` upload, `gr.Audio(source="microphone")`, `gr.Gallery` for top-K thumbnails, `gr.Examples` for one-click presets, `share=True` for a 72-hour public tunnel. ## 6. Pipeline 1 strategy: cached on presets by default, live behind a flag

Pipeline 1's caption step is expensive (~2 s VLM forward pass). It's also deterministic at temperature 0 — the same `(image, mod_text)` always produces the same caption. So precomputing on the preset set gives identical results to running live, just instantly.

| Mode | Pros | Cons | VRAM at request time |
|------|------|------|----------------------|
| **Cached** P1 on presets (default; v0.1 + v0.2) | Instant response. Zero VRAM contention. Works on any GPU class — even no GPU in v0.1. | P1 only covers the presets we precomputed; on free-form uploads its column shows "cached-preset mode". | 0 GB |
| **Live** P1 on any input (v0.3) | Runs on free-form uploads — "fair fight" comparison live. | Concurrent requests serialise. | ~16 GB just for P1 — but combined with live P2 (~16 GB) + Whisper + gallery cache, total reaches ~33 GB → needs ≥49 GB GPU host (A6000-class). |

**Decision:** ship cached as default in v0.1/v0.2; v0.3 flips the `LIVE_PIPELINE_1 = True` config flag. One-line change in `src/demo/config.py`.

### Implementation

- Reuse `make_captioner('speechqwen2vl', image_cache_root)` from `src/baseline/vlm_caption.py:236`.
- Reuse `TextEncoder(model_name='Marqo/marqo-fashionCLIP')` from `src/baseline/text_encoder.py:20-40`.
- Reuse `CaptionDB.load(db_dir)` and `top_k(q, db, k)` from `src/baseline/retrieve.py:22-48`.
- `precompute_presets.py`: load all three components once, run each preset, save `runs/demo/preset_cache.json` keyed by preset id.
- App in cached mode reads this JSON and never instantiates the captioner.

## 7. Pipeline 2 strategy: live from v0.2 onwards (cached in v0.1)

This is the headline result; reviewers must see real inference from v0.2 onwards. In v0.1 it's cached alongside P1 (same precompute script).

### Implementation

- Add `from_checkpoint(ckpt_dir, device_map)` on a wrapper class in `contrastive_pipeline.py`. **Do not modify training code.** Mirror the load pattern at `src/training/train_plan5.py:324-338`:

  ```python
  model = ContrastiveQwen2VL(d_target=512, device_map=device_map)
  set_peft_model_state_dict(model.vlm, load_peft_weights(ckpt_dir / "vlm_lora"))
  model.proj.load_state_dict(torch.load(ckpt_dir / "proj_head.pt", map_location=device_map))
  model.eval()
  ```

- Load gallery cache via `src/training/target_cache.py:load_target_cache()` → `(emb (N,512), gallery_ids, dim)`.
- Move target embeddings to GPU once at startup; per request: `(target_emb @ q.T).topk(k)`.
- Best checkpoint: `runs/plan5/run_bs64_8xA6000_plan6_20260503_011214/ckpt_epoch16/` (step 1664, headline R@10 = 0.402). Configurable via `config.py`.
- EOS pooling already correct in `src/training/contrastive_model.py:174-176` — verified.

## 8. Pipeline 3 strategy: placeholder for native-audio path

Pipeline 3 is the future direction (audio → VLM directly, no ASR). The model is not yet trained, so the UI shows a placeholder column with an architecture sketch and the caveat *"Model not yet trained — coming in a future plan."*

When the native-audio model lands, the column flips from placeholder to live with no other UI change. The audio modality is already wired up via the shared ASR front-end (§9), so the user input experience is forward-compatible.

## 9. Audio input via shared ASR front-end

Audio is one of two ways to provide the modification (the other being typed text). Either way, **Pipelines 1 & 2 are inherently text-based**, so audio is normalised to text before either runs.

- UI: `gr.Audio(source="microphone")` → 16 kHz waveform → `audio_frontend.transcribe(waveform) → str`.
- v0.1 implementation: **mocked** — `audio_frontend.transcribe()` ignores the waveform and returns the active preset's predefined `mock_transcript`. (Useful for screen-recordings where we want a known transcript.)
- v0.2 implementation: **live** — Whisper-base via `transformers.pipeline("automatic-speech-recognition", model="openai/whisper-base")` (~140 MB, no extra install — `transformers` already pinned).
- The transcript is shown in an editable textbox so the user can correct it before submitting.
- The same text feeds Pipelines 1 and 2.
- This ASR step is **explicitly not Pipeline 3**. Pipeline 3 is the path that *skips* ASR; the demo reserves that column for the future native-audio model.

## 11. Demo content

### Curated presets (`src/demo/presets.py`)

8 examples drawn from the 1000-query headline split. Selection criterion (in the file header):

- 5 cases where Pipeline 2 ranks the true target in top-5 (the win).
- 2 deliberate failure cases where the true target is outside top-50 (honesty about ceiling).
- 1 case where Pipeline 1 ranks higher than Pipeline 2 (shows the trade-off — caption sometimes wins).

Each entry: `{preset_id, candidate_image_path, modification_text, true_target_id, mock_transcript, notes}`. The `mock_transcript` is what v0.1's mocked ASR returns when this preset is active and the user records audio.

### Free-form mode

Image upload + text/mic. Behaviour by stage:

- **v0.1** — disabled or shows *"Free-form upload requires v0.2+. Click a preset to compare."*
- **v0.2** — Pipeline 2 + Whisper run live; Pipeline 1 column shows *"cached-preset mode; click a preset to compare P1."*
- **v0.3** — All pipelines run live (P3 still placeholder until native-audio model lands).

## 17. Risks / trade-offs

1. **VRAM tightrope if `LIVE_PIPELINE_1 = True`.** Two VLMs on one ≥49 GB GPU (~33 GB total) can OOM on long mod-text inputs. Mitigation: cached default through v0.2; live mode (v0.3) documented as advanced.
2. **Curated presets can over-claim.** Cherry-picked wins paint rosier than headline R@10 = 0.402. Mitigation: include 2 deliberate failures and 1 P1>P2 case; document the selection criterion in `presets.py`.
3. **v0.1 latencies are recorded, not real.** The cached JSON contains the latencies that we observed during the offline precompute run. If we quote them as "live inference time" we mislead reviewers. Mitigation: in v0.1 mode, label the latency line as *"recorded during precompute"*; in v0.2+ it switches to *"this run"*.
4. **Gallery image path coupled to one GPU host.** `/ssd1/zhuoyuan/facap-images/` lives on a specific A6000 host's local SSD; for v0.2+ on a different host, rsync the gallery first. v0.1 only needs the preset thumbnails (~8 images) bundled in `runs/demo/`. Document in `scripts/run_demo.sh` header.
5. **Gradio share-URL expiry (72 h).** For a scheduled meeting, generate the link <1 h before; have stage 2 (SSH tunnel) as fallback.
