# Progress 7 — Plan-7: Dev Loss Logging + Cosine LR Schedule

**Date:** 2026-05-03

## 1. Context and Motivation

Plan-6 (best: headline R@10 = 0.402, server 10 epoch 16) left two diagnosed gaps:

1. **No dev loss** — only R@K was logged at eval. R@K is noisy (oscillates ±0.02
   epoch-to-epoch); a fixed-set InfoNCE loss is a cleaner overfitting signal. Was
   missing throughout Plan-5 and Plan-6.

2. **Constant LR** — Plan-6 headline R@10 oscillated rather than converging smoothly
   in the final epochs. Cosine decay reduces LR toward the end, which may allow
   finer convergence instead of oscillating around the plateau.

Hard negative mining is explicitly deferred (noted in Plan_7 §5).

## 2. Implementation

### `src/training/online_eval.py` — new `run_dev_loss`

Computes multi-positive InfoNCE on the full 500-item dev set at every eval step.

**Design: encode in batches, compute loss in one shot**

Two separate steps with different constraints:

- *Encoding* (VLM forward) is VRAM-heavy — 500 queries at once would OOM. Encoded
  in batches of 32, embeddings accumulated in CPU memory, then concatenated into a
  `(500, 512)` tensor.

- *Loss* (InfoNCE) needs a large N×N similarity matrix to be meaningful. If loss
  were computed per-batch of 32, each anchor would see only 31 negatives — too few,
  too noisy. Instead all 500 embeddings are moved to device together and loss is
  computed once over the full `(500, 500)` logit matrix. This makes the value
  comparable across epochs.

**Why `gather=False`**

Training uses `gather=True` because each of the 8 GPUs holds 1/8 of the batch and
needs cross-GPU all_gather to assemble 512 effective negatives. `run_dev_loss` runs
only on rank 0 (main process) with all 500 queries already in one process — there is
nothing to gather. Passing `gather=True` here would trigger `dist.all_gather` on
rank 0 while the other 7 ranks are idle, causing a deadlock.

```python
@torch.inference_mode()
def run_dev_loss(
    model, loss_fn, dev_items, gallery_lookup, tid_to_idx,
    base_ds, device, batch_size=32,
) -> float:
```

### `src/training/train_plan5.py`

**New CLI args:**

```
--lr-schedule   {none, cosine}   default: cosine
```

`none` preserves Plan-6 constant-LR behavior exactly.

**Cosine scheduler:**

Created after `accelerator.prepare` (so `len(train_loader)` is final):

```python
_total_steps = len(train_loader) * args.max_epochs
scheduler = CosineAnnealingLR(optimizer, T_max=_total_steps, eta_min=0)
```

Stepped after every `optimizer.step()`. On resume (`--resume-from`), fast-forwarded
to the correct step before training begins:

```python
resume_steps = args.start_epoch * len(train_loader)
for _ in range(resume_steps):
    scheduler.step()
```

**Dev loss logging:**

At every eval trigger, `run_dev_loss` is called before `run_dev_probe`. Result
logged as `dev/loss` in W&B and `metrics.json`, and printed alongside R@K:

```
[DEV @step 52]
  dev/loss=4.2137
  ...R@K table...
```

## 4. Launch Debugging

Two errors hit on first launch attempt; both fixed before runs started.

### Bug 1 — `ModuleNotFoundError: No module named 'src'`

**Where:** both vllab6 (local) and vllab10 (SSH) failed at the same line:
```
from src.data.contrastive_dataset import FacapContrastiveDataset, contrastive_collate
```

**Cause:** `accelerate launch src/training/train_plan5.py` invokes Python with the
script path as `sys.path[0]` — that's `src/training/`, not the repo root. So
`src` is not importable. The repo doesn't install itself as a package
(no `pyproject.toml`/`setup.py`); imports rely on the repo root being on
`PYTHONPATH` or being the cwd.

This worked in Plan-6 because runs were launched from interactive shells where
`PYTHONPATH` was inherited. Running the script via `nohup … &` from a fresh
SSH session strips that.

**Fix:** export `PYTHONPATH` in `scripts/run_plan5.sh`:

```bash
export PYTHONPATH="${REPO_ROOT}:${PYTHONPATH:-}"
```

This makes the launch self-contained — works whether invoked interactively,
via nohup, or over SSH.

### Bug 2 — `accelerate: command not found` on vllab10

**Cause:** `ssh vllab10 "bash -lc '…'"` runs a login shell but on this server
conda is not auto-activated by `.bashrc`. The script calls
`conda run -n fashion_retrieval which accelerate` to find the binary; with no
`conda` in PATH it falls back to plain `accelerate`, which then isn't found.

**Fix:** source conda's profile script before invoking the launch wrapper:

```bash
ssh vllab10 "bash -lc 'source ~/miniconda3/etc/profile.d/conda.sh && \
  cd ~/projects/fashion-retrieval-agent && \
  nohup bash scripts/run_plan5.sh …'"
```

Same NAS-shared conda env, just needs to be discoverable in PATH.

### Future-proofing

The `PYTHONPATH` fix is committed in `run_plan5.sh` so future launches don't
hit Bug 1. Bug 2 is a per-launch SSH-command pattern — every remote launch
must source conda explicitly. (Could be moved into a wrapper script later.)

## 6. Conclusions

### Cosine LR was a mixed result

| Run | Total steps | LR @ end | Plan-6 best | Plan-7 best | Δ |
|-----|-------------|----------|-------------|-------------|---|
| vllab10 | 1872 | 0% of max | 0.402 | 0.378 | **−0.024** |
| vllab6  | 7542 | 0% of max | 0.387 | 0.396 | **+0.009** |

The schedule's effect depends on **total optimizer steps**, not epochs:

- vllab10 has only 1872 steps total. Cosine to η_min=0 collapses the LR to ~7% of
  max by epoch 14, then to 0 by the end. The model loses the ability to escape its
  current plateau.
- vllab6 has 7542 steps. The same schedule fraction means 4× more steps at
  meaningful LRs, allowing the model to keep refining.

### dev/loss logging worked exactly as designed

- vllab10 dev/loss decreased monotonically (6.22 → 2.01) the entire training —
  unambiguous "no overfitting" signal that R@K alone could not have provided.
- The plateau in headline R@10 (0.373–0.378 range from epoch 14 onward) combined
  with dev/loss still slowly dropping was diagnostic: the model was making
  fine-grained continuous-similarity improvements that did not change top-10
  rankings. Confirms the bottleneck is in the *training signal itself* (random
  in-batch negatives saturated), not in optimization.
- vllab6 dev/loss bottomed at epoch 10 (1.955) and has begun to rise — first time
  we have caught onset of overfitting in real time.

### What this means for Plan-8 / future work

1. **Default `--lr-schedule` reverted to `none`.** Cosine remains opt-in for
   experiments (e.g., a deliberate long-schedule rerun) but no longer the default.
2. **Hard-negative mining is the unblocked next step.** Both Plan-6 and Plan-7
   plateau because random in-batch negatives provide no signal once the model has
   separated easy classes. No LR schedule fixes that.
3. **dev/loss should always be logged.** It cleanly distinguishes "stopped
   improving" (R@K plateau + dev/loss flat) from "overfitting" (R@K plateau +
   dev/loss rising). Plan-7 wouldn't have given this read without it.
