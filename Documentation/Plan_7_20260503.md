## 1. Motivation

Plan-6 produced the best results so far (headline R@10 = 0.402, server 10, epoch 16)
but left three diagnosed gaps:

1. **No dev loss** — only R@K was logged at eval. R@K is a coarse, noisy signal;
   multi-positive InfoNCE on a fixed dev set is the primary early-warning for
   overfitting (loss rising while R@K oscillates). Was missing throughout Plan-5 and Plan-6.

2. **Constant LR** — AdamW ran at fixed lr_lora=2e-5, lr_proj=1e-4 for all 18 epochs.
   Plan-6 headline R@10 oscillated rather than converging smoothly in the final epochs
   (e.g. 0.381–0.402 from epoch 10 onward). A cosine decay that reduces LR toward the
   end could squeeze out a cleaner convergence instead of oscillating around the plateau.

3. **Hard negative mining** — current in-batch negatives are random; the model saturates
   them by epoch ~10. Deferred to a future plan (Plan-8 or later). Noted here so it
   is not forgotten.

Plan-7 addresses items 1 and 2. Item 3 is explicitly out of scope.

## 3. Dev Loss Design

### What to compute

At every eval trigger (every 0.5 epoch), run multi-positive InfoNCE on all 500 dev
queries using the current model weights and the frozen gallery embeddings.

- Encode all 500 dev queries in batches (no gradient).
- Stack target embeddings from the gallery cache.
- Build multi-positive mask from `target_ids` (same as training).
- Call `loss_fn(q, t, gather=False, target_ids=tids)` — no cross-GPU gather needed
  since this is a single-process eval over a fixed set.
- Log as `dev/loss`.

### Why this is the right overfitting signal

Training loss reflects whatever batches happened to be sampled; it is noisy and
biased toward easy negatives at the start. Dev loss is computed on a fixed held-out
set with a fixed positive mask, so epoch-to-epoch changes are comparable. A rising
dev loss while training loss continues to fall is the clearest possible sign of
overfitting — cleaner than watching R@K oscillate by ±0.02.

### New function: `run_dev_loss` in `online_eval.py`

```python
@torch.inference_mode()
def run_dev_loss(
    model,
    loss_fn,
    dev_items,
    gallery_lookup,
    tid_to_idx,
    base_ds,
    device,
    batch_size=32,
) -> float:
    """Multi-positive InfoNCE on the full dev set. Returns scalar loss."""
```

Collects all q_embs and t_embs in CPU memory, moves to device in one shot,
calls loss_fn once over the full 500-item dev set. No gather (single-process).

### Logging

Added to every `metric_row`:
```
"dev/loss": <float>
```
Logged to W&B at the same step as R@K metrics.

## 4. Cosine LR Schedule Design

### Scheduler

`torch.optim.lr_scheduler.CosineAnnealingLR`:

```
lr(step) = lr_min + 0.5 * (lr_base - lr_min) * (1 + cos(π * step / T_max))
```

- `T_max` = total training steps (`steps_per_epoch × max_epochs`)
- `lr_min` = 0 (decay to zero)
- Applied to all three param groups (lora, proj, logit_scale) proportionally to
  their base LRs
- Stepped after every `optimizer.step()`

### New CLI args

```
--lr-schedule   {none, cosine}   default: none
```

`none` preserves Plan-6 behavior exactly (constant LR). Cosine remains
opt-in. *(Default reverted from `cosine` to `none` after Plan-7 results — see
Progress_7 §6: cosine harmed short-schedule runs, only mildly helped long ones.)*

### Logging

`train/lr_lora` and `train/lr_proj` already logged every 20 steps.
With the scheduler active these will now show the decaying values — no extra
logging needed, but the W&B chart will show the decay curve automatically.

### Resume behavior

When resuming from a checkpoint (`--resume-from`, `--start-epoch N`), the
scheduler must be fast-forwarded to the right step:

```python
for _ in range(global_step):
    scheduler.step()
```

This is done before training starts so the LR on resume matches where it would
have been had training run continuously.

## 5. Hard Negative Mining — Future TODO

Not implemented in Plan-7. Recorded here so context is not lost.

**Why it matters:** Plan-6 plateaued at headline R@10 ≈ 0.40. The model saturates
random in-batch negatives by epoch ~10; gradients approach zero. Hard negatives
(gallery items most similar to the query but not the correct target) would force
finer discrimination and could push past 0.40 toward the Phase-A bar of 0.533.

**When to revisit:** after Plan-7 results show whether cosine LR + dev loss signal
yield a cleaner convergence. If the plateau persists, hard negatives are the next
lever.
