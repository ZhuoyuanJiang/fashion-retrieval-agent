## Summary

The Plan-12 isolation experiment ran cleanly to 18 epochs on a separate
server (8× A6000-class, bs=8, gather=ON). The PEFT-aware gradient
checkpointing fix worked as designed: gradients flow into both adapters
without silent corruption, τ_inv tracks the same trajectory as the
separate-backbone bs=8 reference, and the model trains stably.

**Headline result — peak head R@10 = 0.6230 at ckpt_epoch12.**

This places the shared-backbone variant **above the Plan_10 §8 decision
threshold (≥ 0.62 = "shared competitive with separate")**. The 0.054
peak gap between bs=4 shared and bs=8 separate decomposes into:

- **batch-size effect: ~0.040** (closed by the Plan-12 fix going bs=4 → bs=8)
- **architecture effect: ~0.014** (residual: shared vs separate, both at bs=8)

So ~74% of the original architecture-vs-batch confound was batch-size
driven, ~26% is a real but small architectural cost. Per the §8 matrix,
this is "shared competitive — Plan 13 V2 ablations should use the
shared variant as default" because the VRAM savings and the V4
audio-swap calculus (Plan_10 Appendix A) both favor shared when the
retrieval gap is small.

## 1. The Plan-12 Question (restated)

Plan-10 V1 produced:

| Variant | bs | effective negatives (8 GPU gather) | peak head R@10 |
|---|---|---|---|
| Separate backbones (Option B) | 8 | 64 | **0.6370** (ckpt_epoch11) |
| Shared backbone (Option A) | 4 | 32 | 0.5870 (ckpt_epoch16-17) |

(Option A's headline R@10 oscillates noisily in 0.55-0.59 from ep 5
onwards; 0.5870 is the run-wide max, 0.5830 at ep 7.5 is an earlier
local high. See Progress_11 §"Peak vs final and the τ_inv runaway story"
for the full trajectory.)

The 0.050 peak gap was confounded: shared backbone was forced to bs=4
because the PEFT footgun (`set_adapter` flips `requires_grad`, and HF
gradient checkpointing's recompute pass reads `model.active_adapter` at
recompute time rather than at forward time) made gradient checkpointing
unsafe — and without checkpointing, bs=8 OOM'd at 47.5/47.5 GB.

Plan 12 asked: **is the 0.054 gap architecture, batch size, or both?**

The fix: a `torch.utils.checkpoint`-aware `context_fn` that snapshots
`requires_grad` on every LoRA param before each recompute, switches to
the forward-time adapter, runs the recompute, then restores the
trainability snapshot on exit. This lets gradient checkpointing run
safely, freeing the VRAM to use bs=8 (= 64 effective negatives, same as
Option B). One variable changes vs Option A: batch size. Architecture
remains shared.

adjust `/ssd2/zhuoyuan/...` paths to whatever local SSD is mounted there.

## 4. Decomposition: batch size vs architecture

Working backwards from the three V1 data points:

| Run | Arch | bs | effective negs | peak head R@10 |
|---|---|---|---|---|
| Option A | shared | 4 | 32 | 0.5870 |
| **Plan 12** | **shared** | **8** | **64** | **0.6230** |
| Option B | separate | 8 | 64 | 0.6370 |

**Holding architecture constant (shared), bs=4 → bs=8 lift:**
0.6230 − 0.5870 = **+0.036 head R@10** = batch-size effect

**Holding batch size constant (bs=8), shared → separate lift:**
0.6370 − 0.6230 = **+0.014 head R@10** = architecture effect

**Total gap (the original Option A vs Option B confound):**
0.6370 − 0.5870 = 0.050 = 0.036 + 0.014 ✓

So:
- **~72%** of the original gap was batch-size.
- **~28%** of the gap is the architectural cost of two adapters sharing
  one frozen substrate vs each tower owning its own backbone.

The architectural cost is real but small. Plan_10 Appendix A's audio-swap
argument for the shared variant (when V4 swaps the query side to audio,
the shared variant only retrains one adapter and keeps the target
adapter; the separate variant retrains the full query backbone) is now
~0.014 head R@10 worth of trade-off — well within the noise of any V2
ablation lift.

## 5. τ_inv: the fix worked exactly as designed

The diagnostic claim from Progress_10 §10 (dev-loss / R@K decoupling driven
by τ_inv runaway with too-few negatives) predicted that bs=8 should
stabilize τ_inv. Plan 12 confirms this empirically:

| Run | bs | effective negs | init τ_inv | final τ_inv | Δ over 18 ep |
|---|---|---|---|---|---|
| Option A (shared) | 4 | 32 | 14.5 | 23.50 | +9.00 |
| **Plan 12 (shared)** | **8** | **64** | **14.3** | **18.40** | **+4.10** |
| Option B (separate) | 8 | 64 | 14.5 | 18.34 | +3.84 |

Plan 12's τ_inv tracks Option B's almost identically. This is the
load-bearing safety check on the PEFT-aware context_fn: if the recompute
had been routing gradients into the wrong adapter, the trainable τ_inv
parameter would not have evolved this cleanly — we'd see anomalous
trajectory, NaN spikes, or a degenerate stall.

The fix is correct.

## 7. §8 Decision Matrix Outcome (Plan_12)

Plan_12 §8 thresholds (head R@10 at peak):

| Threshold | Outcome | Interpretation |
|---|---|---|
| ≥ 0.62 | **Plan 12 peak = 0.6230 → here** | Batch size confound confirmed. Shared backbone is competitive when matched on effective negatives. Plan 13 default = shared. |
| 0.58 - 0.62 | — | Partial — would have required a bs=12 / bs=16 datapoint. |
| < 0.58 | — | Architecture dominates. |

**Decision:** Plan 13 V2 ablations will use the **shared backbone** as
the default architecture, with the following levers (V2 / V3 from
Plan_10 §3 + new levers from Progress_10 §10):

1. **τ_inv clip** at ~15-16 (where peaks landed). Prevents the late-stage
   loss-metric decoupling that costs ~0.06 head R@10.
2. **Cosine LR decay** or **early stop on head R@10** (not dev/loss —
   they diverge past the peak).
3. **Trainable-surface ablation** (V2): heads-only, query-LoRA-only,
   larger rank, unfrozen MLP — all on the shared variant.
4. **Target-prompt sweep** (V3): plain / fashion-attribute list / empty,
   replacing the locked "Describe this image in detail." from Plan_10 §4.2.
5. **Hard-negative mining** (Plan-7 unblocked next lever): in-batch
   negatives are currently random; hard negatives from the gallery would
   push past the current ceiling.

V4 audio extension (swap query side to speechQwen2VL) is now justified
on architecture grounds — the shared variant's swap is a single-adapter
retrain.
